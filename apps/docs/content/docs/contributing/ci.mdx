> Reference for contributors. The authoritative list of pre-PR commands lives in [`CLAUDE.md`](https://github.com/AtlasDevHQ/atlas/blob/main/CLAUDE.md). This page documents the *workflows* that enforce them, plus a few that don't gate merge but still post results to the PR.

## Required gates (the `CI` workflow)

`.github/workflows/ci.yml` runs on every pull request to `main` and must pass before merge:

- **`bun install --frozen-lockfile`** ŌĆö fails if `bun.lock` is out of date.
- **`bun x syncpack lint`** ŌĆö dependency version consistency across the workspace.
- **Dockerfile bun version pins** ŌĆö every `oven/bun:<version>` in any `Dockerfile` matches the workspace `BUN_VERSION`.
- **Dockerfile workspace completeness** ŌĆö `scripts/check-dockerfile-workspace.sh`.
- **Railway watchPatterns coverage** ŌĆö `scripts/check-railway-watch.sh`. Every `COPY` source in a deploy `Dockerfile` is covered by its service's `railway.json` watchPatterns. Prevents the F-10 incident where a `bun.lock` change went unshipped because `watchPatterns` didn't cover it.
- **Template drift** ŌĆö `scripts/check-template-drift.sh`. `create-atlas/templates/nextjs-standalone/src/` is gitignored and regenerated by `prepare-templates.sh`; this check verifies it stays in sync.
- **Security headers drift** ŌĆö `scripts/check-security-headers-drift.sh`.
- **Drizzle schema Ōåö migration drift** ŌĆö `scripts/check-schema-drift.sh`.
- **Builds** ŌĆö SDKs, widget bundle, then the root build. The OpenAPI Ōåö api-reference drift check runs after the builds because `extract-openapi.ts` needs the built dependencies at import time.
- **Type check** ŌĆö `bun run type` (root) and `bun run --filter '@atlas/web' type`.
- **Lint** ŌĆö `bun run lint`.
- **Test** ŌĆö `@atlas/api` tests run in a parallel `api-tests` matrix sharded 4 ways. Other workspace test suites run serially in the main job.

The local equivalent is the `/ci` slash-command ŌĆö runs lint, type, test, syncpack, template-drift, and railway-watch in one shot. The workflow runs additional drift checks (Dockerfile pins, security headers, schema, OpenAPI) that local `/ci` doesn't cover but rarely flag.

### Fast local feedback

The full `bun run test` suite is slow because `@atlas/api` runs every test file in its own subprocess (this avoids the cross-test state leaks that bit us before the isolated runner). For the edit/test cycle, prefer:

```bash
cd packages/api
bun run scripts/test-isolated.ts --affected     # only tests whose source graph your branch touched vs origin/main
bun run scripts/test-isolated.ts --since HEAD~3 # last-3-commit window
bun test path/to/file.test.ts                   # single file
```

Run the full `bun run test` once before opening a PR even if `--affected` has been clean ŌĆö the `--affected` detector occasionally misses transitive imports.

## Lighthouse budget (advisory)

`.github/workflows/lighthouse.yml` runs on PRs that touch `apps/www/**`, `packages/web/**`, the `lighthouserc.js` config, or the workflow itself. It does **not** gate merge ŌĆö the assertions in `lighthouserc.js` are all `warn`-level for the first month while we calibrate flake on shared CI runners.

What it does:

1. Builds `apps/www` (static export) and `packages/web` (Next prod build).
2. Starts both locally ŌĆö apps/www on `:8080`, packages/web on `:3000`.
3. Runs Lighthouse 3├Ś per surface, twice ŌĆö once with `LH_FORM_FACTOR=desktop`, once `mobile`.
4. Uploads reports to:
   - The workflow artifact `lighthouse-reports` (HTML + JSON, 14-day retention).
   - Lighthouse CI's temporary public storage ŌĆö shareable URLs in the report output.
5. Posts (or updates) a single PR comment with a per-surface score table.

### Refreshing the baselines

The baseline `.md` files document what the surface measured at when CI was set up ŌĆö they're the seed for `lighthouserc.js` thresholds, not a source of truth that CI re-asserts against. To re-measure (e.g., after a redesign):

```bash
cd apps/www
bun install
bun run build               # next build ŌåÆ apps/www/out/
PORT=8080 bun serve.ts &    # static server

# Hit http://localhost:8080/ and http://localhost:8080/pricing with the
# scratch runner pattern documented in
# apps/www/.design/demo/lighthouse-baseline.md.
```

Update the `.md` file with the new numbers and adjust the thresholds in `lighthouserc.js` to match. The thresholds should sit a hair below the WSL2 numbers ŌĆö CI runners measure noisier than a dev box.

### Promoting to a hard gate

When an assertion has been stable on PR comments for ~4 weeks, change its level in `lighthouserc.js` from `"warn"` to `"error"`. lhci will then exit non-zero on a regression ŌĆö and the `continue-on-error: true` setting on both the desktop and mobile lhci steps will need to be reconsidered (it's there today so a single flaky run doesn't bury the comment, but once assertions can fail the gate, that hiding behavior needs an explicit policy decision).
