The setup wizard is a browser-based alternative to `atlas init`. It walks you through profiling your database, selecting tables, reviewing and optionally enriching the generated entity YAML, and saving everything to your workspace — all from `/wizard`.

The wizard is the **one shared generation flow** for a [Connection group](/semantic-layer/multi-env). Several in-product doors launch it ([ADR-0012](https://github.com/AtlasDevHQ/atlas/blob/main/docs/adr/0012-group-scoped-semantic-layer-directories.md)): the first-run onboarding wizard, an inline prompt after adding a connection in **Admin → Connections**, and the **Generate semantic layer** empty state in the [Semantic Layer browser](/guides/admin-console#semantic-layer-browser). They all run the same code path — the experience is identical whether this is your first database or your tenth. See [Entry points](#entry-points) below.

<Callout title="Prerequisites">
- An **admin** or **owner** role in your Atlas workspace
- At least one [datasource connection](/guides/admin-console#connections) configured
- [Managed auth](/deployment/authentication#managed-auth) enabled with an internal database (`DATABASE_URL`)
</Callout>

---

## How It Works

The wizard is a five-step flow at `/wizard`:

| Step | What happens |
|------|-------------|
| 1. **Select datasource** | Choose a configured database connection |
| 2. **Select tables** | Browse discovered tables and views, filter by name, select which to include |
| 3. **Review & enrich** | Inspect the **mechanical baseline** (columns, types, relationships, flags), edit YAML, and optionally run **LLM enrichment** per table — see [Two-phase generation](#two-phase-generation) |
| 4. **Preview** | Ask a test question to see how the agent would use your semantic layer (optional) |
| 5. **Done** | Entities saved to the resolved Connection group, ready to query |

The wizard calls the same profiler and the same shared generation engine that powers `atlas init`, so you get identical heuristics: foreign key inference, enum detection, abandoned table flags, and denormalization warnings — and identical enrichment output. Generated entities save into the connection's Connection group (the default group flat at `semantic/`, a non-default group under `semantic/groups/<group>/`); see [Group scoping](#group-scoping).

## Step-by-Step Walkthrough

### 1. Select Datasource

Navigate to `/wizard`. The first step lists all database connections configured in your workspace. Select one from the dropdown and click **Next**.

If no connections appear, add one first via **Admin → Connections** or the [signup flow](/guides/signup).

### 2. Select Tables

The wizard queries your database and lists all tables and views (plus materialized views for PostgreSQL). All objects are selected by default.

- Use the **filter** input to narrow the list by name
- Toggle individual tables or use the header checkbox to select/deselect all visible rows
- The badge shows how many tables are currently selected

Click **Generate Entities** to proceed. The wizard will profile each selected table — inspecting columns, data types, cardinality, sample values, primary keys, and foreign keys.

### 3. Review & Enrich

For each profiled table, the wizard generates a **mechanical baseline** entity YAML file — names, types, sample values, cardinality, PKs/FKs. This baseline is a complete, queryable semantic layer on its own; nothing here costs money or waits on a model. Click any table row to expand it and see:

- **Columns** — name, data type, flags (PK, FK, enum-like, nullable), and sample values
- **Relationships** — declared foreign keys (from constraints) and inferred foreign keys (from naming conventions)
- **Flags** — warnings like "possibly abandoned" (table name matches legacy/temp patterns such as `old_*`, `temp_*`, `*_backup`) or "denormalized" (name matches reporting/cache patterns such as `*_summary`, `*_stats`, `*_cache`)
- **Entity YAML** — editable in-place so you can refine descriptions, adjust column types, or add custom measures before saving
- **Profiler notes** — additional observations from the heuristic analysis

Optionally run **LLM enrichment** here to deepen the baseline with business descriptions, use cases, query patterns, and virtual dimensions — see [Two-phase generation](#two-phase-generation). Enrichment never fires automatically; the baseline is saveable as-is. When you're done, click **Preview** to continue.

### 4. Preview (Optional)

Type a natural-language question (e.g., "How many orders by status?") and click the sparkle button. The wizard shows a summary of the entities available to the agent, including the table count and names. This is a static summary — it does not simulate actual agent behavior or query generation.

This step is optional — you can skip straight to **Save & Finish**.

### 5. Done

Click **Save & Finish** in the preview step. The wizard persists your entities to the workspace's org-scoped semantic layer. Once saved:

- Entity YAML files are written to disk under the org-scoped semantic layer, in the resolved [group's directory](#group-scoping) (`semantic/entities/` for the default group, `semantic/groups/<group>/entities/` otherwise)
- The semantic layer whitelist cache is flushed, so new tables are queryable immediately

You'll see a confirmation screen with links to **View Entities** in the admin console or **Start Chatting**.

<Callout type="info">
The wizard generates entity YAML files only. For a complete semantic layer scaffold including catalog, glossary, and metric files, use `atlas init` from the CLI.
</Callout>

## Two-phase generation

Generation is deliberately split into two phases so the LLM **never fires by accident**:

**Phase 1 — Mechanical baseline (automatic, free, instant).** Profiling produces a complete YAML template for every selected table from names, types, sample values, cardinality, and keys. No model call, no spend, no wait. This baseline is a usable, queryable semantic layer on its own — you can save and start querying without ever enriching.

**Phase 2 — Enrichment (explicit, per-table, cost-gated).** An LLM round-trip re-profiles one table for fresh database grounding, then adds business descriptions, use cases, query patterns, and virtual dimensions on top of its baseline. Controls in the Review step:

- **Enrich all** — one deliberate action, gated behind a confirmation dialog that states the table count (e.g. "Enrich 24 tables?") so the spend is never a surprise.
- **Enrich selected** — a per-table selector to enrich only the tables you care about.
- **Ignore** — exclude tables you don't need (dead/legacy, junction, cache) from both enrichment and the final save. The ignore list is **pre-seeded** from the profiler's `possibly_abandoned` flag, so you confirm rather than hunt.

Enrichment runs one request per chosen table with bounded concurrency and upgrades each row in place the instant it settles. Partial completion is safe — each table is independent.

<Callout type="warn">
Enrichment requires a configured LLM provider. On a key-less self-host the **Enrich** actions return `503` with one actionable banner (instead of N per-table auth errors), and the mechanical baseline stays fully saveable.
</Callout>

## Group scoping

Generated entities save into the connection's **Connection group**, not a raw connection id ([ADR-0012](https://github.com/AtlasDevHQ/atlas/blob/main/docs/adr/0012-group-scoped-semantic-layer-directories.md)). The wizard resolves the group server-side, so the entry-point doors stay group-unaware:

- The **default group** writes flat at `semantic/entities/` (`connection_group_id = NULL`) — a single-DB workspace sees no extra nesting.
- A **non-default group** writes under `semantic/groups/<group>/` — one directory per group, so multi-DB workspaces stay legible on disk and in the [grouped tree](/guides/admin-console#semantic-layer-browser).
- Adding a **member to an already-populated group** (e.g. `eu-prod` into `prod`) reuses that group's existing entities rather than generating a second copy — members of a group share one schema and one set of entities.

## Entry points

The wizard is one shared flow with several doors — all DB-count-agnostic, all landing in the correct Connection group:

1. **Inline-on-add (Admin → Connections).** Adding a SQL connection that forms a **new** group surfaces an inline "Generate semantic layer?" prompt. Adding a member to an already-populated group stays silent — that group already has its schema. The prompt deep-links into `/wizard` at the table-picker step for the new connection.
2. **Per-group empty state (Semantic Layer browser).** A group with no entities shows a **Generate semantic layer** button (replacing the old "Run `atlas init`" instruction). Always-available way back in.
3. **Onboarding wizard.** The first-run `/wizard` is just another caller of the same flow — skipping it costs nothing; you pick the thread up later via (1) or (2).

## What Gets Generated

The wizard generates entity YAML files — one per profiled (non-ignored) table — written into the resolved [Connection group's directory](#group-scoping):

| File | Description |
|------|-------------|
| `entities/<table>.yml` | Dimensions, measures, joins, and query patterns for each table (default group) |
| `groups/<group>/entities/<table>.yml` | Same, for a non-default group |

All files are stored scoped to your organization. In a multi-tenant deployment, each workspace gets its own isolated semantic layer. For a complete scaffold including `catalog.yml`, `glossary.yml`, and `metrics/*.yml`, use `atlas init` from the CLI.

## Wizard vs CLI

| | Wizard (`/wizard`) | CLI (`atlas init`) |
|---|---|---|
| **Interface** | Browser UI — point and click | Terminal — command line |
| **Auth** | Requires admin session | Uses `ATLAS_DATASOURCE_URL` directly |
| **Storage** | Org-scoped, in the [group's directory](#group-scoping) | Local `semantic/` (group via `--connection`) |
| **Table selection** | Visual checklist with filter | `--tables` flag or interactive prompt |
| **YAML editing** | In-place editor before save | Edit files after generation |
| **Enrichment** | Per-table, in the Review step (Enrich all / selected) | `atlas init --enrich` |
| **Preview** | Built-in question preview step | N/A — run `atlas query` separately |
| **Best for** | SaaS deployments, non-technical admins | Self-hosted, CI pipelines, version-controlled semantic layers |

<Callout type="info">
Both tools use the same [shared generation engine](https://github.com/AtlasDevHQ/atlas/blob/main/docs/design/semantic-onboarding.md) — same profiler heuristics, same enrichment prompt and merge rules — so the generated entity YAML is identical. The difference is where it's stored and how you interact with it. The CLI additionally generates catalog, glossary, and metric files.
</Callout>

## Troubleshooting

### Connection not found

The wizard resolves connections from the internal database or the runtime connection registry. If a connection doesn't appear:

- Verify it's configured in **Admin → Connections**
- Ensure the connection belongs to your active workspace (org-scoped connections are filtered by org)
- For self-hosted setups using `ATLAS_DATASOURCE_URL`, the `default` connection should appear automatically

### Profiling timeout

Large databases with many tables or wide tables with millions of rows can cause profiling to take longer than expected. If profiling fails:

- Select fewer tables at a time (start with 10–20)
- Check that `ATLAS_QUERY_TIMEOUT` is set high enough (default 30s)
- Ensure the database user has `SELECT` permission on `information_schema` tables

### MySQL backtick escaping

MySQL identifiers use backtick escaping. If you see unexpected errors with table or column names containing special characters, verify that the generated YAML uses the correct quoting. The profiler handles this automatically, but manual YAML edits should preserve backtick-quoted identifiers for MySQL datasources.

### Partial results or errors

If some tables fail to profile while others succeed, the wizard reports per-table errors alongside successful results. Common causes:

- **Permission denied** — the database user lacks `SELECT` on specific tables
- **Unsupported column types** — exotic types may not map cleanly to dimension types
- **Empty tables** — tables with zero rows still generate entities but may produce less useful sample values

Check the error details in the review step. You can proceed with the successfully profiled tables and re-run the wizard for failed tables after fixing the underlying issue.

### Unsupported database type

The wizard profiles **any datasource Atlas can connect to** — not just PostgreSQL and MySQL. Profiling rides connection resolution: if a [datasource plugin](/plugins/datasources) can open a live connection, the wizard introspects and profiles it through the same flow. ClickHouse, Snowflake, DuckDB, BigQuery, Elasticsearch/OpenSearch, and Salesforce all profile in-product this way; `atlas init` from the CLI uses the same engine for terminal-driven setup.

A datasource only falls back to "unsupported" when its plugin is query-only — i.e. it doesn't implement the optional introspection contract (`listObjects`/`profile`). See [authoring a datasource plugin](/plugins/authoring-guide#datasource-profiling-optional) to add profiling support.

## API Endpoints

The wizard API is mounted at `/api/v1/wizard`. All endpoints require admin authentication.

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `POST` | `/profile` | Admin session | List tables and views from a connected datasource |
| `POST` | `/generate` | Admin session | Profile selected tables and generate the mechanical baseline entity YAML (Phase 1) |
| `POST` | `/enrich` | Admin session | LLM-enrich a single table's entity YAML (Phase 2); `503` when no LLM provider is configured |
| `POST` | `/preview` | Admin session | Preview agent behavior with generated entities |
| `POST` | `/save` | Admin session | Persist entities to the resolved Connection group |
