## Scope boundary — this is NOT the amendment elevation

The `learned_patterns` table stores **two** row types. `query_pattern` rows — learned SQL query shapes mined from live execution — are this elevation's subject. `semantic_amendment` rows are owned by the **in-flight** Semantic-Improve Elevation (PRD **#4502**, milestone `v0.0.47`; audit `.claude/research/semantic-improve-audit-2026-07-10.md`). This audit does **not** re-decide amendment UX, identity, or the decide seam. But the two types share one table, one route, and one read function, and that shared machinery is where the sharpest findings live — every such collision is flagged as an **interlock with #4502**, cited, never re-decided.

## Verdict

**A working capture loop with a broken payoff and a governance seam it shares, unguarded, with a bigger sibling.** The parts that are usually flimsy in a young feature are genuinely solid here and should be preserved wholesale:

- **Capture is careful and correctly scoped.** `proposePatternIfNovel` fires fire-and-forget after every successful `executeSQL` (`sql.ts:1454`), capturing `activeOrganizationId` + `connectionGroupId` **synchronously before the promise detaches** — with an explicit comment on why reading ALS later would write `org_id=NULL` and leak cross-tenant (`pattern-proposer.ts:19-35`). Connection-group scope is stamped at insert and honored at read/dedup (`internal.ts:1467`, `findPatternBySQL:1404-1410`, `getApprovedPatterns:1938-1944`), pinned by `pattern-injection.test.ts:623-699` — `us-prod` patterns stay out of `eu-prod` sessions.
- **The promote/decay core is a pure, well-tested decision function.** `decidePromoteDecay` holds no I/O (`promote-decay.ts:126-138`); the scheduler owns only I/O and uses `Promise.allSettled` so one side's failure can't skip the other's cache eviction (`promote-decay-scheduler.ts:141-183`). The #3636 invariants verifiably hold: recency gate prevents approved→pending flip-flop (`promote-decay.ts:100-109`), decay never touches human approvals (`:113-118`), a human review clears `auto_promoted` (`admin-learned-patterns.ts:517,626`), rejected rows are frozen on both proposer and DB backstop (`pattern-proposer.ts:91-97`, `internal.ts:1854`).
- **Side-effect ordering on the backend is correct and shared.** Amendment YAML is applied **before** the status flips to approved on both the single and bulk paths, via the *same* `applyAmendmentFromPayload` the improve review endpoint uses (`admin-learned-patterns.ts:104-118, 498-507, 614-621`; #3613); the pattern cache is invalidated on every status flip and delete (`:533, :575, :646`; #3612) — the improve path should copy this, not the reverse.
- **The admin cockpit has best-in-repo bulk ergonomics.** A partial-failure bulk response narrows the row selection to exactly the failed IDs so retry hits only unfinished work (`page.tsx:222-242`); auto-approved rows carry a distinct violet Sparkles badge so machine promotion never reads as human sign-off (`columns.tsx:44-50, 115-124`; #3636); nuqs URL state, filter-aware empty states, and dark-mode-complete colorization are all done right (`search-params.ts:3-8`, `page.tsx:463-469, 511-518`).

**Where the problems live: at the payoff and at the shared seam, not in the capture.** Three clusters. (1) **The loop doesn't pay off.** An admin who approves a valuable-but-rare pattern gets *nothing* — it stays below the injection threshold forever (H3); on SaaS the auto-promote half is unreachable by tenants, so patterns pile up `pending` in perpetuity (M3); and nothing anywhere measures whether an injected pattern ever helped (M1). (2) **The cockpit lies about what it can do.** Column sort headers render and do nothing (H4); approve/reject failures render behind the modal that triggered them (H5). (3) **The route is a second, weaker decide path over the same amendment rows the #4502 elevation is carefully building one trustworthy seam for** (H2) — no `admin:semantic` gate, no audit rows on bulk, a second audit vocabulary, no claim-then-apply. The grill's job is to decide the payoff model and settle how this surface and #4502's one-Amendment-identity coexist.

## Ranked findings

Severity ladder: CRITICAL > HIGH > MEDIUM > LOW. `verified` = orchestrator re-read the anchor. Anchors are `file:line`; all paths under `/home/user/atlas`.

### HIGH

#### H1 — `getApprovedPatterns` injects non-query-pattern and NULL-org rows into the agent — live garbage SQL + latent SaaS cross-tenant leak `verified` — FILED #4534
The read function feeding approved patterns into every agent turn has two isolation gaps its amendment-read sibling (`reviewSemanticAmendment`, #4487) does not.
- **No `type = 'query_pattern'` filter (LIVE):** `getApprovedPatterns` selects `WHERE status = 'approved'` with no `type` clause (`internal.ts:1946-1953`). Approved `semantic_amendment` rows — sentinel `pattern_sql = amendment:<entity>:<Date.now()>` (`internal.ts:1638`), reachable via auto-approve or either review door — are returned as "query patterns" and rendered into the prompt verbatim by `renderPattern` (`pattern-format.ts:69-75`). Their human descriptions carry real keywords, so they clear the relevance + confidence gates and actually inject. `getPromoteDecayCandidates` (`internal.ts:1999`) *does* filter `type='query_pattern'`, proving the omission is an oversight.
- **No SaaS NULL-org guard (LATENT):** the org clause is `(org_id = $n OR org_id IS NULL)` unconditionally, even on SaaS (`internal.ts:1931-1936`); the amendment path drops the `OR org_id IS NULL` arm on SaaS (`internal.ts:1771-1782`). Latent because tenant admins can only approve `org_id = $orgId` rows (`admin-learned-patterns.ts:80-85`) — armed the moment the platform operator enables `ATLAS_LEARN_PROMOTE_DECAY_ENABLED` in a SaaS region.
- **Failure scenario:** live — an org that approves any amendment gets `SQL: amendment:orders:1720000000` injected as a validated query shape, degrading its business users' answer quality; latent — with the scheduler on, a NULL-org approved pattern injects cross-tenant, exposing tenant A's schema/query shapes to tenant B.
- **Why fix-invariant (filed):** query-pattern injection must return only approved query patterns for the right tenant under *every* possible elevation outcome; the two-clause guard is the fix regardless. Interlocks #4487 (hardened the amendment read surface, stopped short of this injection read) and #4502.

#### H2 — The learned-patterns route is a second amendment-decide path, with weaker governance than the seam #4502 is building `verified` (doc-only — the elevation's central structural question)
`PATCH /:id` and `POST /bulk` can approve `semantic_amendment` rows and rewrite published entity YAML (`admin-learned-patterns.ts:502-507, 619-621`) — a full second decide path over the exact rows PRD #4502 is consolidating into "one `decide` seam … approved means applied by construction." #4502's problem statement, user stories, and decisions are scoped **entirely** to `/admin/semantic/improve` and never mention this route; every "the Amendment" decision there silently assumes a single seam. The divergences, each a sub-finding:
- **H2a — No `admin:semantic` permission gate `verified`:** the router is `createAdminRouter()` + `requireOrgContext()` only (`:391-393`); the improve router adds `requirePermission("admin:semantic")` (`admin-semantic-improve.ts:377`). A workspace admin whose EE custom role was *denied* `admin:semantic` cannot touch the improve console but can `PATCH`/`bulk` the same amendment row here and rewrite the published YAML the agent reads. Two review surfaces, two authZ boundaries. *(Fix is entangled — add the permission, or fold amendment-approve out of this route entirely — so it's a grill decision, not a standalone file.)*
- **H2b — Two audit vocabularies for one decision:** approving here emits `pattern.approve` / `targetType:"pattern"` / `{patternId}` (`:537-551`, `actions.ts:303-307`); approving the same row via improve emits `semantic.improveApply` / `targetType:"semantic"` / `{id,decision}` (`admin-semantic-improve.ts:934-943`). A compliance query for "every semantic-layer amendment decision" built on `target_type='semantic'` silently misses every decision made through this door. Directly interlocks #4502 US#36 ("one audit vocabulary … no dead actions").
- **H2c — Bulk writes zero audit rows `verified`:** the entire bulk handler (`:593-651`) contains no `logAdminAction` (confirmed: the only calls are in single-PATCH `:537,:545` and DELETE `:577`). A `POST /bulk` approving up to 100 amendments rewrites up to 100 published entities with **no** forensic trail — the query_pattern half of this gap is a straight compliance bug independent of the vocabulary question.
- **H2d — Weaker race guard, no claim, no baseline hash:** bulk `UPDATE … WHERE id=$3 AND {org}` has no `AND status='pending'` claim (`:626`); single PATCH applies then updates without checking current status (`:502-523`). Improve's helper at least claims `AND status='pending' RETURNING` (`internal.ts:1781`). Two admins bulk-approving overlapping IDs both apply, YAML written twice; re-approving an already-approved row re-applies and re-snapshots. Interlocks #4502's "claim-then-apply with compensation … baseline hash."
- **H2e — Reviewer attribution diverges:** here `reviewed_by = user?.id` (real UUID, `:517,:623,:626`); improve stamps the literal string `"admin"` (`internal.ts:1780`). The row's provenance depends on the door used.
- **H2f — Cache-invalidation asymmetry:** this route evicts on every flip (`:533,:575,:646`); the improve `/amendments/{id}/review` endpoint does **not** call `invalidatePatternCache` (absent from the caller set) even though flipping an amendment to `approved` changes what `getApprovedPatterns` reads. Approve via improve → agent serves the stale ~5-min TTL copy; approve here → immediate. (See interlock table below for the full map.)

#### H3 — Admin-approved patterns below 0.7 confidence are silently never injected — human curation is a no-op for exactly the rare patterns it exists to promote `verified` (doc-only — a design fork the grill must resolve)
`getRelevantPatterns` unconditionally filters `p.confidence >= getConfidenceThreshold` (default **0.7**) with no human-approved bypass (`pattern-cache.ts:150-151`, `learn-settings.ts:56`). The admin approve path sets `status`/`reviewed_by`/`auto_promoted=false` but **never raises confidence** (`admin-learned-patterns.ts:509-523`); proposals are born at confidence **0.1** (`internal.ts:1468`, +0.1 per organic repeat). So an admin who approves a valuable low-repetition pattern (confidence 0.1–0.6) sees zero effect: the pattern reaches the agent only if the *same normalized SQL* organically re-executes ~6+ times to climb to 0.7 — which for a rare-but-important query may never happen. No UI signal explains why. This defeats the core promise that admin review makes the agent better.
- **Why doc-only:** two plausible fixes — treat human-`approved` as a threshold *bypass* in `getRelevantPatterns`, **or** stamp a floor confidence on manual approve — and choosing between them is a design decision (does an admin approval mean "always inject" or "inject at floor priority"?). That's the grill's call.

#### H4 — Column sort headers render and do nothing — the cockpit's primary triage axis is a no-op lie
Confidence/latency/repetition/created columns render sort toggles (`columns.tsx:207-289`) — no `enableSorting:false`, and the toolbar renders `DataTableSortList` by default (`server-data-table.tsx:99`) — with `manualSorting:true` so there's no client-side fallback (`use-data-table.ts:298`). But `buildPath` never reads `binding.sorting` (`page.tsx:312-321`) and the API hard-codes `ORDER BY created_at DESC` with no `sort` query param (`admin-learned-patterns.ts:457, 170-178`). An admin clicks "Confidence" to review the riskiest patterns first; the arrow flips, `?sort=` updates, rows do not move. They review in a different order than they believe, or conclude the page is broken.
- **Fix direction:** either `enableSorting:false` + hide the sort toolbar (honest), or thread `sortId/sortDesc` through `buildPath` + a whitelisted API `sort` param.

#### H5 — Approve/reject failures render behind the modal that triggered them — admins believe failed reviews succeeded
The single `actionError` `ErrorBanner` renders in the page body (`page.tsx:448-453`), but Approve/Reject/Delete live **inside** the detail `Sheet` (`:610-645`); `updatePatternStatus` only calls `setActionError` on failure — nothing sheet-local, no toast (`:170-188`). An admin opens an amendment's sheet, clicks Approve, the apply fails (e.g. the #3613 apply-first path throws on an ambiguous entity); the banner renders under the open sheet overlay, the buttons re-enable, the admin assumes success and closes the sheet. Worse, the banner's button is labeled "Retry" but its handler is `() => setActionError(null)` — a dismiss masquerading as a retry (`:451`).
- **Fix direction:** surface mutation errors inside the sheet (or a toast); relabel the button "Dismiss" (the `actionLabel` prop already exists).

### MEDIUM

#### M1 — No efficacy/feedback signal: confidence is a monotonic repetition proxy, not a measure of whether a pattern ever helped
The sole reinforcement is `incrementPatternCount` on an *exact* normalized-SQL re-execution (`sql.ts:1454`, `pattern-proposer.ts:98-106`); confidence only ever increases (`internal.ts:1847`, `LEAST(1.0, confidence + 0.1)`) and nothing decrements it; injection is plain prompt text with no record of which patterns entered which turn (`agent.ts:736`). When the agent *adapts* an injected pattern the executed SQL normalizes to a different fingerprint → a brand-new `pending` row, crediting the source nothing; when an injected pattern contributes to a wrong answer nothing demotes it. For an operator or admin there is no metric answering the feature's central claim — "does injecting patterns measurably change what the agent does?" `avg_duration_ms → perfWeight` is the one genuine outcome signal, and it ranks by speed, not correctness.

#### M2 — Retrieval truncates at 100 candidates ordered by confidence DESC, with confidence saturating at 1.0
`getApprovedPatterns` narrows to `ORDER BY confidence DESC LIMIT 100` *before* any keyword scoring (`internal.ts:1950-1951`); confidence saturates at 1.0 after ~9 repeats (`:1847`), so dozens of patterns tie at the ceiling and the cutoff among ties is unspecified. A pattern that is a perfect keyword match for the user's question but sits at rank 101 is never fetched into the cache and never scored. Retrieval quality silently degrades exactly as an org's library grows — when the feature should be most valuable.

#### M3 — On SaaS the promotion half of the loop is unreachable by tenants — patterns accumulate `pending` forever
The write half runs inline on every `executeSQL` in all deploy modes, so `query_pattern` rows accumulate on SaaS. But the only promotion paths are (a) the auto-promote fiber — on SaaS default-off, platform-scoped, `saasVisible:false`, `requiresRestart` (env-only, invisible even in the operator console: `settings.ts:1084-1131`) — or (b) a tenant admin manually approving each row. A workspace whose admin never visits `/admin/learned-patterns` gets zero learned-pattern injection forever; the "self-maintaining loop" (`promote-decay-scheduler.ts:6-13`) simply does not hold for SaaS tenants. Interlocks #4502's "SaaS-first scheduler … self-hosted's single workspace is the degenerate case" — the query_pattern promoter has the *opposite* posture (SaaS-hostile) from what that PRD sets for the amendment scheduler.

#### M4 — No DB-level dedup for `query_pattern` inserts — concurrent fire-and-forget proposals race to duplicate rows
`findPatternBySQL` dedups read-then-insert, "enforced application-side here … NOT by a DB unique constraint" (`internal.ts:1377-1425`), inside a detached promise (`pattern-proposer.ts:66-121`) spawned fire-and-forget with no serialization (`sql.ts:1454`); no unique index exists (`0000_baseline.sql:259-277`, `schema.ts:648-652`). Two concurrent executions of the same normalized SQL both observe "not found" and both INSERT — duplicate `pending` rows that accrue confidence/repetition independently and can both inject, inflating one pattern's apparent weight and cluttering the queue. The `query_pattern`-side analogue of the amendment `Date.now()`-uniquifier problem #4502 is deciding; the two should share one dedup decision (partial unique index + `ON CONFLICT`).

#### M5 — Unvalidated wire seam: no `@useatlas/schemas` schema, the page casts the response with `as`, and the route hand-rolls its Zod shape
The page selects with `r as { patterns?…; total?… }` — the *unvalidated* `useServerDataTable` variant (`page.tsx:308-311`), whose own doc says the validated variant is preferred "so wire drift surfaces as `schema_mismatch` instead of a silently empty table" (`use-server-data-table.ts:84-93`). The route defines `LearnedPatternSchema` inline with hardcoded enums (`admin-learned-patterns.ts:124-143`) rather than importing from `@useatlas/schemas` (grep of `packages/schemas` for `LearnedPattern`: empty), violating the CLAUDE.md SSOT rule. A future rename of the `patterns` key or a field-type change renders as a silently empty "No learned patterns" state for every admin, and the route/type Zod shapes can drift with no cross-package guard.

#### M6 — Approval's effect is never communicated, and nothing signals that patterns await review
The subtitle ("Review and manage agent-proposed query patterns", `page.tsx:331-332`) and the approve buttons (`:610-631`) never state the consequence — that approving injects the SQL into future agent prompts (the docs explain this at `semantic-expert.mdx:330`; the UI never does). The nav entry is plain (`admin-nav.ts:58`), its `badge?: number` field unused (`:28`), and the admin overview has no learned-patterns tile (grep of `admin/page.tsx` for "learned": empty). A trial SaaS admin never learns the agent has been accumulating pending patterns, so the loop stalls at "everything pending forever" unless they enable the nightly job. The empty state also uses the wrong CLI name — "atlas learn" vs the real `atlas-operator learn` (`page.tsx:466` vs `admin-console.mdx:399`).

#### M7 — Stats bar and entity dropdown are hand-rolled raw fetches: 5 extra requests, silently truncated, counts conflate amendments with query patterns
Four `limit=1` list fetches read `total` per status with no `type` filter, so "pending" includes `semantic_amendment` rows owned by the improve surface (`page.tsx:113-118`); a fifth pulls 200 full rows (full SQL + amendment payloads) to build the entity dropdown, so orgs with >200 patterns get a silently incomplete list (`:147-158`). All five use raw `fetch` + hand-computed credentials + a cache-buster (`:87-94`), bypassing the `useAdminFetch` idiom CLAUDE.md mandates; auth failures are `console.debug`-only, so the stats row just vanishes. An admin reads "12 pending", filters to Query Patterns, sees 3, and can't reconcile the 9 amendments.

#### M8 — Reviewer attribution shows a raw user UUID
The detail sheet renders `Reviewed by: {detailPattern.reviewedBy ?? "Unknown"}` (`page.tsx:588`); `reviewed_by` is persisted as `user?.id` and never joined to a name/email (`admin-learned-patterns.ts:517, 623-626`). A self-hosted admin auditing who approved a pattern reads "Reviewed by: 9f3c2a1e-…" with no way to resolve it (the UI gives no link to `/admin/users`).

#### M9 — Operator export drops amendment identity — migrated workspaces lose `semantic_amendment` payloads
`atlas-operator export` selects only `pattern_sql, description, source_entity, confidence, status` (`packages/cli/src/commands/operator/export.ts:104-119`), omitting `type`, `amendment_payload`, `connection_group_id`, `reviewed_by`, `repetition_count`. Every amendment round-trips as a bare pattern — its YAML change, type marker, and group scope gone — so on re-import the amendment identity #4502 makes central is unreconstructable and pending amendments become orphaned query-pattern rows. (Cite #4460 for bundle scope; this is a projection gap, not a scope gap.)

#### M10 — The route's `orgFilter` is fail-OPEN where the improve side is fail-CLOSED
`orgFilter` returns `org_id IS NULL` whenever `orgId` is falsy, for both reads and writes, in every handler (`admin-learned-patterns.ts:75-85`). Not exploitable today — `requireOrgContext` 400s when `activeOrganizationId` is absent (`admin-router.ts:151-162`), the same source `AuthContext.orgId` reads — but the query-level defense-in-depth #4487 added on the improve path (`if (saas && !orgId) return null` + drop the NULL-org arm, `internal.ts:1771-1775`) is absent. A future refactor that lets `AuthContext.orgId` diverge from `activeOrganizationId` (e.g. a service-actor path) silently reads/writes the NULL-org global rows #4487 went to lengths to hide. Interlocks #4502's "shared org-scope clause helper with a reader-enumeration test" — this route is a reader that enumeration must cover.

### LOW

- **L1 — Birth has no quality gate beyond "the query didn't throw"** — the proposer runs after execution with no inspection of row count or whether the answer satisfied the user (`sql.ts:1443-1461`), and novelty is exact-normalized-SQL only, so every distinct successful SELECT (empty-result, exploratory, wrong-but-valid) becomes a `pending` row at repetition 1 (`pattern-proposer.ts:108-121`). The review queue fills with single-observation noise. Fix direction: require `repetition_count ≥ 2` before persisting (mirror suggestions' `MIN_PATTERN_COUNT`).
- **L2 — No `CHECK` constraint on `status`/`type`** — unlike peer status tables (`approval_queue` `0000_baseline.sql:550`, starter-prompt `0030:18`, dashboard-stage `0083:61`), `learned_patterns.status`/`type` are unconstrained `TEXT` (`0000_baseline.sql:259-274`, `0019:2`, `schema.ts:622,628`); a bad value becomes an orphan invisible to both the `'approved'` filter and the `pending` scheduler. (The `pending/approved/rejected` vs content-mode `draft/published` divergence is *deliberate* — a moderation queue predating content-mode, not in `CONTENT_MODE_TABLES`; only the enum CHECK is missing.)
- **L3 — Rolling-mean SQL `CASE` fold is an untested manual twin of `foldRollingMean`** — the UPDATE fold in `incrementPatternCount` (`internal.ts:1834-1866`) hand-mirrors the TS function whose own doc warns "the coupling is *manual* … will silently diverge" (`rolling-mean.ts:9-13`); the test exercises only the TS function (`rolling-mean.test.ts`), never the SQL fold that is the production path for every repeat observation. A divergent edit silently corrupts `avg_duration_ms`, which drives both the auto-promote latency gate and retrieval down-weighting. Fix direction: a pg-fixture test asserting the SQL fold equals `foldRollingMean`.
- **L4 — Auto-promote candidate cap orders `updated_at ASC`, truncating the freshest patterns first** — `getPromoteDecayCandidates` scans `ORDER BY updated_at ASC LIMIT 10000` (`internal.ts:1991-2005`, `promote-decay-scheduler.ts:57,114-121`); past 10k candidates the fiber evaluates the *stalest* rows first and drops the freshest — starving exactly the patterns a tenant is actively re-running. (Only bites when the scheduler is enabled.)
- **L5 — Two disjoint learning pipelines with duplicated normalize/fingerprint logic** — `query_suggestions` are mined from the audit log at threshold 2 and surfaced to end-users (`suggestions.ts:9,49-51,131`, `routes/suggestions.ts:190,212`); `learned_patterns` are mined from live SQL at threshold 1 and surfaced only to the agent (`pattern-proposer.ts:69-83`). They never reconcile — a pattern the agent has validated is invisible to the user-facing suggestion grid. (Approved suggestions *do* fold into the agent prompt as a separate tier, `org-knowledge-section.ts:113-126`; the reverse direction doesn't exist.) Document the split as intentional if it is.
- **L6 — Description-only edits are unaudited** — `PATCH {description:"…"}` writes no audit row (`admin-learned-patterns.ts:512` vs the status-gated log at `:536-552`); an admin rewrites the human-facing description other reviewers trust with no forensic trace.
- **L7 — No conceptual/admin guide** — only generated API reference exists (`apps/docs/content/docs/api-reference/admin-learned-patterns/`); marketing promises the review workflow (`comparisons/vanna.mdx:38`, `index.mdx:43,55`, `raw-mcp.mdx:32`) but there is no how-to for pending→approved→injected, auto-promotion/decay, or how it differs from the improve console. The admin-console guide that *does* mention it is stale — claims optimistic UI the page doesn't have and omits the Type filter, Avg-latency column, and Auto-approved badge (`admin-console.mdx:394-424`).
- **L8 — No connection-group visibility in the UI though patterns are group-scoped** — the wire type omits `connectionGroupId` (`packages/types/src/learned-pattern.ts:56-88`); the DB column is read for apply but never serialized (`toLearnedPattern`, `admin-learned-patterns.ts:27-73`). An admin with prod + staging groups can't tell which group two near-identical pending patterns apply to — approving the wrong one injects into the wrong group's sessions.
- **L9 — Detail sheet is mouse-only** — the row gets `onClick` + `cursor-pointer` with no `tabIndex`/`role`/keydown (`data-table.tsx:66-67`), and the sheet is the only place full SQL/source-queries/diff are readable (`page.tsx:470-473`). A keyboard/screen-reader admin can never open it and must review from the 80-char-truncated SQL cell. (Shared-component issue; bites hardest here because review *requires* the detail view.)
- **L10 — API supports confidence-range filtering the UI never exposes** — `min_confidence`/`max_confidence` are implemented and validated (`admin-learned-patterns.ts:174-175, 444-445`) but absent from the UI (`page.tsx:312-321`, `search-params.ts:3-8`); combined with the dead sort (H4), the feature's primary triage axis is unusable from its own cockpit.
- **L11 — `atlas-operator learn` does not touch `query_pattern` rows** (informational) — the CLI `learn` command analyzes the audit log → YAML + `query_suggestions` (`packages/cli/src/commands/operator/learn.ts:100-135, 208-218`); `query_pattern` rows are produced *only* inline per-query. A SaaS operator has no command to trigger or backfill `query_pattern` analysis for a specific tenant — "run analysis for tenant X" is unsupported.
- **L12 — No SDK / MCP surface** (informational) — `packages/sdk` exposes no learned-pattern methods (only a prose mention, `README.md:209`); env knobs are properly documented (`.env.example:379-385`). Likely intentional (admin-only), but worth a recorded decision if #4502's "one Amendment identity" ever wants programmatic review.

## Grill agenda

The design questions the findings force, for the `/grill-with-docs` session. Phrased as questions — the grill walks this list.

1. **What does "approve" mean for a query pattern?** Today it changes `status` but not confidence, so a human-approved rare pattern is never injected (H3). Does an admin approval mean *always inject* (threshold bypass), *inject at a floor priority* (stamp floor confidence), or something ranked? This is the payoff question the whole cockpit hangs on.
2. **How does this surface coexist with #4502's one-Amendment-identity?** Fold amendment-approve out of the learned-patterns route (query-pattern-only), or route it through #4502's decide seam? (H2, interlock table.) Nothing else about governance parity — permission gate, audit vocabulary, claim-then-apply, reviewer attribution, cache eviction — can be settled until this is.
3. **What is the SaaS lifecycle for query patterns?** The write half runs everywhere but the promote half is SaaS-hostile (default-off, `saasVisible:false`, `requiresRestart`), so tenant patterns pile up `pending` forever (M3). Tenant-tunable workspace-scoped auto-promotion (mirroring #4502's SaaS-first scheduler posture), or documented manual review as the intended SaaS model?
4. **Does the loop need to measure its own payoff?** There is no signal that an injected pattern was used, helped, or harmed; confidence is a monotonic repetition proxy (M1). Is an efficacy signal (attribute injected patterns to a turn, credit structural matches, a negative/decay path) in scope for the elevation, or explicitly deferred?
5. **What is the identity of a query pattern, and should it be enforced in the DB?** No unique constraint means concurrent inserts duplicate (M4); the amendment side has the same `Date.now()`-shaped problem #4502 is deciding. One shared dedup decision (partial unique index + `ON CONFLICT`) across both types, or two?
6. **Is the cockpit honest about what it can do?** Dead sort headers (H4), a confidence filter the API supports but the UI hides (H10), failures hidden behind the modal (H5), amendment rows shown in a table that can't really review them (M7, and the H2 second-surface question). What is the minimum the review UI must show and do to be trustworthy — and should amendment rows even appear here?
7. **Where does the wire contract live?** The route hand-rolls its Zod schema and the page casts with `as` (M5), against the CLAUDE.md SSOT rule. Does the elevation move `LearnedPattern*` into `@useatlas/schemas` as part of the cleanup?
8. **What is the retrieval ceiling as a library grows?** The 100-row confidence-DESC pre-truncation drops perfectly-relevant patterns past the cap (M2). Does relevance need to enter the SQL (FTS) rather than be applied after a confidence cut?
