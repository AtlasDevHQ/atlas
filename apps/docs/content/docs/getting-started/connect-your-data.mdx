Connect Atlas to your database. Atlas has built-in support for **PostgreSQL** and **MySQL**. Additional datasources â€” **BigQuery**, **ClickHouse**, **Snowflake**, **DuckDB** (CSV/Parquet), and **Salesforce** â€” are available as plugins.

<Callout type="info" title="Using app.useatlas.dev?">

You do not need to set environment variables or edit config files â€” the platform manages connection configuration, SSL, and pooling for you. The database-specific setup below (creating read-only users, SSL settings) still applies when preparing your database for Atlas access.
</Callout>

Pick your database below and follow the steps. Each section is self-contained.

## PostgreSQL

**Supported versions:** 12+

**Prerequisites:**
- Network access from Atlas to your PostgreSQL host
- A user with `SELECT` and `USAGE` privileges (read-only recommended)

<Steps>
<Step>
### Create a read-only PostgreSQL user

Atlas enforces SELECT-only via a multi-layer SQL validation pipeline. For defense-in-depth, connect with a read-only user:

```sql
CREATE USER atlas_reader WITH PASSWORD 'your-strong-password';
GRANT CONNECT ON DATABASE your_db TO atlas_reader;
GRANT USAGE ON SCHEMA public TO atlas_reader;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO atlas_reader;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT ON TABLES TO atlas_reader;
```
</Step>

<Step>
### Set the PostgreSQL connection string

On **app.useatlas.dev**, paste the connection string into **Admin &gt; Connections &gt; Add Connection**. For **self-hosted** deployments, set the environment variable:

```bash
ATLAS_DATASOURCE_URL=postgresql://atlas_reader:your-strong-password@your-host:5432/your_db
```

Both `postgresql://` and `postgres://` prefixes are accepted.

**SSL:** Most managed providers (AWS RDS, Supabase, Neon, Railway) require SSL. Append `?sslmode=require`:

```bash
ATLAS_DATASOURCE_URL=postgresql://atlas_reader:password@host:5432/db?sslmode=require
```

<Callout type="info">
The `pg` driver (v8+) treats `sslmode=require` as `verify-full`. Atlas normalizes this automatically. For self-signed certificates, use `?sslmode=no-verify` â€” not recommended for production.
</Callout>

**Non-public schemas:** Set `ATLAS_SCHEMA` to query a schema other than `public`:

```bash
ATLAS_SCHEMA=analytics
```

Atlas validates the schema name at startup (must be a valid SQL identifier â€” letters, digits, and underscores only) and checks that it exists via `pg_namespace`.
</Step>

<Step>
### Generate the PostgreSQL semantic layer

```bash
ATLAS_DATASOURCE_URL="your-connection-string" bun run atlas -- init
```

<Callout type="info" title="You should see">
```
Atlas Init â€” profiling postgres database...
  [1/N] Profiling table_name...
Done! Semantic layer for "default" is at ./semantic/
```

Verify: `ls semantic/entities/` should list your tables as `.yml` files.
</Callout>
</Step>

<Step>
### Verify PostgreSQL connectivity

Start the dev server and check the health endpoint:

```bash
bun run dev
curl http://localhost:3001/api/health
```

Confirm `checks.datasource.status` is `"ok"` and `sources.default.dbType` is `"postgres"`.
</Step>
</Steps>

<Accordions>
<Accordion title="Cannot connect to database">
- Verify the connection string works directly: `psql "$ATLAS_DATASOURCE_URL"`
- Check that the host is reachable (firewalls, VPC peering, security groups)
- For SSL issues, try `?sslmode=require` or `?sslmode=no-verify`
</Accordion>
<Accordion title="No tables were successfully profiled">
- Ensure the user has `SELECT` and `USAGE` on the target schema
- Check that tables exist in the expected schema (`public` by default)
- Use `--tables users,orders,products` to target specific tables
</Accordion>
<Accordion title="pg_hba.conf: no entry for host">
Your PostgreSQL server is rejecting the connection based on client IP. Add a line to `pg_hba.conf` allowing the Atlas host, or use a managed provider's connection settings to whitelist your IP.
</Accordion>
</Accordions>

## MySQL

**Supported versions:** 8.0+

**Prerequisites:**
- Network access from Atlas to your MySQL host
- A user with `SELECT` privileges

<Steps>
<Step>
### Create a read-only MySQL user

```sql
CREATE USER 'atlas_reader'@'%' IDENTIFIED BY 'your-strong-password';
GRANT SELECT ON your_db.* TO 'atlas_reader'@'%';
FLUSH PRIVILEGES;
```

Atlas also enforces `SET SESSION TRANSACTION READ ONLY` on every connection as a defense-in-depth measure.
</Step>

<Step>
### Set the MySQL connection string

On **app.useatlas.dev**, paste the connection string into **Admin &gt; Connections &gt; Add Connection**. For **self-hosted** deployments, set the environment variable:

```bash
ATLAS_DATASOURCE_URL=mysql://atlas_reader:your-strong-password@your-host:3306/your_db
```

Both `mysql://` and `mysql2://` prefixes are accepted.

**SSL:** Append `?ssl=true` for TLS connections. For self-signed certificates: `?ssl={"rejectUnauthorized":false}` â€” not recommended for production.
</Step>

<Step>
### Generate the MySQL semantic layer

```bash
ATLAS_DATASOURCE_URL="your-connection-string" bun run atlas -- init
```

Verify: `ls semantic/entities/` should list your tables.
</Step>

<Step>
### Verify MySQL connectivity

```bash
bun run dev
curl http://localhost:3001/api/health
```

Confirm `checks.datasource.status` is `"ok"` and `sources.default.dbType` is `"mysql"`.
</Step>
</Steps>

<Accordions>
<Accordion title="Cannot connect to database">
- Verify: `mysql -u atlas_reader -p -h your-host your_db`
- Check that the host allows remote connections (bind-address, firewall rules)
- For SSL: try `?ssl=true`
</Accordion>
<Accordion title="Character set issues">
If you see garbled text, ensure your database uses `utf8mb4`. Atlas uses the `mysql2` driver with `supportBigNumbers: true` and `bigNumberStrings: true`.
</Accordion>
</Accordions>

## BigQuery

BigQuery is a plugin-based datasource. Install the plugin and configure it in `atlas.config.ts`.

**Prerequisites:**
- A GCP project with BigQuery enabled
- A service account with `roles/bigquery.dataViewer` and `roles/bigquery.jobUser`
- Network access from Atlas to Google APIs (no VPC peering needed â€” BigQuery uses the public API)

<Steps>
<Step>
### Install the BigQuery plugin

```bash
bun add @useatlas/bigquery @google-cloud/bigquery
```
</Step>

<Step>
### Configure BigQuery in atlas.config.ts

<Tabs items={["Service Account Key File", "Inline Credentials", "Application Default Credentials"]}>
<Tab value="Service Account Key File">
```typescript
import { defineConfig } from "@atlas/api/lib/config";
import { bigqueryPlugin } from "@useatlas/bigquery";

export default defineConfig({
  plugins: [
    bigqueryPlugin({
      projectId: "my-gcp-project",
      dataset: "analytics",
      location: "US",
      keyFilename: process.env.GOOGLE_APPLICATION_CREDENTIALS,
    }),
  ],
});
```
</Tab>
<Tab value="Inline Credentials">
```typescript
import { defineConfig } from "@atlas/api/lib/config";
import { bigqueryPlugin } from "@useatlas/bigquery";

export default defineConfig({
  plugins: [
    bigqueryPlugin({
      projectId: "my-gcp-project",
      dataset: "analytics",
      credentials: JSON.parse(process.env.GCP_CREDENTIALS!),
    }),
  ],
});
```
</Tab>
<Tab value="Application Default Credentials">
```typescript
import { defineConfig } from "@atlas/api/lib/config";
import { bigqueryPlugin } from "@useatlas/bigquery";

export default defineConfig({
  plugins: [
    bigqueryPlugin({
      projectId: "my-gcp-project",
      dataset: "analytics",
    }),
  ],
});
```

ADC works automatically in GCE, Cloud Run, and GKE. For local development, run `gcloud auth application-default login`.
</Tab>
</Tabs>

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `projectId` | `string` | from credentials/ADC | GCP project ID |
| `dataset` | `string` | â€” | Default dataset for unqualified table references |
| `location` | `string` | â€” | Query job location (`US`, `EU`, `us-east1`, etc.) |
| `keyFilename` | `string` | â€” | Path to service account JSON key file |
| `credentials` | `object` | â€” | Parsed service account JSON key |
| `costApproval` | `"auto" \| "threshold" \| "always"` | `"threshold"` | When to gate queries on estimated cost |
| `costThreshold` | `number` | `1.0` | USD threshold for `"threshold"` mode |
</Step>

<Step>
### Create the service account (if needed)

```bash
# Create service account
gcloud iam service-accounts create atlas-reader \
  --display-name="Atlas Reader"

# Grant BigQuery read access
gcloud projects add-iam-policy-binding my-gcp-project \
  --member="serviceAccount:atlas-reader@my-gcp-project.iam.gserviceaccount.com" \
  --role="roles/bigquery.dataViewer"

gcloud projects add-iam-policy-binding my-gcp-project \
  --member="serviceAccount:atlas-reader@my-gcp-project.iam.gserviceaccount.com" \
  --role="roles/bigquery.jobUser"

# Download key file
gcloud iam service-accounts keys create atlas-reader-key.json \
  --iam-account=atlas-reader@my-gcp-project.iam.gserviceaccount.com
```

Set the key file path in your environment:

```bash
GOOGLE_APPLICATION_CREDENTIALS=./atlas-reader-key.json
```

<Callout type="warn">
Never commit service account key files to version control. Add `*.json` key files to `.gitignore`.
</Callout>
</Step>

<Step>
### Generate the BigQuery semantic layer

```bash
bun run atlas -- init
```

The plugin registers as the default datasource. Atlas will profile your BigQuery tables and generate `semantic/entities/*.yml`.
</Step>

<Step>
### Verify BigQuery connectivity

```bash
bun run dev
curl http://localhost:3001/api/health
```

Confirm the health response includes a source with `dbType` set to `"bigquery"` (the plugin registers as `"bigquery-datasource"` in the `sources` object).

<Callout type="info">
BigQuery runs a dry-run cost estimate before every query. By default, queries estimated above $1.00 USD are blocked. Configure this with the `costThreshold` and `costApproval` plugin options.
</Callout>
</Step>
</Steps>

<Accordions>
<Accordion title="Permission denied or 403">
- Ensure the service account has both `bigquery.dataViewer` and `bigquery.jobUser` roles
- Check that the `projectId` matches the project containing your datasets
- Verify the key file path is correct and the file is readable
</Accordion>
<Accordion title="Dataset not found">
- Check the `dataset` option matches an existing dataset in your project
- Verify the `location` matches where the dataset was created
</Accordion>
</Accordions>

## ClickHouse

ClickHouse is a plugin-based datasource using the HTTP transport.

**Prerequisites:**
- Network access from Atlas to your ClickHouse HTTP interface (default port: 8123 for HTTP, 8443 for HTTPS)
- A user with `SELECT` privileges

<Steps>
<Step>
### Create a read-only ClickHouse user

```sql
CREATE USER atlas_reader IDENTIFIED BY 'your-strong-password';
GRANT SELECT ON your_db.* TO atlas_reader;
```

Atlas enforces `readonly: 1` on every query via ClickHouse settings, preventing mutations even if the user has broader privileges.
</Step>

<Step>
### Install the ClickHouse plugin

```bash
bun add @useatlas/clickhouse @clickhouse/client
```
</Step>

<Step>
### Configure ClickHouse in atlas.config.ts

```typescript
import { defineConfig } from "@atlas/api/lib/config";
import { clickhousePlugin } from "@useatlas/clickhouse";

export default defineConfig({
  plugins: [
    clickhousePlugin({
      url: "clickhouses://atlas_reader:your-strong-password@your-host:8443/your_db",
    }),
  ],
});
```

**URL schemes:**
- `clickhouse://` â€” plain HTTP (typically port 8123)
- `clickhouses://` â€” HTTPS/TLS (typically port 8443)

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `url` | `string` | required | ClickHouse connection URL |
| `database` | `string` | from URL path | Database name override |

<Callout type="info">
If you use `clickhouse://` with port 8443, Atlas will warn that you likely intended `clickhouses://` for TLS.
</Callout>
</Step>

<Step>
### Generate the ClickHouse semantic layer

```bash
bun run atlas -- init
```
</Step>

<Step>
### Verify ClickHouse connectivity

```bash
bun run dev
curl http://localhost:3001/api/health
```

Confirm the health response includes a source with `dbType` set to `"clickhouse"` (the plugin registers as `"clickhouse-datasource"` in the `sources` object).
</Step>
</Steps>

<Accordions>
<Accordion title="Missing or invalid 'meta' field">
- Ensure the query returns a result set (not a command like `SHOW TABLES`)
- Check that the connection URL points to the correct database
</Accordion>
<Accordion title="Connection refused on port 8123/8443">
- Verify the HTTP interface is enabled in your ClickHouse config (`<http_port>` or `<https_port>`)
- Check firewall rules allow traffic on the HTTP port
- For ClickHouse Cloud, use port 8443 with `clickhouses://`
</Accordion>
</Accordions>

## Snowflake

Snowflake is a plugin-based datasource.

**Prerequisites:**
- A Snowflake account with a running warehouse
- A user/role with `SELECT` privileges on the target schema

<Steps>
<Step>
### Create a read-only Snowflake role

Snowflake has no session-level read-only mode. Atlas enforces SELECT-only via SQL validation, but for defense-in-depth, use a role with only SELECT privileges:

```sql
CREATE ROLE atlas_readonly;
GRANT USAGE ON WAREHOUSE your_warehouse TO ROLE atlas_readonly;
GRANT USAGE ON DATABASE your_db TO ROLE atlas_readonly;
GRANT USAGE ON SCHEMA your_db.your_schema TO ROLE atlas_readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA your_db.your_schema TO ROLE atlas_readonly;
GRANT SELECT ON FUTURE TABLES IN SCHEMA your_db.your_schema TO ROLE atlas_readonly;

CREATE USER atlas_reader
  PASSWORD = 'your-strong-password'
  DEFAULT_ROLE = atlas_readonly;
GRANT ROLE atlas_readonly TO USER atlas_reader;
```
</Step>

<Step>
### Install the Snowflake plugin

```bash
bun add @useatlas/snowflake snowflake-sdk
```
</Step>

<Step>
### Configure Snowflake in atlas.config.ts

```typescript
import { defineConfig } from "@atlas/api/lib/config";
import { snowflakePlugin } from "@useatlas/snowflake";

export default defineConfig({
  plugins: [
    snowflakePlugin({
      url: "snowflake://atlas_reader:your-strong-password@your-account/your_db/your_schema?warehouse=YOUR_WH&role=atlas_readonly",
    }),
  ],
});
```

**URL format:** `snowflake://user:pass@account/database/schema?warehouse=WH&role=ROLE`

- `account` â€” plain identifier (e.g. `xy12345`) or fully-qualified locator (e.g. `xy12345.us-east-1`)
- `/database` and `/database/schema` path segments are optional
- `warehouse` and `role` query params are case-insensitive

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `url` | `string` | required | Snowflake connection URL |
| `maxConnections` | `number` | `10` | Connection pool size (max 100) |

Atlas sets `STATEMENT_TIMEOUT_IN_SECONDS` per session and tags all queries with `QUERY_TAG = 'atlas:readonly'` for audit trail in `QUERY_HISTORY`.
</Step>

<Step>
### Generate the Snowflake semantic layer

```bash
bun run atlas -- init
```
</Step>

<Step>
### Verify Snowflake connectivity

```bash
bun run dev
curl http://localhost:3001/api/health
```

Confirm the health response includes a source with `dbType` set to `"snowflake"` (the plugin registers as `"snowflake-datasource"` in the `sources` object).
</Step>
</Steps>

<Accordions>
<Accordion title="Session or authentication errors">
- Verify the account identifier format (e.g. `xy12345` or `xy12345.us-east-1`)
- Check that the warehouse is running and the role has `USAGE` on it
- Ensure the password is correct â€” Snowflake locks accounts after too many failed attempts
</Accordion>
<Accordion title="Warehouse suspended">
If the warehouse is suspended and `AUTO_RESUME` is off, start it manually: `ALTER WAREHOUSE your_warehouse RESUME`. For Atlas, consider enabling `AUTO_RESUME` on the warehouse.
</Accordion>
</Accordions>

## DuckDB (CSV / Parquet)

DuckDB is an in-process analytical engine â€” no external database server required. Use it to query CSV or Parquet files directly.

**Prerequisites:**
- CSV or Parquet files accessible from the Atlas host, or an existing `.duckdb` database file

<Steps>
<Step>
### Install the DuckDB plugin

```bash
bun add @useatlas/duckdb @duckdb/node-api
```
</Step>

<Step>
### Option A: Ingest files with the CLI

The fastest path â€” Atlas creates an in-memory DuckDB, loads files, profiles them, and generates the semantic layer in one step:

```bash
# CSV files
bun run atlas -- init --csv file1.csv,file2.csv

# Parquet files
bun run atlas -- init --parquet file1.parquet,file2.parquet
```

No `atlas.config.ts` needed for this path.
</Step>

<Step>
### Option B: Configure a persistent DuckDB database

For an existing `.duckdb` file, configure in `atlas.config.ts`:

```typescript
import { defineConfig } from "@atlas/api/lib/config";
import { duckdbPlugin } from "@useatlas/duckdb";

export default defineConfig({
  plugins: [
    duckdbPlugin({ url: "duckdb:///absolute/path/to/analytics.duckdb" }),
  ],
});
```

**URL formats:**
- `duckdb://` or `duckdb://:memory:` â€” in-memory database
- `duckdb:///absolute/path.duckdb` â€” absolute path (note the triple slash)
- `duckdb://relative/path.duckdb` â€” relative path

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `url` | `string` | â€” | DuckDB URL (alternative to `path`) |
| `path` | `string` | â€” | Direct file path or `:memory:` |
| `readOnly` | `boolean` | `true` for files | Open in read-only mode |

File databases open in read-only mode by default. In-memory databases are writable.

Then generate the semantic layer:

```bash
bun run atlas -- init
```
</Step>

<Step>
### Verify DuckDB connectivity

```bash
bun run dev
curl http://localhost:3001/api/health
```

Confirm the health response includes a source with `dbType` set to `"duckdb"` (the plugin registers as `"duckdb-datasource"` in the `sources` object).
</Step>
</Steps>

<Accordions>
<Accordion title="File not found or permission denied">
- Check the file path is correct and readable by the Atlas process
- For absolute paths, use triple-slash: `duckdb:///home/user/data.duckdb`
</Accordion>
<Accordion title="CSV parsing errors">
DuckDB auto-detects CSV delimiters and types. If detection fails, pre-convert to Parquet for more reliable schema inference.
</Accordion>
</Accordions>

## Salesforce

<Callout type="warn">
Salesforce uses **SOQL**, not SQL. Atlas provides a separate `querySalesforce` tool. The query syntax and capabilities differ from SQL datasources â€” no `JOIN`, no `SELECT *`, and relationship queries use dot notation.
</Callout>

**Prerequisites:**
- A Salesforce user with API access enabled in their profile
- The user's security token (reset from Salesforce Setup > My Personal Information > Reset Security Token)

<Steps>
<Step>
### Install the Salesforce plugin

```bash
bun add @useatlas/salesforce jsforce
```
</Step>

<Step>
### Configure Salesforce in atlas.config.ts

```typescript
import { defineConfig } from "@atlas/api/lib/config";
import { salesforcePlugin } from "@useatlas/salesforce";

export default defineConfig({
  plugins: [
    salesforcePlugin({
      url: "salesforce://user:pass@login.salesforce.com?token=SECURITY_TOKEN",
    }),
  ],
});
```

**URL format:** `salesforce://username:password@hostname?token=TOKEN&clientId=ID&clientSecret=SECRET`

| Component | Description |
|-----------|-------------|
| `hostname` | `login.salesforce.com` (production) or `test.salesforce.com` (sandbox) |
| `token` | Salesforce security token (appended to password for authentication) |
| `clientId` / `clientSecret` | Optional â€” for OAuth connected app authentication |

<Callout type="info">
Store credentials in environment variables: `salesforcePlugin({ url: process.env.SALESFORCE_URL! })`. Never hardcode passwords in `atlas.config.ts`.
</Callout>
</Step>

<Step>
### Generate the Salesforce semantic layer

```bash
bun run atlas -- init
```

Object discovery uses `describe()` and `listObjects()` API calls rather than system catalogs.
</Step>

<Step>
### Verify Salesforce connectivity

```bash
bun run dev
curl http://localhost:3001/api/health
```

Confirm the health response includes a source with `dbType` set to `"salesforce"` (the plugin registers as `"salesforce-datasource"` in the `sources` object).
</Step>
</Steps>

<Accordions>
<Accordion title="INVALID_SESSION_ID">
- Atlas auto-retries on session expiry, but persistent failures indicate credential issues
- Reset your security token from Salesforce Setup if needed
- Check that the user has API access enabled in their Salesforce profile
</Accordion>
<Accordion title="SOQL differences from SQL">
- No `SELECT *` â€” always list specific fields
- No `JOIN` â€” use relationship queries: `SELECT Account.Name FROM Contact`
- Date literals: `YESTERDAY`, `TODAY`, `THIS_MONTH`, `LAST_WEEK`, `LAST_N_DAYS:30`
- The agent uses `querySalesforce` instead of `executeSQL`
</Accordion>
</Accordions>

## Generate the semantic layer

<Callout type="info" title="Using app.useatlas.dev?">
  The hosted platform generates the semantic layer automatically when you add a connection. You can review and edit entity files in **Admin &gt; Semantic Layer**.
</Callout>

After configuring any datasource (self-hosted), generate the YAML files the agent reads before writing queries:

```bash
bun run atlas -- init
```

This queries system catalogs and generates:

- `semantic/entities/*.yml` â€” one file per table with columns, types, sample values, joins, measures, and query patterns
- `semantic/catalog.yml` â€” entry point listing all entities
- `semantic/glossary.yml` â€” auto-detected ambiguous terms and enum definitions
- `semantic/metrics/*.yml` â€” per-table metric definitions

### Useful flags

| Flag | Description |
|------|-------------|
| `--tables users,orders` | Profile specific tables only |
| `--enrich` | Use your LLM to add richer descriptions and query patterns |
| `--no-enrich` | Skip LLM enrichment explicitly |
| `--csv file1.csv,file2.csv` | Load CSV files into DuckDB |
| `--parquet file1.parquet` | Load Parquet files into DuckDB |

The generated YAMLs are a starting point. Review them and add business context, remove tables that should not be queryable, and adjust sample values for sensitive columns. Better YAMLs produce better queries.

See [Semantic Layer](/getting-started/semantic-layer) for a deep dive, or [CLI Reference](/reference/cli#init) for all `init` flags.

## Multi-source configuration

<Callout type="info">
For a comprehensive guide on how the agent routes queries across datasources, semantic layer organization per source, cross-source relationships, and troubleshooting, see [Multi-Datasource Routing](/deployment/multi-datasource).
</Callout>

To query multiple datasources from a single Atlas deployment, list them in `atlas.config.ts`:

```typescript
import { defineConfig } from "@atlas/api/lib/config";
import { clickhousePlugin } from "@useatlas/clickhouse";
import { snowflakePlugin } from "@useatlas/snowflake";
import { duckdbPlugin } from "@useatlas/duckdb";

export default defineConfig({
  datasources: {
    default: { url: process.env.ATLAS_DATASOURCE_URL! },
  },
  plugins: [
    clickhousePlugin({ url: "clickhouses://user:pass@host:8443/analytics" }),
    snowflakePlugin({ url: "snowflake://user:pass@account/db/schema?warehouse=WH" }),
    duckdbPlugin({ url: "duckdb:///data/reports.duckdb" }),
  ],
  tools: ["explore", "executeSQL"],
  auth: "auto",
  semanticLayer: "./semantic",
});
```

The agent's `executeSQL` tool accepts an optional `connectionId` parameter to target a specific datasource. The `"default"` datasource is used when no ID is specified. Plugins register under their own IDs: `"clickhouse-datasource"`, `"snowflake-datasource"`, `"duckdb-datasource"`, etc.

## Safety configuration

<Callout type="info" title="Using app.useatlas.dev?">
  Safety limits are pre-configured on the hosted platform. You can adjust row limits and query timeouts in **Admin &gt; Settings**.
</Callout>

Atlas enforces several safety limits. Self-hosted operators can tune them via environment variables:

| Variable | Default | Description |
|----------|---------|-------------|
| `ATLAS_TABLE_WHITELIST` | `true` | Only allow queries against tables in `semantic/entities/*.yml` |
| `ATLAS_ROW_LIMIT` | `1000` | Maximum rows returned per query (auto-appended as `LIMIT`) |
| `ATLAS_QUERY_TIMEOUT` | `30000` | Per-query timeout in milliseconds. Enforced via `SET statement_timeout` (PostgreSQL), `MAX_EXECUTION_TIME` (MySQL), `max_execution_time` (ClickHouse), or `STATEMENT_TIMEOUT_IN_SECONDS` (Snowflake) |

Non-SELECT SQL (INSERT, UPDATE, DELETE, DROP, etc.) is always rejected. There is no toggle to disable this.

### Read-only enforcement by datasource

| Datasource | Read-only mechanism |
|------------|---------------------|
| PostgreSQL | SQL validation (regex + AST) |
| MySQL | SQL validation + `SET SESSION TRANSACTION READ ONLY` |
| BigQuery | SQL validation + dry-run cost gate |
| ClickHouse | SQL validation + `readonly: 1` per-query setting |
| Snowflake | SQL validation (use a SELECT-only role for defense-in-depth) |
| DuckDB | SQL validation + `READ_ONLY` open mode |
| Salesforce | SOQL is inherently read-only |

### Recommended production settings

```bash
# .env
ATLAS_TABLE_WHITELIST=true
ATLAS_ROW_LIMIT=500
ATLAS_QUERY_TIMEOUT=15000
```

Lower the row limit to reduce load. Reduce the timeout to kill runaway queries faster.

For multi-tenant deployments, see [Row-Level Security](/security/row-level-security).
