A **Zendesk Guide** connector mirrors your help center's published articles into [Knowledge Base](/guides/knowledge-base) collections. Atlas pulls articles on a schedule, converts each translation's HTML to markdown, and queues every document as a **draft** for review — the same review gate as every other knowledge source. Connectors never publish on their own.

<Callout title="Prerequisites">
The Knowledge Base requires the internal database and managed auth (see the [Knowledge Base guide](/guides/knowledge-base)). Your account needs at least one active brand with a **Guide help center** enabled, and API **token access** enabled in the Zendesk Admin Center. Images and embedded media are **not** mirrored (text-first); they are replaced with links back to the source article.
</Callout>

---

## Create an API token

Atlas authenticates with your Zendesk account email plus an API token (Zendesk's `{email}/token` Basic auth).

<Steps>
<Step>
In Zendesk, go to **Admin Center → Apps and integrations → Zendesk API**, make sure **Token access** is enabled, and add an API token. Give it a label like `atlas-knowledge`.
</Step>
<Step>
Copy the token immediately — Zendesk shows it once.
</Step>
<Step>
Note your account **subdomain** — the `acme` in `acme.zendesk.com`. Atlas only ever talks to `*.zendesk.com` hosts composed from that subdomain.
</Step>
</Steps>

## Install

Navigate to **Admin → Integrations**, choose **Knowledge Base (Zendesk Guide)**, and provide:

- **Zendesk subdomain** — the `acme` in `acme.zendesk.com` (pasting the full URL works too).
- **Zendesk account email** — the account the API token belongs to.
- **API token** — stored encrypted at rest; never shown again.

At install time Atlas verifies the credentials and enumerates your account's **brands**. A bad token, a wrong subdomain, or an account with no help-center-enabled brand fails the install **loudly** with an actionable message. On success, collections are created and the first sync is scheduled.

### One collection per brand

Zendesk is multi-brand, and Atlas mirrors that: the install discovers every **active brand with a Guide help center** and creates one collection per brand, so each brand's articles are independently searchable with their own review queue.

- An account with one help-center-enabled brand gets one collection under the plain slug (default `zendesk`).
- With several help-center-enabled brands, each collection is suffixed with its brand's subdomain (`zendesk-acme`, `zendesk-support`).

<Callout type="warn" title="Adding a brand later">
Brands added (or Guide-enabled) after install are not discovered automatically — **re-install the connector** to fan out to new brands (the install is idempotent; existing collections are unaffected). One edge to know: if an account goes from one help-center-enabled brand to several, the re-install creates brand-suffixed collections and the original plain-slug collection stays behind — uninstall it by hand.
</Callout>

## How sync works

Atlas runs two kinds of cycle, decided per collection:

- **Incremental** (the common, cheap cycle) — uses Zendesk's **native incremental article feed**, which reports articles changed since the collection's high-water mark; Atlas then fetches the changed articles' translations. Zendesk caps this feed at **10 requests/minute** — Atlas honors the `429` / `Retry-After` signals with bounded backoff rather than hammering it. Deletions are invisible here.
- **Reconciliation** (the correctness anchor, weekly by default, and always on the first sync) — enumerates every published article with its translations. This is the only cycle that **archives** articles that were deleted or unpublished (set back to draft in Zendesk), and the only one that validates the ingest caps over the full set.

Each collection card shows the last sync's status and time, with a **Sync now** button for an on-demand pull. If a crawl comes back knowingly incomplete (Zendesk returned malformed entries), Atlas upserts what it got but defers all archiving until a clean crawl — deletions are never inferred from a partial view.

### What syncs — published articles, per language

Only **published** content syncs: a draft article — or a draft translation of a published article — is never pulled. Each published translation is its own document, filed under its locale (`zendesk/en-us/getting-started-123.md`, `zendesk/de/erste-schritte-123.md`), so a multi-language help center mirrors as a per-locale tree.

### Paths, and what a rename does

A document's path is built from its **locale, title, and article id** (`zendesk/en-us/reset-your-password-4567.md`). Two consequences:

- **Renaming an article's title archives the old path and re-drafts the new one** for re-review — expected churn; nothing is ever hard-deleted.
- **Moving an article between sections or categories does *not* change its path** — the Guide category/section tree is not mirrored into document paths in this version.

### HTML, images, and honest conversion

Article HTML converts structurally — headings, lists, tables, code blocks, quotes, details/summary. Images, videos, iframes, and other embeds **degrade to visible placeholders that link back to the source article** and are counted — never silently dropped (text-first). Relative links between help-center articles are rewritten to absolute URLs against the brand's host, so cross-references in mirrored documents stay live.

## Review and publish

Every synced article lands as a **draft**. Review it in **Admin → Knowledge Base**, and publish when it's ready — only then does the agent read it. An article whose content changes in Zendesk and had already been published is **demoted back to draft** on the next sync, so an edit re-enters review rather than silently going live. Provenance (`atlas:` — vendor, brand, article id, locale, last-modified time) travels with every document so any answer traces back to its help-center article.

---

## Caps and limits

A very large help center can overflow the Knowledge Base ingest caps. When that happens, reconciliation fails with the **actual document count** (one per published translation — a 700-article, 3-locale help center is ~2100 documents) and names the knob to raise:

- `ATLAS_KNOWLEDGE_INGEST_MAX_DOCS` — maximum documents per collection (default 1000).
- `ATLAS_KNOWLEDGE_INGEST_MAX_DOC_BYTES` — maximum size of a single document. An oversized document is rejected individually (visible in the sync report) rather than failing the cycle, and its previously-synced version is never archived out from under it.

Raise the cap in the settings registry, or keep brands in separate collections (the default) so each stays under it. The reconciliation cadence itself is the `ATLAS_KNOWLEDGE_SYNC_RECONCILE_INTERVAL_HOURS` knob (weekly default); the incremental cadence is `ATLAS_KNOWLEDGE_SYNC_INTERVAL_HOURS` (nightly default).

---

## Uninstall

Uninstalling a Zendesk collection **archives** its documents (never hard-deletes) and removes the stored token and sync bookkeeping for that brand. Re-installing re-syncs it, which brings the archived articles back as drafts for re-review.
