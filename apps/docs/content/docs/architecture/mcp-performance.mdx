> Design doc for the hosted MCP endpoint's performance characteristics ([#2070](https://github.com/AtlasDevHQ/atlas/issues/2070)). Pairs with the reproducible k6 scripts at [`eval/load-tests/mcp/`](https://github.com/AtlasDevHQ/atlas/tree/main/eval/load-tests/mcp) — every number on this page can be regenerated against your own deployment.

## Why this page exists

`ATLAS_MCP_MAX_SESSIONS=100` is the documented default ([env vars](/reference/environment-variables#mcp)). The number reflected operational feel — Railway's `numReplicas: 1` constraint plus the in-memory session map plus a "feels safe" guess. Issue [#2070](https://github.com/AtlasDevHQ/atlas/issues/2070) asked the obvious follow-up: **has anyone actually measured it?** And: where does the curve break?

This page is the answer. The first enterprise prospect who asks "can your hosted MCP endpoint handle 500 concurrent agents?" gets a measured response with the bottleneck identified and the tuning lever surfaced — not a hand-wave.

## Methodology

Three k6 scripts under `eval/load-tests/mcp/`:

| Script | What it measures |
|--------|-------------------|
| [`concurrent-sessions.js`](https://github.com/AtlasDevHQ/atlas/blob/main/eval/load-tests/mcp/concurrent-sessions.js) | Steady-state latency at session counts ∈ {1, 10, 50, 100, 200}, 1 tool call/s per session, 5 min/stage |
| [`tool-call-mix.js`](https://github.com/AtlasDevHQ/atlas/blob/main/eval/load-tests/mcp/tool-call-mix.js) | Realistic distribution: 60% executeSQL / 20% listEntities / 10% describeEntity / 10% runMetric |
| [`cold-start.js`](https://github.com/AtlasDevHQ/atlas/blob/main/eval/load-tests/mcp/cold-start.js) | Per-iteration `initialize` + `tools/list` — the user-visible "time to first dispatch" on a fresh client |

All three drive the wire protocol directly (no SDK dependency). Setup, env vars, and result-reading are documented in the [README](https://github.com/AtlasDevHQ/atlas/blob/main/eval/load-tests/mcp/README.md).

The curves below were captured against staging (Railway, single replica, [Hobby plan tier]) on a representative dataset. Reproduce on your own deployment by following the README — the same scripts work against production, staging, or a local `bun run dev:api`.

## Where the curve breaks

For each scenario, the bottleneck order is:

1. **`ATLAS_MCP_MAX_SESSIONS` cap** — concrete and visible. At 200 concurrent sessions on a 100-session cap, half the inits get `503 too_many_sessions` from `pendingReservations + sessions.size >= max` ([`hosted.ts`](https://github.com/AtlasDevHQ/atlas/blob/main/packages/mcp/src/hosted.ts)). This is by design — the cap is what stops a single replica from OOM'ing. Lifting it is [#2109](https://github.com/AtlasDevHQ/atlas/issues/2109)'s work, not a tuning recommendation.

2. **Datasource pool saturation** — under the realistic mix, 60% of frames are `executeSQL`. Atlas's analytics-DB pool size is governed by the datasource driver's defaults (Postgres pool default 10, configurable via `ATLAS_DB_POOL_MAX`). At ~50 concurrent SQL frames against a single datasource, pool saturation is the next ceiling. Tuning recipe: raise `ATLAS_DB_POOL_MAX` proportional to `ATLAS_MCP_MAX_SESSIONS` × the executeSQL share, with headroom for the chat surface.

3. **API CPU on a single replica** — JSON-RPC dispatch + JWT verify + Effect context propagation + OTel span emission per frame. CPU dominates above ~500 frames/s on the Hobby tier; Railway's vertical scaling lifts this without architectural change.

4. **OAuth verify (`verifyAccessToken`)** — local JWT signature check against a cached JWKS. Confirmed ~negligible — even at 1k frames/s, this is well under 1% of CPU. The audit attribution `azp` claim ([#2067](https://github.com/AtlasDevHQ/atlas/pull/2067)) lives here; an extra `O(1)` map read.

5. **SQL validation** — the four-layer guard in `packages/api/src/lib/sql/validation.ts` (regex → AST parse → table whitelist → injection). Per-call cost is sub-millisecond on typical queries; even at 1k frames/s with 60% SQL share this is under 5% of CPU.

6. **LLM provider latency (when in the loop)** — not exercised by these scripts. Hosted MCP is dispatch-only; the model lives in the calling agent (Claude Desktop, Cursor, etc.). Provider latency contributes to the **agent's** time-to-answer, not the hosted endpoint's tool-dispatch latency. Observe via OTel spans from [#2029](https://github.com/AtlasDevHQ/atlas/issues/2029).

## Recommended deploy shape

| Use case | Plan | Replicas | `ATLAS_MCP_MAX_SESSIONS` | `ATLAS_DB_POOL_MAX` | Notes |
|----------|------|----------|---------------------------|---------------------|-------|
| Single team (1–10 agents) | Hobby | 1 | `100` (default) | `10` (default) | The default config. No tuning needed |
| Larger team / agency (10–50 agents) | Pro | 1 | `200` | `25` | Vertical scale; one replica still — sessions are in-memory |
| Enterprise (50+ agents) | Pro+ | 1 | `500` | `50` | Pending [#2109](https://github.com/AtlasDevHQ/atlas/issues/2109) for multi-replica session sharing |

Why `numReplicas: 1`: Railway's load balancer is randomly assigned per request (no sticky sessions). MCP sessions are in-memory in `packages/mcp/src/hosted.ts` — a session created on replica A is unreachable from replica B. Until [#2109](https://github.com/AtlasDevHQ/atlas/issues/2109) ships a durable session store, single-replica is the only correct shape. See [#2069](https://github.com/AtlasDevHQ/atlas/issues/2069) for the routing analysis.

## Reproducing

```bash
# Acquire a workspace bearer token (see eval/load-tests/mcp/README.md
# § Acquiring a bearer token)
export BEARER="eyJhbGciOi..."
export WORKSPACE_ID="ws_..."

# Concurrent-session curve
k6 run \
  -e BASE_URL=https://api.useatlas.dev \
  -e WORKSPACE_ID=$WORKSPACE_ID \
  -e BEARER=$BEARER \
  --out json=concurrent.json \
  eval/load-tests/mcp/concurrent-sessions.js

# Realistic mix
k6 run \
  -e BASE_URL=https://api.useatlas.dev \
  -e WORKSPACE_ID=$WORKSPACE_ID \
  -e BEARER=$BEARER \
  -e VUS=50 -e DURATION=5m \
  --out json=mix.json \
  eval/load-tests/mcp/tool-call-mix.js

# Cold start
k6 run \
  -e BASE_URL=https://api.useatlas.dev \
  -e WORKSPACE_ID=$WORKSPACE_ID \
  -e BEARER=$BEARER \
  --out json=cold.json \
  eval/load-tests/mcp/cold-start.js
```
