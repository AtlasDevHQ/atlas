The built-in **OpenAPI (Generic REST)** datasource (`openapi-generic`) lets you point Atlas at any REST API that publishes an OpenAPI 3.x spec ‚Äî Twenty, Stripe, an internal microservice ‚Äî and have the agent discover its operations and query them directly. No per-API client to hand-code: paste the spec URL and a credential, and the agent reads the operation surface from the spec.

It installs from **Admin ‚Üí Connections** like any other datasource (the catalog row is code-seeded into every Atlas deployment). Credentials are encrypted at rest in `workspace_plugins.config`.

<Callout type="info">
A workspace can install several OpenAPI datasources side by side (one for Twenty, one for Stripe, ‚Ä¶). Each is its own install with its own credential, write allowlist, and rate limit.
</Callout>

## Reading

GET (and HEAD) operations execute immediately. The agent calls `executeRestOperation` with an `operationId` from the spec plus its parameters, and gets the response back as data. Multi-step questions ("show me Matt's notes") run as a sequence of operations across agent steps.

<Callout type="warn">
Read vs write is classified by HTTP **method** by default ‚Äî but only as a default, not ground truth. Some legacy / RPC-style APIs **mutate state through a GET** (e.g. `GET /jobs/{id}/cancel`, `GET /admin/resetPassword`). If yours does, you **must** mark those operations side-effecting so Atlas gates them like writes ‚Äî see [Side-effecting GETs](#side-effecting-gets) below. An unmarked mutating GET runs as an unconfirmed read.
</Callout>

## Writing ‚Äî opt-in, then confirm

**Writes are off by default.** A non-GET operation (POST / PATCH / PUT / DELETE) only runs if its `operationId` is in that install's **write allowlist** ‚Äî and even then, never silently.

### 1. Allowlist the operations you want to permit

The `write_allowlist` config field is a JSON array of `operationId`s permitted to execute a write:

```json
["createOnePerson", "createOneNote"]
```

Anything not listed is denied. An empty or omitted allowlist means the datasource stays read-only. A malformed value fails **closed** (read-only) ‚Äî a typo never opens up writes.

### 2. The agent stages the write; the user confirms it

When the agent decides to call an allowlisted write, `executeRestOperation` does **not** dispatch it. It returns a `needs_confirmation` result, and the chat surface renders a **confirm-before-write banner** describing exactly what will happen:

> **POST** ‚Äî createOnePerson ‚Äî POST /people on Twenty
>
> This will write to **Twenty**. It has not run yet ‚Äî confirm to proceed, or cancel to leave it un-run.
>
> `[ Confirm write ]` `[ Cancel ]`

The write fires only when the user clicks **Confirm write** ‚Äî the request is POSTed to `/api/v1/rest-operations/confirm`, which re-validates the operation against the allowlist server-side before executing. The agent cannot self-confirm, and a non-allowlisted operation is refused at the confirm endpoint too (defense in depth). **Cancel** leaves the write un-run.

The staged write carries a short-lived, server-signed, **single-use** confirm token binding it to that exact `(workspace, datasource, operation, parameters)`. The confirm endpoint verifies the token and **burns it** ‚Äî so the human-in-the-loop step is enforced server-side, not just in the UI: a replayed, tampered, expired, or foreign-workspace confirm is rejected, and the same approval can never fire a second write. If a confirmation expires before the user acts, ask the agent to retry so it re-stages a fresh one. The token lifetime defaults to 10 minutes (`ATLAS_OPENAPI_CONFIRM_TTL_SECONDS`).

<Callout type="warn">
Prompting a write that is **not** allowlisted (e.g. "delete every Person") returns `writes_disabled` ‚Äî the request never fires, and the agent is told not to claim it happened. Add the operation to `write_allowlist` to enable it.
</Callout>

## Side-effecting GETs

Atlas classifies read vs write by HTTP method **as a default, not ground truth**. A GET or HEAD that actually mutates state (a "mutating RPC-over-GET", common in legacy / internal services) can be flagged side-effecting, and is then forced through the **same write allowlist + confirm path** as a POST ‚Äî it must be allowlisted, and it stages for confirmation rather than running silently.

There are two ways to flag one, and they combine (either is enough). The classification is **monotonic**: a flag can only ever *escalate* a read to a write, never downgrade a write method to a read.

**1. In the spec ‚Äî `x-atlas-side-effecting: true`.** If you control the OpenAPI document, mark the operation directly:

```yaml
paths:
  /jobs/{id}/cancel:
    get:
      operationId: cancelJob
      x-atlas-side-effecting: true   # this GET mutates ‚Äî gate it like a write
```

Only an explicit boolean `true` escalates; `false` (or absent) leaves classification to the method. A present-but-non-boolean value (e.g. the string `"true"`) is **rejected at probe time** ‚Äî a mistyped flag fails the install loudly rather than silently leaving the GET ungated.

**2. In the install config ‚Äî `side_effecting_operations`.** When you don't own the spec, list the offending `operationId`s as a JSON array on the install:

```json
["cancelJob", "resetPassword"]
```

A malformed value degrades to "none" (classification stays method-only) and is logged. Note this is **not** fail-closed the way `write_allowlist` is: an empty or broken side-effecting list leaves a mutating GET running unconfirmed, so fix a logged parse warning promptly.

<Callout type="warn">
This is an **operator-validated security boundary**. Atlas cannot know which of an API's GETs mutate state ‚Äî you must. If a GET on your API changes data and you don't flag it (either way above), the agent will run it as an unconfirmed read.
</Callout>

## The safety stack (`validateRestOperation`)

Every operation ‚Äî read or write ‚Äî is authorized by `validateRestOperation`, the REST sibling to Atlas's SQL validator. Five layers, in order:

1. **Operation in graph** ‚Äî an `operationId` not in the probed spec is rejected (`unknown_operation`); Atlas never dispatches a fabricated operation.
2. **Write allowlist** ‚Äî GET/HEAD pass **unless flagged side-effecting** (see above); any other method, or a side-effecting GET/HEAD, requires the `operationId` in `write_allowlist`, else `writes_disabled` (default-deny).
3. **Parameter shape** ‚Äî required parameters and request body must be present; parameters the spec doesn't declare are rejected (`invalid_params`).
4. **Rate limit** ‚Äî a per-`(workspace, datasource, operation)` token bucket, **60 calls/min** by default, overridable per install via the config-only `rate_limit_per_minute` key. Throttles runaway agent loops on a paid upstream (`rate_limited`, with a retry hint).
5. **Timeout** ‚Äî each request is capped at `ATLAS_OPENAPI_TIMEOUT` (default **30s**). A per-install `request_timeout_ms` (config-only) above the cap is rejected.

Writes are **never written to any cache** ‚Äî the write path uses the un-cached client primitive, and the paginating read cache refuses to store non-GET responses by construction.

## Keeping the spec current

Atlas caches the probed spec per install. When the upstream API ships new operations or changes a shape, the cache needs to be re-discovered so the agent's view stays accurate. Two ways to refresh it:

- **Refresh now** ‚Äî the **Rediscover schema** button on the datasource card (Admin ‚Üí Connections) re-probes the spec endpoint immediately and replaces the cached snapshot. Use this right after the upstream API changes.
- **Auto-refresh interval** ‚Äî the **Auto-refresh** control sets how often Atlas re-discovers the spec on its own:

| Value | Meaning |
|-------|---------|
| `off` (default) | Never auto-refresh ‚Äî re-discover manually with **Rediscover schema**. |
| `daily` | Re-discover every 24 hours. |
| `weekly` | Re-discover every 7 days. |
| `<N>h` | A custom interval of `N` hours, clamped to **1h ‚Ä¶ 720h** (30 days). Set via `atlas.config.ts`. |

The interval is stored per install (it's not a secret, so it's a plain `workspace_plugins.config` field). An out-of-range custom value is **clamped** into the bounds; an unparseable value is **rejected** with a message ‚Äî it never silently falls back to `off`.

A background scheduler walks due installs on their per-install interval, re-probes the spec, and updates the cached snapshot unattended ‚Äî so once you set `daily` / `weekly` / `<N>h`, the agent's view stays current without anyone clicking **Rediscover schema**. A failed scheduled probe never degrades the live snapshot.

### Query-time drift recovery (`spec_drift_mode`)

Both paths above are time-based. There's a third, **demand-driven** knob for the moment drift actually bites ‚Äî the agent calls an `operationId` that the upstream added (or renamed to) *after* the last discovery, and the cached snapshot doesn't know it yet:

| Value | Meaning |
|-------|---------|
| `strict` (default) | An unknown `operationId` is rejected until the next manual / scheduled rediscovery. |
| `auto-refresh` | On an unknown `operationId`, Atlas re-probes the spec **once**, and retries the call if the fresh spec contains the operation. |

In `auto-refresh` mode the re-probe runs through the same egress-guarded discovery path as **Rediscover schema** (SSRF guard, credential host-match gate), is **rate-limited per install** (at most one re-probe per minute, so a hallucinated `operationId` can't hammer your upstream), and **fails closed** ‚Äî if the probe fails, the old snapshot stays and the call is rejected as before. If the fresh spec moved its `servers[0].url`, the retry follows the new base URL only after it re-passes the egress guard; a blocked target keeps the previously-validated base. If the re-probe finds **breaking** drift, the new snapshot is still adopted (the agent's valid call succeeds) but the [breaking-change alert](#breaking-change-alerts) is raised exactly as a scheduled refresh would raise it ‚Äî a breaking change is never adopted silently.

Set it via `PATCH /api/v1/admin/openapi-datasources/{installId}` (`{"specDriftMode": "auto-refresh"}`) or as `spec_drift_mode` on the install's `config` in `atlas.config.ts`. The default is `strict`: a re-probe is real upstream traffic triggered by agent input, so each install opts in explicitly.

### Breaking-change alerts

When a **scheduled** re-discovery finds that the upstream spec **removed or retyped** something the agent relied on ‚Äî an operation disappeared, an operation re-routed (method/path/auth changed) or became side-effecting, a field the agent reads or sends was removed/retyped, or a new **required** request field appeared ‚Äî Atlas raises a **breaking-change alert** on the datasource card (Admin ‚Üí Connections) and writes an audit-log entry (`connection.spec_drift_breaking`). The point is that you learn your upstream API changed under you *before* the agent's calls start failing.

Purely **additive** drift (new operations, new optional params, new response fields) stays quiet ‚Äî it's surfaced in the **Last refresh** line but doesn't raise the alert.

The alert is **dismissible**: click **Acknowledge** on the card to clear it. It also clears on its own on the next **clean** refresh (manual or scheduled) ‚Äî i.e. once the upstream stops changing, the "all good now" refresh removes it. A manual **Rediscover schema** never raises the persisted alert (you're already looking at the inline diff), but a clean manual refresh still clears a standing one.

<Callout type="info">
Breaking-vs-additive is a heuristic over the structured diff, bounded by the same field-walk depth the diff uses ‚Äî treat the alert as "worth a look", not a proof. Always confirm against the upstream API's own changelog.
</Callout>

## Configuration fields

Install-form fields (Admin ‚Üí Connections):

| Field | Required | Notes |
|-------|----------|-------|
| `openapi_url` | yes | URL of the OpenAPI 3.x document, e.g. `https://crm.example.com/rest/open-api/core` |
| `auth_kind` | yes | `none` / `bearer` / `basic` / `apikey-header` / `apikey-query` (oauth2 is coming soon) |
| `auth_value` | ‚Äî | Bearer token, API key, or `username:password`. Encrypted at rest. |
| `auth_header_name` | ‚Äî | Header name for `apikey-header`, e.g. `X-API-Key` |
| `auth_param_name` | ‚Äî | Query param for `apikey-query`, e.g. `api_key` |
| `base_url_override` | ‚Äî | When the spec's `servers[0].url` is wrong (dev/staging). **Security:** the configured `auth_value` is sent to whatever host this resolves to ‚Äî only point `base_url_override` at a host the same credential is valid for, or you risk leaking the credential to an unintended host. |
| `write_allowlist` | ‚Äî | JSON array of `operationId`s permitted to write. Empty = read-only (default). |
| `side_effecting_operations` | ‚Äî | JSON array of `operationId`s whose GET/HEAD **mutates** state ‚Äî forced through the write allowlist + confirm path. Empty = method-only classification (default). See [Side-effecting GETs](#side-effecting-gets). |
| `display_name` | ‚Äî | Friendly name shown in Admin ‚Üí Connections |

Advanced overrides (set on the install's `config` via `atlas.config.ts`, not surfaced in the install form):

| Key | Notes |
|-----|-------|
| `rate_limit_per_minute` | Per-operation rate limit override (default 60) |
| `request_timeout_ms` | Per-request timeout override; must be ‚â§ `ATLAS_OPENAPI_TIMEOUT` (the cap), else the operation is rejected |
| `spec_refresh_interval` | How often to auto-re-discover the spec: `off` (default) / `daily` / `weekly` / `<N>h` (1h‚Äì720h, clamped). Also settable from the **Auto-refresh** control on the datasource card. |
| `spec_drift_mode` | Query-time drift handling: `strict` (default ‚Äî unknown operations are rejected until the next rediscovery) or `auto-refresh` (re-probe once and retry when the agent hits an operation the cached spec doesn't know). See [Query-time drift recovery](#query-time-drift-recovery-spec_drift_mode). |

## Pre-wired vendor datasources

Some vendors ship as their own cards in Admin ‚Üí Connections, pre-wired over this same generic primitive ‚Äî the spec URL and auth kind are filled in for you, so you only supply the credential. They are not separate clients: the operation graph, validator, paginator, and client are the same generic machinery.

| Card | Auth | What you supply |
|------|------|-----------------|
| **Stripe** (`stripe-data`) | Bearer | Your Stripe secret key |
| **GitHub** (`github-data`) | **OAuth2 (GitHub App)** | Nothing to paste ‚Äî connect your GitHub App installation |

### GitHub (`github-data`) ‚Äî OAuth2 via GitHub App

GitHub connects through **OAuth2**, not a pasted token. It reuses the **same GitHub App** as the GitHub *action* integration, so a deploy already wired for GitHub gets the datasource for free. Clicking **Connect** runs the GitHub App install dance; Atlas stores the resulting `installation_id` (encrypted) and mints a short-lived installation access token on demand at query time (re-minted automatically on ~1-hour expiry ‚Äî there is no long-lived token to rotate or leak). GitHub's list endpoints are paged via the standard `Link` header.

Operators must set the GitHub App env for the handler to register (the same variables the GitHub action integration uses):

| Variable | Notes |
|----------|-------|
| `GITHUB_APP_ID` / `GITHUB_APP_SLUG` | App identity (the `iss` claim + the `github.com/apps/<slug>` install URL) |
| `GITHUB_APP_PRIVATE_KEY` | PKCS8 PEM ‚Äî signs the App JWT used to mint installation tokens |
| `GITHUB_APP_CLIENT_ID` / `GITHUB_APP_CLIENT_SECRET` | User-OAuth-during-install credentials ‚Äî used to verify the installing user owns the installation (cross-tenant binding guard) |
| `ATLAS_PUBLIC_API_URL` | The API origin, so the OAuth callback URL resolves |

If the App access is later revoked, the next query simply fails to mint a token and the datasource is skipped ‚Äî reconnect from Admin ‚Üí Connections to restore it.

> The generic form's own `oauth2` `auth_kind` (above) remains deferred ‚Äî `github-data`'s OAuth lives in its dedicated install handler, not the generic credential form.
