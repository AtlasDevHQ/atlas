title: Semantic Expert Agent
description: Automatically analyze and improve your semantic layer using data profiling, audit log patterns, and validated proposals.
---

The semantic expert agent examines your database structure, query history, and semantic layer definitions to identify improvement opportunities. It proposes validated changes that go through the standard admin review queue.

## How It Works

The expert agent runs 9 analysis categories:

| Category | What it checks |
|----------|---------------|
| **Coverage gaps** | Database columns not represented as entity dimensions |
| **Description quality** | Empty or auto-generated descriptions on tables and dimensions |
| **Type accuracy** | Dimension types that don't match actual database column types |
| **Missing measures** | Aggregatable numeric columns with no defined measures |
| **Missing joins** | Foreign key relationships not captured in entity joins |
| **Glossary gaps** | Business abbreviations in column names not defined in glossary |
| **Sample value staleness** | Declared sample values that no longer appear in the data |
| **Query pattern coverage** | Frequently-asked queries not captured in query_patterns |
| **Virtual dimension opportunities** | Common SQL expressions (EXTRACT, DATE_TRUNC) that should be virtual dimensions |

Each finding is scored by `impact x confidence x (1 - staleness)` to prioritize the most valuable improvements.

## Running `atlas improve`

The `atlas improve` CLI command runs the analysis engine in batch mode:

```bash
# Analyze and show proposals (dry run — default)
atlas improve

# Apply changes directly to entity YAML files
atlas improve --apply

# Filter by minimum confidence
atlas improve --min-confidence 0.7

# Limit to specific entities
atlas improve --entities orders,users,products

# Only analyze audit log entries after a date
atlas improve --since 2026-03-01

# Use a specific datasource subdirectory
atlas improve --source warehouse
```

### Requirements

- `ATLAS_DATASOURCE_URL` must be set (PostgreSQL required)
- Semantic layer must exist (run `atlas init` first)
- `DATABASE_URL` is optional but enables audit log analysis

### CLI Flags

| Flag | Description | Default |
|------|-------------|---------|
| `--apply` | Write changes to YAML files | Dry run |
| `--min-confidence <n>` | Minimum confidence threshold (0-1) | 0.5 |
| `--entities <list>` | Comma-separated entity names to analyze | All |
| `--since <date>` | Analyze audit log entries after this date | All time |
| `--source <name>` | Read from `semantic/{name}/` subdirectory | `semantic/` |
| `--schema <name>` | PostgreSQL schema | `public` |

## Reviewing Proposals in the Admin UI

When the expert agent runs (via CLI or API), proposals are written to the `learned_patterns` table with `type = semantic_amendment`. Review them in the admin UI:

1. Navigate to **Admin > Learned Patterns**
2. Use the **Type** filter to select "Amendments"
3. Click a proposal to see the YAML diff with color-coded additions (green) and removals (red)
4. **Approve** to apply the change, **Reject** to suppress re-proposal for 30 days

Bulk approve/reject is supported — select multiple proposals and use the toolbar buttons.

## Auto-Approval Threshold

By default, all proposals require manual review. To auto-approve high-confidence proposals, set:

```bash
ATLAS_EXPERT_AUTO_APPROVE_THRESHOLD=0.9
```

Proposals with `confidence >= 0.9` will be automatically approved and applied. Set to any value above 1.0 to disable (the default behavior).

<Callout type="warn">
  Auto-approval is powerful but should be used carefully. Start with a high threshold (0.9+) and monitor the results before lowering it.
</Callout>

## Architecture

The expert agent has 5 dedicated tools:

- **profileTable** — Get column statistics (cardinality, nulls, sample values)
- **checkDataDistribution** — Analyze a column's value distribution
- **searchAuditLog** — Find query patterns from actual usage
- **proposeAmendment** — Create a structured YAML change proposal
- **validateProposal** — Dry-run validate a proposal (YAML parse, whitelist, test query)

These are registered in a dedicated `ToolRegistry` alongside the standard `explore` and `executeSQL` tools, giving the expert agent 7 tools total.

## Interactive Mode

Interactive mode lets you review proposals one at a time with colorized YAML diffs, approving or rejecting each change inline.

```bash
atlas improve -i
```

### Interactive Session Flow

1. The analysis engine runs (same as batch mode) and ranks all proposals
2. Each proposal is presented with a colorized unified diff showing the YAML change
3. You choose an action for each proposal:
   - **y** (yes) — apply the change to the YAML file immediately
   - **n** (no) — reject the change (skipped for this session)
   - **s** (skip) — skip for now (may appear again next run)
   - **q** (quit) — end the session early
4. A session summary is printed at exit showing accepted/rejected/skipped counts

### Example Session


You can combine interactive mode with other flags:

```bash
# Interactive mode with high-confidence filter
atlas improve -i --min-confidence 0.7

# Interactive mode for specific entities
atlas improve -i --entities orders,users
```

## Version Tracking with `atlas migrate`

Changes made by `atlas improve` (both batch and interactive mode) are automatically tracked via semantic layer snapshots. Before applying any amendments, a snapshot of the current state is captured so you can review changes and roll back if needed.

### Key Commands

```bash
# Check current state vs last snapshot
atlas migrate status

# Create a manual snapshot with a description
atlas migrate snapshot -m "Before quarterly review"

# See what changed since last snapshot
atlas migrate diff

# View snapshot history
atlas migrate log

# Rollback to a previous state
atlas migrate rollback <hash>
```

### How It Integrates

- **`atlas init`** — creates an initial snapshot after generating the semantic layer
- **`atlas improve --apply`** — auto-snapshots before applying batch amendments
- **`atlas improve -i`** — auto-snapshots before starting an interactive session
- **`atlas migrate rollback`** — auto-snapshots before restoring (so rollbacks are reversible)

Snapshots are stored in `semantic/.history/` as JSON files. Each snapshot captures all entity YAMLs, glossary, metrics, and catalog at that point in time. Use `atlas migrate log` to see the full history with hashes, timestamps, and trigger labels.

See the [CLI reference](/docs/reference/cli#migrate) for the full `atlas migrate` documentation.

## Web Interactive Mode

The admin UI includes a dedicated improvement page with a split-view layout: a chat panel on the left for conversing with the expert agent, and a proposals panel on the right showing pending changes with approve/reject buttons.

### Getting Started

1. Navigate to **Admin > Semantic Layer** and click the **Improve** button in the header, or go directly to **Admin > Data > Improve Layer** in the sidebar
2. Click **Run Analysis** for autonomous mode, or type a message to guide the agent toward specific areas (e.g., "Focus on the orders table" or "Check for missing joins")
3. The agent will analyze your semantic layer and present proposals as they're identified

### Proposal Cards

Each proposal appears as a card showing:

- **Entity name** and amendment type badge
- **Rationale** explaining why the change improves the semantic layer
- **Amendment preview** with the structured data that will be added/modified
- **Confidence score** (color-coded: green ≥80%, yellow ≥50%, red <50%)
- **Approve / Reject** buttons

### Session Persistence

Improvement sessions are preserved across page navigations. The session ID is tracked in the `x-session-id` response header from the streaming endpoint. When you return to the improve page, the agent maintains context about which proposals have been reviewed.

### API Endpoints

The web interactive mode uses these admin API endpoints:

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/api/v1/admin/semantic-improve/chat` | Streaming chat with the expert agent |
| `GET` | `/api/v1/admin/semantic-improve/sessions` | List improvement sessions |
| `GET` | `/api/v1/admin/semantic-improve/sessions/:id` | Get session with proposals |
| `POST` | `/api/v1/admin/semantic-improve/proposals/:id/approve` | Apply a proposal |
| `POST` | `/api/v1/admin/semantic-improve/proposals/:id/reject` | Reject a proposal |

### Integration with Version History

Approved amendments are recorded in the semantic editor's version history with `author: "Semantic Expert Agent"`. You can view and rollback expert-agent changes from the entity's History tab in the semantic editor.

## Scheduled Improvements (Phase 4)

The expert agent can run on a schedule, automatically analyzing your semantic layer and proposing improvements without manual intervention.

### Enabling the Scheduler

Set these environment variables or configure through the admin Settings page under **Intelligence**:

| Setting | Env var | Default | Description |
|---------|---------|---------|-------------|
| Expert Scheduler | `ATLAS_EXPERT_SCHEDULER_ENABLED` | `false` | Enable periodic analysis |
| Schedule Interval | `ATLAS_EXPERT_SCHEDULER_INTERVAL_HOURS` | `24` | Hours between runs |
| Auto-Approve Threshold | `ATLAS_EXPERT_AUTO_APPROVE_THRESHOLD` | (disabled) | Proposals with confidence ≥ this value are auto-applied |
| Auto-Approve Types | `ATLAS_EXPERT_AUTO_APPROVE_TYPES` | `update_description,add_dimension` | Comma-separated amendment types eligible for auto-approval. Types not in this list always queue for review |

### Auto-Approval

When `ATLAS_EXPERT_AUTO_APPROVE_THRESHOLD` is set (e.g., `0.9`), proposals are auto-approved only if they meet **both** conditions: confidence at or above the threshold, **and** the amendment type is in the `ATLAS_EXPERT_AUTO_APPROVE_TYPES` list. By default only low-risk types (`update_description`, `add_dimension`) are eligible — higher-risk types like `add_join`, `add_measure`, and `update_dimension` always queue for human review regardless of confidence.

Auto-applied amendments:

- Are recorded in the semantic editor version history
- Create a snapshot via `atlas migrate` (if available)
- Are logged at info level for audit trail

**Recommendation:** Start with a high threshold (0.9+) and only for low-risk amendment types like description improvements.

### Notification Badge

When scheduled runs produce proposals that need human review (below the auto-approve threshold), a numeric badge appears on the **Improve Layer** item in the admin sidebar. The badge updates every 60 seconds.

### Health Score Widget

The semantic editor page shows a health score widget with four sub-scores:

- **Coverage** — fraction of database columns represented as dimensions
- **Descriptions** — fraction of dimensions with meaningful (non-auto-generated) descriptions
- **Measures** — fraction of entities with at least one measure defined
- **Joins** — fraction of detected foreign key relationships captured as entity joins

The overall score is a weighted average. Use this to track your semantic layer quality over time.
