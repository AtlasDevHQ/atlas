<Callout type="info" title="Self-hosted only">
  The demo dataset is for self-hosted local development and evaluation. On [app.useatlas.dev](https://app.useatlas.dev), the onboarding wizard offers a single "Use demo data" option when you create your workspace — no CLI needed.
</Callout>

Atlas ships a single canonical demo dataset: **NovaMart**, a direct-to-consumer e-commerce brand. 13 entities (orders, products, customers, payments, returns, shipments, sellers, …), 52 tables total, ~480K rows spanning 2020-2025.

NovaMart was founded during the pandemic, started with bedding, expanded to kitchen/bath/outdoor, and launched a small marketplace in 2022. The schema includes the same kinds of real-world tech-debt patterns you'd hit in a real codebase — missing FK constraints, abandoned tables, inconsistent enums, denormalized reporting tables, and schema-evolution artifacts where old and new columns coexist.

<Callout type="info" title="Single canonical seed since 1.4.0">
  Earlier releases shipped three demo seeds (`simple`, `cybersec`, `ecommerce`). 1.4.0 (#2021) collapsed the picker to a single canonical seed — the e-commerce one. Legacy `--demo simple` and `--demo cybersec` invocations now error with a migration message; `--seed` was removed entirely.
</Callout>

## Loading the demo

Requires PostgreSQL (the seed uses `GENERATE_SERIES` for data generation).

```bash
# Start local Postgres + sandbox sidecar
bun run db:up

# Load the demo seed and generate the semantic layer
bun run atlas -- init --demo

# Start Atlas (containers already running from db:up)
bun run dev
```

To reset and reload from scratch:

```bash
bun run db:reset
bun run atlas -- init --demo
```

## Canonical questions

These eight questions exercise the full surface of the semantic layer — simple metrics, segmentation, joins, time-series, virtual dimensions, glossary disambiguation, filtered patterns, and normalized enum aggregation. They're the same questions the eval harness ([#2025](https://github.com/AtlasDevHQ/atlas/issues/2025)) runs against the demo and the public-surface rewrite ([#2026](https://github.com/AtlasDevHQ/atlas/issues/2026)) features in the README.

| Question | Exercises |
|---|---|
| What's our GMV this quarter? | Simple metric (`total_gmv`) |
| What's our top-performing category by GMV this month? | Segmentation (`revenue_by_category` + date filter) |
| Top 10 products by units sold last 30 days, with their category | Multi-join (orders → order_items → products → categories) |
| Monthly GMV trend over the past 6 months | Time-series (`monthly_gmv_trend`) |
| Repeat purchase rate by acquisition source | Virtual dimension (`acquisition_source_normalized`) |
| Show me revenue last quarter | **Glossary disambiguation** — `revenue` is `status: ambiguous` (GMV vs. net revenue vs. seller revenue), so the agent should ask which definition you mean |
| Which customers haven't ordered in 60 days? | Filtered pattern (customers + orders join with date filter) |
| What are our most common return reasons? | Normalized enum aggregation (`returns_by_reason`) |

## Tech debt patterns

The demo deliberately includes patterns that mirror real codebases — they're what the profiler is built to detect.

**19 missing FK constraints** — plus ~1.5% of payments reference nonexistent orders (orphaned from deleted test orders).

**4 abandoned tables** — `old_orders_v1`, `temp_product_import_2023`, `legacy_analytics_events`, `payment_methods_backup`. The profiler flags these with `possibly_abandoned` and prepends a warning in `use_cases`.

**Inconsistent enums** — e.g. `customers.acquisition_source`: `Google`/`google`/`GOOGLE`; `loyalty_accounts.tier`: `Gold`/`gold`/`GOLD`. The profiler detects these and adds `LOWER()` guidance in the glossary.

**5 denormalized tables** — `orders_denormalized`, `daily_sales_summary`, `monthly_revenue_summary`, `product_performance_cache`, `customer_ltv_cache`. The profiler flags these with `possibly_denormalized`.

## Schema evolution artifacts

Five places where old and new columns coexist — this is exactly the kind of state that confuses a naive text-to-SQL agent and that the semantic layer's data-quality notes are designed to handle.

| Table | Old column | New column | Issue |
|-------|-----------|------------|-------|
| `products` | `price` (dollars) | `price_cents` (cents) | ~40% NULL `price_cents` |
| `customers` | `phone` | `mobile_phone` | ~15% NULL `mobile_phone` (post-2022 customers only) |
| `shipments` | `carrier` (text) | `carrier_id` (integer) | ~60% NULL `carrier_id` |
| `orders` | `shipping_cost` | — | dollars pre-2023-06, cents after |
| `product_reviews` | `rating` (int) | `rating_decimal` (numeric) | ~70% NULL `rating_decimal` |

## Schema overview

**Table groups:**
- Core Commerce (6 tables): customers, addresses, segments, loyalty
- Product Catalog (7 tables): products, variants, images, tags, inventory
- Marketplace (4 tables): sellers, applications, payouts, performance
- Orders & Transactions (7 tables): orders, items, events, payments, refunds, gift cards
- Shipping & Fulfillment (5 tables): shipments, carriers, returns
- Marketing & Promotions (5 tables): promotions, email campaigns, UTM tracking
- Reviews (3 tables): product reviews, responses, helpfulness
- Reporting (5 tables): denormalized/rollup/cache tables
- Site Analytics (3 tables): page views, cart events, search queries
- Internal / Ops (3 tables): admin users, audit log, settings
- Legacy (4 tables): abandoned tables
