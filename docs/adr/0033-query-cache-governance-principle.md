# Query Cache governance: the key captures row-determining inputs; hits re-apply presentation governance

Status: accepted (2026-07-10, admin-cache elevation grill)

The Query Cache accreted governance behavior without a principle: approval gate, PII masking, and the current row limit were re-evaluated on every hit, but RLS was baked into cached rows at write time (it is injected into the SQL, so the DB filters before the entry exists) with **no invalidation on RLS config change** — an admin tightening RLS kept serving pre-change rows for up to TTL — and plugin `beforeQuery`/`afterQuery` hooks were silently skipped on hits even though `beforeQuery` can veto or rewrite a query. We decided the principle is: **the cache key captures every input that determined the entry's rows; the hit path re-applies every layer that transforms rows on the way out; anything that can veto a query runs before the cache check.** Concretely: (1) the resolved RLS config hashes into the cache key alongside the user claims already there, so any RLS change instantly makes pre-change entries unreachable — correct by construction; (2) plugin `beforeQuery` dispatch moves above the cache check, so a plugin rejection blocks warm hits and a rewrite lands in the key; (3) `afterQuery` and connection `recordQuery`/SLA metrics stay live-path-only, documented as such — `afterQuery` observes *executions* and a hit doesn't execute; hit-traffic visibility belongs to the cache's own stats, not to pretending a hit is an execution.

The rejected alternative for (1) was flush-the-cache-on-RLS-settings-change (mirroring the existing config-reload flush). It is a smaller diff but event-based: it only holds if every present and future RLS mutation path remembers to fire the flush, and a single workspace's RLS change would nuke the whole region's cache since flush is fleet-wide. Key-hashing is invariant-based — there is no event to miss — at the cost of pre-change entries lingering as unreachable garbage until TTL/eviction (bounded by `maxSize`) and of resolving RLS config before key-build rather than at execution (a reordering, not new work). The same by-construction reasoning is why claims were already in the key; RLS config was the one row-determining input the key missed.
