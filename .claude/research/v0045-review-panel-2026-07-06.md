## Must fix (HIGH)

### H1. [silent-failure] Notion depth-cap returns a knowingly-incomplete enumeration ‚Üí reconciliation archives live pages, sync reports success
`packages/api/src/lib/knowledge/notion/client.ts:394-408` (walkBlocks depth cap), consumed by `fetchAll` ‚Üí archiving at `packages/api/src/lib/knowledge/connector-sync.ts:457-463`.
The module header's own invariant (lines 14-16) says an incomplete enumeration must THROW; every other incompleteness path honors it (`MAX_ENUMERATED_PAGES`, `advancePageCursor`, `requireResponseObject`). But hitting `MAX_BLOCK_WALK_DEPTH` only `log.warn`s and continues, so `fetchAll` returns a partial set, `ingestDocuments({ archiveAbsent: true })` archives the missed page's document, and status is `success`. On re-discovery the doc resurrects as `draft`, losing published state. Trigger: a page reachable only through >6 nested container blocks that Notion search (non-exhaustive by contract) stops returning.
**Fix:** throw at the enumeration depth cap (matching `MAX_ENUMERATED_PAGES`), **or** ‚Äî structurally better, also resolves M1 ‚Äî return a `coverageIncomplete` flag from the vendor client and have the engine skip `archiveAbsent` for that cycle (upserts proceed; deletions wait for a clean crawl). The content-render `blockWalkMarkdown` depth cap can stay lenient ‚Äî only enumeration feeds the subtractive diff.

### H2. [type-design] Web Zod drift guard is one-directional ‚Äî the next `KnowledgeCollectionSource` member type-checks green and hard-fails `/admin/knowledge` at runtime
`packages/web/src/ui/lib/admin-schemas.ts:667` (hand-maintained `z.enum(["upload","bundle-sync","notion","confluence"])`), guards at `:759-778`.
The `_Expect<z.infer<...> extends WireType>` guards only check schema-output ‚äÜ wire type; a narrower enum is always assignable to a wider union, so adding a fifth source (this milestone added two; `satisfies KnowledgeCollectionListResponse` at `packages/api/src/api/routes/admin-knowledge.ts:495` guarantees it gets emitted) breaks the whole admin page at Zod-parse time, including during deploy overlap. The comment calling this "the actual SSOT bridge" overstates it.
**Fix (type-only, no publish dance):** add the reverse guard: `export type _KnowledgeSourceExhaustive = _Expect<KnowledgeCollectionSource extends z.infer<typeof KnowledgeCollectionSchema>["source"] ? true : false>;` Consider the same for the route-local enum at `admin-knowledge.ts:144`.

### H3. [test-coverage] Notion pagination cursor-threading untested ‚Äî and the `for(;;)` enumeration loops can hang the entire sequential sync cycle
`packages/api/src/lib/knowledge/notion/client.ts:252-277, 289-305, 314-340, 371-414`.
The test fake (`notion.test.ts:88`) always returns `has_more: false`, so cursor echo (`next_cursor` ‚Üí `start_cursor`) is never exercised. Worse: a vendor/proxy glitch returning the same cursor+results forever hangs the client ‚Äî `refs` dedupes by id so `MAX_ENUMERATED_PAGES` never fires (Confluence counts duplicates and eventually throws; its pagination IS tested at `confluence/__tests__/client.test.ts:132`). `syncConnectorCollection` awaits `fetchAll()` with no timeout and `runKnowledgeSyncCycle` is sequential ‚Äî one wedged collection starves every workspace's sync.
**Fix:** add a request-count bound or repeated-cursor detection to the Notion loops, plus tests: (a) two-page fixture with `has_more: true, next_cursor: "CUR"` asserting the second request carries `start_cursor: "CUR"` and both pages collect; (b) hostile same-cursor-forever fixture asserting a throw.

### H4. [comment] `@atlas/fumadocs-okf` README "Dogfood consumer" section describes a consumer that no longer exists
`packages/fumadocs-okf/README.md:237-244`.
Claims `apps/docs/scripts/build-docs-kb-bundle.ts` builds the portal bundle "through this adapter" and is "the reference for adapting sites". After #4374 the script imports only `@atlas/okf-bundle`; the portal is not a consumer of this package at all.
**Fix:** rewrite: portal moved to `@atlas/okf-bundle` directly (#4373/#4374); point to `packages/fumadocs-okf/__tests__/roundtrip.test.ts` as the in-repo consumer.

## Should consider (MEDIUM)

### M1. [silent-failure] Confluence malformed-page skip still returns enumeration as complete ‚Äî same archive-live-pages exposure as H1
`packages/api/src/lib/knowledge/confluence/client.ts:272-293` (`normalizePage` skip + `skippedMalformed` warn).
A page missing `id`/`title`/`version.createdAt` is warn-skipped; in reconciliation mode its path is absent from the full set, so its previously-ingested document is archived while sync records success ‚Äî exactly the posture the Notion client refuses (`requireResponseObject`). Cross-PR drift on the same invariant.
**Fix:** in reconciliation (`withBody === true`), treat `skippedMalformed > 0` as a thrown error; incremental can keep warn-and-skip (no deletions there). Subsumed by the structural `coverageIncomplete` fix in H1 if taken.

### M2. [silent-failure√ótype-design√ótest-coverage] Notion install-row failure orphans the just-written credential ‚Äî Confluence rolls it back; the path is also untested
`packages/api/src/lib/integrations/install/notion-knowledge-form-handler.ts:162-168` vs `confluence-form-handler.ts:188-206` (tested at `confluence-form-handler.test.ts:200`).
Both write credential ‚Üí install row. On upsert failure Confluence best-effort-deletes the orphaned credential (with non-masking inner catch); Notion logs and rethrows, leaving an encrypted secret with no owning install ‚Äî `uninstallCollection` can never reach it. `notion-knowledge-form-handler.test.ts` never exercises the failing-upsert path. Flagged independently by three reviewers.
**Fix:** backport the `deleteSyncCredential` rollback block (or extract a shared `persistSyncedCollectionInstall` helper ‚Äî the two sequences are near-identical) + a test pinning the rollback.

### M3. [silent-failure] Notion install never verifies the token ‚Äî bad credential "installs successfully", fails only at first sync
`packages/api/src/lib/integrations/install/notion-knowledge-form-handler.ts:81-135`.
Confluence's contract: verify before persisting (`verifyConnection`, `confluence-form-handler.ts:218-243`); Notion validates shape only. A typo'd/revoked token yields a green install and a collection that silently never syncs until the scheduled cycle records an error.
**Fix:** one-request verification (`GET /users/me` via `NotionHttpClient`) before `saveSyncCredential`, mapping 401 ‚Üí field-level `FormInstallValidationError` on `integration_token`.

### M4. [type-design] Notion re-declares the OKF frontmatter render the wire module single-homed
`packages/api/src/lib/knowledge/notion/document.ts:83-100`.
`renderNotionOkfDocument` hand-rolls frontmatter with literal `"type: Document"` / `"atlas:"` instead of `DEFAULT_OKF_TYPE` / `ATLAS_EXTENSION_KEY` / `renderOkfDocument` (`@atlas/okf-bundle` `okf.ts:30-45`), which Confluence uses (`confluence/documents.ts:156`). Also emits no provenance `tags` (Confluence does). Encoding drift here churns drafts via the lenient parser's change-comparison.
**Fix:** extend `renderOkfDocument` with an optional extension-block param and delete the hand-rolled renderer; at minimum import the constants.

### M5. [type-design] Malformed Notion `last_edited_time` prematurely stops the incremental walk ‚Äî silent partial incremental
`packages/api/src/lib/knowledge/notion/client.ts:181-188` (compare + `"stop"`), `:268` (`asString` ‚Üí `""`).
One malformed page object mid-stream makes `"" >= since` false ‚Üí `"stop"` ‚Üí newer changed pages after it are skipped; healed only by weekly reconciliation. The type claims ISO-8601 but nothing enforces it at construction.
**Fix:** validate the timestamp when building `NotionPageRef` (warn-skip malformed pages, matching Confluence's `normalizePage`), or make the stop condition `Date.parse`-based with unparseable ‚Üí "continue".

### M6. [type-design] Vendor timestamps compared lexicographically as raw strings, not normalized instants
`confluence/client.ts:186,191`; `notion/client.ts:180-181,223-224` vs engine-normalized `toISOString()` (`connector-sync.ts:388`).
Correct only while both vendors emit exactly `YYYY-MM-DDTHH:mm:ss.sssZ`; an offset-format timestamp compares wrong with no error. `validVendorTimestamp` (`connector-sync.ts:255-269`) normalizes only the persisted mark, after comparisons ran.
**Fix:** parse to epoch ms (or normalized ISO) when constructing `EnumeratedPage`/`NotionPageRef`; consider an `IsoInstant` helper in `connectors.ts` to make the `ConnectorChanges.highWaterMark` contract self-enforcing.

## Structural takeaway

H1 + M1 + M11 + L1 are one design question worth settling once at the seam: **when a vendor client knows its enumeration is incomplete, reconciliation must not archive.** The engine could enforce this structurally ‚Äî a `coverageIncomplete` flag on `ConnectorChanges` gating `archiveAbsent` ‚Äî instead of relying on each vendor client to remember to throw.
