# Company-Brain / Agentic-Memory Landscape — Patterns, Failure Modes, and Atlas's Data-Grounded Wedge

**Purpose:** Feed the wayfinder ADR for Atlas's "company brain" bet. Distills the reusable architecture patterns and failure modes from the agentic-memory landscape and locates where Atlas's warehouse-grounded data-analyst posture structurally wins. Research for [AtlasDevHQ/atlas#4756](https://github.com/AtlasDevHQ/atlas/issues/4756).

> Sourcing discipline: every non-obvious claim carries an inline URL. Claims marked **[confirmed]** come from the owning party's own material (founder launch text, arXiv paper, product docs). Claims marked **[inferred]** or **[uncertain]** are secondary or my synthesis. An "Unable to verify" list sits at the end.

## Executive summary — the 5–8 things that most matter for Atlas's decision

1. **The whole field has converged on one substrate shape, and Atlas already owns most of it.** Episode-vs-fact split, subject-predicate-object triples, typed edges, bi-temporal invalidation, provenance-to-source, per-fact ACL, and hybrid retrieval (semantic + full-text + graph, fused) are now the *de facto* standard — independently arrived at by Hyper ([Launch HN](https://news.ycombinator.com/item?id=48387095)), Zep/Graphiti ([arXiv 2501.13956](https://arxiv.org/abs/2501.13956)), and mem0 ([arXiv 2504.19413](https://arxiv.org/pdf/2504.19413)). Atlas's KB pillar (review-gated draft→published OKF collections), durable memory, ingest spine, and MCP server are the same primitives under different names. The build is substrate-assembly, not invention.

2. **The moat these players are chasing is *trustworthy staleness handling*, and it's unsolved.** Every serious system models "when was this true" separately from "when did we learn it" (bi-temporal) and invalidates rather than deletes. The canonical failure — "context that's right on Monday is quietly wrong by Friday, and nobody notices until a decision gets made on it" — is a direct user quote from Hyper's launch thread ([HN](https://news.ycombinator.com/item?id=48387095)) and the explicit design target of LongMemEval's "Knowledge Updates" task ([arXiv 2410.10813](https://arxiv.org/html/2410.10813v1)).

3. **Benchmarks are saturating but adoption isn't — the gap is UX/trust, not retrieval accuracy.** Hyper's founders say it plainly: "many SOTA memory providers are converging on the top end of these benchmarks, and yet we don't see mass adoption… UX affordances are underrated and critical to get 'company brains' working in real, messy businesses" ([HN](https://news.ycombinator.com/item?id=48387095)). This is the single most important strategic read for Atlas: the win is not a higher LoCoMo score.

4. **The connector/ingest layer is the acknowledged bottleneck for the extraction-based players — and it's a moat you can't buy your way out of.** nao-os's founder: "If your tool doesn't have an MCP, the agent won't help much" ([thenewaiorder](https://thenewaiorder.substack.com/p/i-built-a-company-brain-to-run-my)). Atlas inverts this: the warehouse *is already the connector* for the highest-value question class (metrics, counts, trends), and the semantic layer is a hand-verified schema rather than an LLM guess.

5. **Warehouse-grounded beats chat-extracted for exactly the questions a "company brain" is bought to answer, but loses on decisions/rationale/ownership.** "How many enterprise accounts churned last quarter" is a deterministic SQL query against authoritative rows; "why did we deprioritize the SSO project" is a fact an LLM extracted from a Slack thread. Atlas wins the first class *by construction* (no extraction error, no staleness, provenance = the row itself); it must extract or ingest for the second. These are complementary, not competing — the strongest product does both and knows which is which.

6. **Silent auto-write memory (hooks) is a trust liability the whole field is quietly nervous about.** Hyper injects context "on every prompt" via lifecycle hooks and "never actually deletes memories" ([HN](https://news.ycombinator.com/item?id=48387095)) — powerful, but it means the brain writes facts the user never reviewed. Atlas's **review-gated draft→published** KB flow is the antidote and a genuine differentiator: it is the only design in this survey that puts a human approval gate between "agent extracted a fact" and "fact is authoritative."

7. **Per-fact access control is table stakes, and it's a data problem Atlas already solves at the row level.** Hyper: "Every fact carries… access-control tags… two people on the same team can ask the same question and get different answers" ([HN](https://news.ycombinator.com/item?id=48387095)). Glean mirrors source-system permissions at the item level ([docs.glean.com](https://docs.glean.com/security/knowledge-graph)). Atlas's warehouse path can enforce this via RLS/whitelist at query time — enforcement by construction rather than by tagging a derived fact.

8. **Open-core self-hostability is a real structural wedge against Glean/Hyper.** Glean stores the graph in a per-customer GCP project ([AWS marketplace blog](https://aws.amazon.com/blogs/awsmarketplace/transform-enterprise-search-knowledge-discovery-glean-amazon-bedrock/)); Hyper is a hosted SaaS. Neither can offer "run the whole brain inside your own VPC under AGPL." Atlas can. For regulated buyers this is not a feature, it's a procurement unblock.

## Section 1 — Recurring failure modes (with evidence)

### 1.1 Staleness — "right on Monday, wrong on Friday"
The defining failure. Verbatim from a Hyper launch commenter: "context that's right on Monday is quietly wrong by Friday, and nobody notices until a decision gets made on it" ([HN](https://news.ycombinator.com/item?id=48387095)). **Designed against by:** bi-temporal modeling everywhere — Zep tracks `t_valid`/`t_invalid` (world validity) separately from `t_created`/`t_expired` (system lifecycle) and invalidates rather than deletes ([arXiv 2501.13956](https://arxiv.org/abs/2501.13956)); Hyper puts a `valid_until` timestamp on every fact plus a "neighbourhood update on every new write" that re-checks staleness/contradiction/supersession ([HN](https://news.ycombinator.com/item?id=48387095)); LongMemEval makes "Knowledge Updates" one of its five scored abilities and shows commercial systems drop 37% (ChatGPT) to 64% (Coze) versus offline-reading baselines ([arXiv 2410.10813](https://arxiv.org/html/2410.10813v1)). **Atlas relevance:** warehouse facts are *never* stale in this sense — the query re-reads the live rows. Extracted facts (KB) need the bi-temporal treatment.

### 1.2 Intent capture — spike/brainstorm notes leaking into authoritative facts
Raised directly in Hyper's thread: "Architecture notes documented during a spike… should not be referenced in everyday work. Yet… that type of memory always works its way into unrelated sessions" ([HN](https://news.ycombinator.com/item?id=48387095)). No surveyed system claims to have solved it. This is a *provenance-plus-classification* problem: the fact is true-ish but was authored in an exploratory context. **Atlas relevance:** the draft→published gate is a partial native answer — a spike note stays `draft` and never reaches the agent's authoritative set unless a human promotes it.

### 1.3 Conflict resolution — recency vs authority (the intern-vs-CEO problem)
Hyper currently resolves conflicts by **recency only** — "we assume more recent information is generally more true than old information" — and explicitly flags role-based ranking as *future* work: "be more skeptical when an intern contradicts the CEO" ([HN](https://news.ycombinator.com/item?id=48387095)). Glean's graph carries "confidence scores" and authority signals as edge properties to arbitrate this ([glean.com/blog](https://www.glean.com/blog/knowledge-graph-agentic-engine)). **Nobody has a clean solution.** Recency is a heuristic that fails exactly when it matters (a junior's stale correction overwriting a senior's standing decision). **Atlas relevance:** for warehouse facts this dissolves — the row is the authority; there is no "who said it." For extracted facts, Atlas would inherit the same open problem.

### 1.4 The connector bottleneck
The extraction-based brain is only as good as its ingest surface. nao-os: connectors "represent the primary bottleneck… Tools like LinkedIn and Luma lack integration capability, creating blind spots" ([thenewaiorder](https://thenewaiorder.substack.com/p/i-built-a-company-brain-to-run-my)). Glean's entire pitch leans on "100+ connectors" as the hard part ([AWS blog](https://aws.amazon.com/blogs/awsmarketplace/transform-enterprise-search-knowledge-discovery-glean-amazon-bedrock/)). **Atlas relevance:** Atlas's highest-value input — the warehouse — is a *single* connector that already contains the structured truth of the business. The extraction players must integrate dozens of noisy sources to approximate what a warehouse already holds cleanly.

### 1.5 Silent-hook trust — auto-writing memory the user never reviewed
Hyper injects context "on every prompt" via lifecycle hooks in Claude Code/Cursor/Codex and "never actually deletes memories" ([HN](https://news.ycombinator.com/item?id=48387095)). The upside is zero-friction capture; the downside is an authoritative store growing from unreviewed LLM extractions. No surveyed extraction system has a human-in-the-loop review gate before a fact becomes authoritative. **Atlas relevance:** this is Atlas's clearest process differentiator — the KB's review-gated draft→published pipeline is *exactly* the missing gate.

### 1.6 Lossy graph transformation
A technical skeptic on Hyper's thread: "You lose… much meaningful context when you transform something into a knowledge graph" ([HN](https://news.ycombinator.com/item?id=48387095)). SPO triples discard nuance, hedging, and conditionality. This is *why* every serious system keeps the raw **episode** as source-of-truth alongside the extracted facts (see 2.1). **Atlas relevance:** warehouse queries never suffer this — no transformation from prose to triple occurs. Extraction paths must keep raw source retrievable.

### 1.7 Benchmark saturation without adoption
LoCoMo shows LLM+RAG still lags humans by 56% overall and **73% on temporal reasoning** ([snap-research.github.io/locomo](https://snap-research.github.io/locomo/), [arXiv 2402.17753](https://arxiv.org/abs/2402.17753)); yet Hyper reports the top of these benchmarks is crowded while "we don't see mass adoption" ([HN](https://news.ycombinator.com/item?id=48387095)). The lesson: retrieval accuracy is necessary but not the product. Trust, UX, and correctness-you-can-verify are the gating factors.

## Section 2 — Reusable substrate patterns worth stealing

| Pattern | What it is | Who does it (primary) | Map onto Atlas |
|---|---|---|---|
| **Episode ⁄ fact split** | Keep the raw source item verbatim (episode) as source-of-truth; store extracted meaning (facts) separately, so disputes can be traced back and the lossy transform is recoverable. | Hyper: "Episodes are the raw source items kept as the source of truth. Facts are the meaning pulled out of each episode" ([HN](https://news.ycombinator.com/item?id=48387095)); Zep episode subgraph ([arXiv 2501.13956](https://arxiv.org/abs/2501.13956)). | KB should store the ingested source blob alongside the OKF fact; `ingest-bundle.ts` is the natural seam. Warehouse path needs no episode — the row *is* the episode. |
| **Subject-predicate-object triples** | Facts as `(subject, predicate, object)` records, e.g. `person works_at company`, each with introduced/invalidated timestamps. | Hyper ([HN](https://news.ycombinator.com/item?id=48387095)); mem0's extraction phase converts messages to "entities and relation triplets" ([arXiv 2504.19413](https://arxiv.org/pdf/2504.19413)). | Optional structured layer over KB facts; complements (does not replace) the semantic layer's typed columns/measures. |
| **Typed edges** | Relationships carry semantics: `supersedes`, `derives-from`, `in-tension-with`. Enables reasoning about *why* facts relate, not just *that* they do. | Hyper: "X is in tension with Y, A is derived from B, J supersedes K" ([HN](https://news.ycombinator.com/item?id=48387095)). | New capability for Atlas KB — model supersession/tension between knowledge cards; drives conflict surfacing in the UI. |
| **Bi-temporal invalidation** | Two clocks per fact — *valid time* (true in the world) and *transaction/ingest time* (known to the system). Superseded facts are invalidated, never deleted; enables "what did we believe on date X." | Zep `t_valid`/`t_invalid` + `t_created`/`t_expired` ([arXiv 2501.13956](https://arxiv.org/abs/2501.13956)); Hyper `valid_until` ([HN](https://news.ycombinator.com/item?id=48387095)). | KB status columns could grow a temporal-validity dimension beyond draft/published/archived. Warehouse facts get this free from the data's own timestamps. |
| **Provenance to source** | Every fact links back to the exact source item + carries confidence. Audit and dispute resolution. | Hyper: "Every fact carries provenance back to its source" ([HN](https://news.ycombinator.com/item?id=48387095)); Glean edge properties "timestamps, access control, confidence scores, or provenance" ([glean.com/blog](https://www.glean.com/blog/knowledge-graph-agentic-engine)). | Warehouse answers already cite `columns`/`rows` + the query as provenance — strongest form. KB facts should carry a source pointer. |
| **Per-fact ACL** | Access-control tags on each fact; retrieval evaluated only against facts the asker may see, so identical questions yield different answers per person. | Hyper ([HN](https://news.ycombinator.com/item?id=48387095)); Glean item-level permission mirroring ([docs.glean.com](https://docs.glean.com/security/knowledge-graph)). | Atlas enforces at *query time* via whitelist/RLS for warehouse data — enforcement by construction. KB needs per-collection/per-card ACL to match. |
| **Hybrid retrieval + RRF** | Query-expand, then fuse dense (embedding) + sparse (Postgres FTS/BM25) results with reciprocal rank fusion; optional graph traversal + rerank. | Hyper: "query-expand, then fuse semantic search over embeddings with Postgres full-text search using reciprocal rank fusion" ([HN](https://news.ycombinator.com/item?id=48387095)); Zep cosine + BM25 + graph BFS + rerank ([arXiv 2501.13956](https://arxiv.org/abs/2501.13956)). | Atlas `searchKnowledge` already does FTS/graph over KB (ADR-0028); adding embedding + RRF fusion is a bounded, high-leverage upgrade. |
| **Hooks-vs-MCP read/write duality** | Two write paths: MCP (explicit tool calls, but "once the session dies, so does the insight") and lifecycle hooks (auto-inject context every prompt, zero friction, but unreviewed). | Hyper ([HN](https://news.ycombinator.com/item?id=48387095)). | Atlas already has the MCP path. A hooks-style auto-capture path is optional — but gate it behind draft-review, not silent-authoritative. |
| **Git-repo-as-memory** | CONTEXT.md (orientation, read first) + MEMORY.md (cross-domain learnings index) + per-domain folders + skills + agent definitions. No platform lock-in; portable across Claude/Codex. | nao-os ([thenewaiorder](https://thenewaiorder.substack.com/p/i-built-a-company-brain-to-run-my)); echoed by RepoBrain, brain.md, gbrain ([github.com/topics/company-brain](https://github.com/topics/company-brain)). | Validates Atlas's own repo-memory habit (this very file, MEMORY.md). Portable, greppable, diff-reviewable memory is a legitimate product primitive, not just a dev convenience. |
| **Agent-team composition** | A small org of role-scoped agents (Chief of Staff, CMO, Head of Data, PM, CSM, EA, Brand Designer, +1) each combining skills + connectors, all reading the shared brain. | nao-os 8-agent team ([thenewaiorder](https://thenewaiorder.substack.com/p/i-built-a-company-brain-to-run-my)). | The "AI-employee layer on top" in the bet. Atlas's tool registry + answer-styles + proactive chat are the ingredients; role-scoped agents over a shared KB/warehouse is the composition. |
| **Community/summary subgraph** | Cluster related entities into higher-level communities for coarse recall and summarization. | Zep community subgraph ([arXiv 2501.13956](https://arxiv.org/abs/2501.13956)). | Later-stage; maps to KB collection-level rollups. |
| **Extract→update two-phase pipeline** | Ingest is two stages: extract salient facts, then reconcile against existing store with conflict detection/resolution — minimizes tokens/latency. | mem0: 91% lower p95 latency, 90% fewer tokens vs full-context on LOCOMO ([arXiv 2504.19413](https://arxiv.org/pdf/2504.19413)). | Shape for any Atlas KB auto-ingest: don't dump raw; extract + reconcile at the `ingest-bundle` seam. |

## Section 3 — Where warehouse-grounded beats chat/email-extracted (and where it doesn't)

**The structural argument.** An extracted fact is an LLM's lossy summary of prose, carrying extraction error, staleness, and an authority-arbitration problem (§1.1–1.3, §1.6). A warehouse answer is a deterministic query against authoritative rows through a *hand-verified* semantic layer: no extraction step, no "when was this true" ambiguity (the data's own timestamps answer it), and provenance = the exact rows + SQL. Where the two overlap, warehouse-grounded is strictly more trustworthy because it removes the transformation that every extraction failure mode lives inside.

**Question classes warehouse-grounded wins (decisively):**
- **Quantitative / aggregate:** counts, sums, rates, cohorts, trends ("how many enterprise accounts churned last quarter", "MRR by region MoM"). LoCoMo shows extraction systems lag humans **73% on temporal reasoning** ([snap-research](https://snap-research.github.io/locomo/)) — precisely the class SQL does exactly.
- **Exhaustive/deterministic list queries:** "list all AEs in Asia." Glean itself concedes this is where probabilistic LLM generation fails and a structured lookup is required ([glean.com/blog](https://www.glean.com/blog/knowledge-graph-agentic-engine)) — and a warehouse *is* the structured lookup, no graph-building required.
- **Anything with a freshness SLA:** the row is re-read live; there is no Monday-to-Friday staleness window (§1.1).
- **Entity disambiguation on real keys:** joins on IDs beat vector-proximity guesses (the "which Reddit / which Claude Sonnet version" failure Glean documents).
- **Governed answers:** per-user correctness via RLS/whitelist at query time, enforced by construction rather than by tagging derived facts (§2 ACL row).

**Question classes extraction wins (be honest):**
- **Decisions, rationale, and their history:** "why did we deprioritize SSO," "what did we promise this customer on the call." These never touch the warehouse. This is Hyper/nao-os/Glean's home turf.
- **Ownership, tacit knowledge, org relationships:** "who owns the billing pipeline," "who knows about the EU migration." Glean's people/activity pillars ([docs.glean.com](https://docs.glean.com/security/knowledge-graph)) exist for exactly this; a warehouse rarely models it.
- **Unstructured commitments and status:** anything living in Slack/email/docs and never written to a table.

**Complementary, not competing.** The strongest company brain routes by question class: structured/quantitative/governed → warehouse via semantic layer (Atlas's core strength); decision/rationale/ownership/tacit → extracted fact graph (KB + ingest). Atlas's differentiated position is to make the **warehouse the authoritative spine** and treat extracted facts as a *clearly-labeled, review-gated, provenance-bearing* second tier — never letting an LLM-extracted Slack fact silently outrank a queryable number. That labeling is the trust story the whole field (§1.7) is failing to deliver.

## Section 4 — Per-product findings

### Hyper (YC P26) — the direct competitor
Founders Kanyes & Shalin Shah; hosted SaaS at [heyhyper.ai](https://heyhyper.ai); [Launch HN #48387095](https://news.ycombinator.com/item?id=48387095); [YC launch page](https://www.ycombinator.com/launches/QKg-hyper-the-self-driving-company-brain). Positions as "the self-driving company brain" that "learns your entire company and puts it back into every AI you use" — extract from docs/email/Slack/GitHub/calendar → synthesize → inject into AI tools "without ugly tool calls." **[confirmed]** architecture: episode/fact split; SPO triples with introduced/invalidated timestamps; typed edges (in-tension/derives/supersedes); provenance + per-fact ACL; query-expand + embeddings + Postgres FTS fused via RRF; lifecycle-hook injection preferred over MCP; recency-primary conflict resolution with role-based ranking as future work; never deletes, reranks by confidence; evaluated on LoCoMo + LongMemEval. Traction (self-reported): top-5 HN / top-12 PH, 50+ teams onboarded, $0→$1k MRR in 12 days, paid pilots scoped with Razorpay and Snorkel AI ([HN](https://news.ycombinator.com/item?id=48387095)). **Read for Atlas:** Hyper is the sharpest articulation of the extraction-first company brain and validates the substrate — but it has *no* warehouse-grounding, *no* review gate, and *no* self-host story. Those three are Atlas's wedge.

### nao-os — git-repo-as-brain
Founder "Claire," [thenewaiorder substack](https://thenewaiorder.substack.com/p/i-built-a-company-brain-to-run-my). A Git repo (no SaaS, no lock-in, runs in Claude/Codex): `CONTEXT.md` (read-first orientation), `MEMORY.md` (cross-domain learnings index), per-domain folders (Sales/Marketing/Content…), skills, and 8 role agents (Chief of Staff, CMO, Head of Data, Product Manager, Customer Success Manager, Executive Assistant, Brand Designer, +1). Connectors via CLI/MCP: Gmail, Calendar, Notion, Granola, Canva, nao, Twenty, Slack, Excalidraw, GitHub. **[confirmed]** headline lesson: connectors are "the primary bottleneck — if your tool doesn't have an MCP, the agent won't help much." **Read for Atlas:** validates repo-memory (Atlas already lives this) and the agent-team composition as the "AI-employee layer"; its bottleneck is precisely what warehouse-grounding sidesteps.

### Zep / Graphiti — the reference temporal-KG memory engine
[arXiv 2501.13956](https://arxiv.org/abs/2501.13956); Graphiti open-source (getzep reports 20k+ stars, [getzep.com](https://www.getzep.com/ai-agents/temporal-knowledge-graph/)). **[confirmed]** three subgraphs (episode / semantic-entity / community); bi-temporal (`t_valid`/`t_invalid` + `t_created`/`t_expired`); contradiction → edge invalidation not deletion; entity resolution/dedup; hybrid retrieval = cosine + BM25 + graph BFS + rerank. Benchmarks: DMR 94.8% vs MemGPT 93.4%; LongMemEval "up to 18.5%" accuracy improvement with ~90% latency reduction vs full-context. **Read for Atlas:** the canonical blueprint for the temporal-KG mechanics Atlas would add to KB. Steal the bi-temporal model and edge invalidation wholesale.

### mem0 — token-efficient two-phase memory
[arXiv 2504.19413](https://arxiv.org/pdf/2504.19413) (ECAI 2025), [mem0.ai/research](https://mem0.ai/research). Two-phase Extract→Update pipeline; graph variant `Mem0g` (entities + relation triplets, conflict detect/resolve on update). **[confirmed]** LOCOMO: +26% accuracy vs OpenAI native memory, 91% lower p95 latency, 90% fewer tokens vs full-context. **Read for Atlas:** the efficiency argument for extract-then-reconcile ingest rather than raw-dump; directly informs the `ingest-bundle` seam design.

### Glean — enterprise search + knowledge graph (why "brain" ≠ "search")
[docs.glean.com/security/knowledge-graph](https://docs.glean.com/security/knowledge-graph); [glean.com/blog/knowledge-graph-agentic-engine](https://www.glean.com/blog/knowledge-graph-agentic-engine). Three pillars: content, people, activity. Item-level permission mirroring of source ACLs; per-customer GCP project ([AWS blog](https://aws.amazon.com/blogs/awsmarketplace/transform-enterprise-search-knowledge-discovery-glean-amazon-bedrock/)). **[confirmed]** their own argument that a graph beats plain search/RAG: captures *relationships* not just content; enables multi-hop reasoning; precise entity disambiguation; deterministic exhaustive queries; and governance via edge properties (timestamps, ACL, confidence, provenance). **Read for Atlas:** Glean's four "why not just search" advantages are the same four a warehouse+semantic-layer delivers natively for structured questions — *and Glean concedes exhaustive/relational queries need a structured store, which is what a warehouse is.* Glean's weakness vs Atlas: it approximates structure by building a graph over crawled documents; Atlas queries structure that already exists.

### Shared-memory / "Memory Store" pattern
The brief's "Memory Store" maps to a *pattern*, not a single confirmed product (see Unable-to-verify). The pattern: all agents read/write one shared memory layer so knowledge accumulates rather than fragmenting per-agent ([zylos multi-agent memory](https://zylos.ai/research/2026-03-09-multi-agent-memory-architectures-shared-isolated-hierarchical/)). Realized by mem0, Zep, Cognee, Google **Vertex AI Memory Bank**, AWS **Bedrock AgentCore Memory**, OpenAI Memory, and dedicated **MCP memory servers** ([mem0 state-of-memory 2026](https://mem0.ai/blog/state-of-ai-agent-memory-2026)). **Read for Atlas:** Atlas's KB + durable memory + MCP already *is* a shared cross-agent memory layer; the differentiator is that Atlas's shared layer is anchored to authoritative warehouse data, not a pile of extracted chat facts.

### Benchmarks — what they actually measure
- **LoCoMo** ([arXiv 2402.17753](https://arxiv.org/abs/2402.17753), [snap-research.github.io/locomo](https://snap-research.github.io/locomo/)): ~300 turns / 9k tokens / up to 35 sessions per conversation. Tasks: QA (single-hop, multi-hop, temporal, commonsense, adversarial), event-graph summarization, multimodal dialog. Finding: LLM+RAG lags humans by **56% overall, 73% on temporal reasoning**.
- **LongMemEval** ([arXiv 2410.10813](https://arxiv.org/html/2410.10813v1)): 500 questions across five abilities — information extraction, multi-session reasoning, temporal reasoning, knowledge updates, abstention. LongMemEval-S ≈115k tokens, -M ≈1.5M tokens. Finding: **30–60% accuracy drop** on full history vs oracle retrieval; ChatGPT −37%, Coze −64%. **Read for Atlas:** both benchmarks punish exactly the temporal/update reasoning that warehouse queries do deterministically — a natural place to show a data-grounded system's structural advantage if Atlas ever wants a public number.
