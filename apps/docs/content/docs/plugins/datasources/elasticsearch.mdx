A single unified plugin (`@useatlas/elasticsearch`) that connects an
**Elasticsearch** or **OpenSearch** cluster as a read-only Atlas datasource over
a thin `fetch`-based HTTP client — no official SDK dependency. The agent queries
a single index-as-table through the standard `executeSQL` tool; under the hood
the connection POSTs to the cluster SQL API and normalizes the response.

## Installation

```bash
bun add @useatlas/elasticsearch
```

No Elasticsearch/OpenSearch SDK is required — the connector talks to the
cluster's read endpoints over `fetch`.

## Configuration

```typescript
// atlas.config.ts
import { defineConfig } from "@atlas/api/lib/config";
import { elasticsearchPlugin } from "@useatlas/elasticsearch";

export default defineConfig({
  plugins: [
    elasticsearchPlugin({
      url: "elasticsearch://my-deployment.es.us-east-1.aws.found.io:9243",
      apiKey: process.env.ES_API_KEY!,
    }),
  ],
});
```

When admins install the datasource from **Admin → Connections**
([admin console guide](/guides/admin-console#connections)), the same fields are
entered through a form; the `secret: true` fields are encrypted at rest and
masked in the UI.

### Connection target

A connection needs an endpoint — supply **either** a `url` **or** an Elastic
`cloudId` (not both).

| Field     | Secret | Description |
| --------- | ------ | ----------- |
| `url`     | no     | `elasticsearch://host[:port][/prefix]` or `opensearch://host[:port][/prefix]`. HTTPS by default; append `?ssl=false` (alias `?tls=false`) for a plaintext local cluster. Credentials are **never** placed in the URL — the parser rejects userinfo and auth query params. |
| `cloudId` | no     | An Elastic **Cloud ID** (`<name>:<base64>`), decoded to the cluster's HTTPS endpoint. An `atlas.config.ts` convenience — not offered in the admin form. |

### Engine selection

The engine (`elasticsearch` or `opensearch`) selects the SQL endpoint
(`/_sql` vs `/_plugins/_sql`). It is resolved with this **precedence**:

1. An explicit `engine` config field (`"elasticsearch"` | `"opensearch"`) — wins over everything.
2. Otherwise the **URL scheme**: `elasticsearch://` → `elasticsearch`, `opensearch://` → `opensearch`.
3. Otherwise (a Cloud ID, which carries no scheme) → `elasticsearch`.

```typescript
// opensearch:// scheme → OpenSearch engine
elasticsearchPlugin({ url: "opensearch://localhost:9200?ssl=false", apiKey: "…" })

// elasticsearch:// scheme, but force the OpenSearch SQL surface
elasticsearchPlugin({ url: "elasticsearch://host:9200", engine: "opensearch", apiKey: "…" })
```

### Authentication

Supply exactly one mode's fields. When more than one set is present the resolver
picks by a documented **precedence: AWS SigV4 → HTTP Basic → API key.**

| Mode | Fields | Secret fields |
| ---- | ------ | ------------- |
| **API key** | `apiKey` (Base64-encoded; sent as `Authorization: ApiKey`) | `apiKey` |
| **HTTP Basic** | `username` + `password` (both required) | `password` |
| **AWS SigV4** | `awsRegion` (selects the mode) + credentials (below) + optional `awsService` (default `es`) | `awsSecretAccessKey`, `awsSessionToken` |

Every secret field is encrypted at rest (`secret: true` drives Atlas's
`encryptSecretFields`); the AWS region, access-key id, and service code carry no
credential and are stored in clear.

```typescript
// HTTP Basic
elasticsearchPlugin({
  url: "elasticsearch://logs.internal:9200?ssl=false",
  username: process.env.ES_USER!,
  password: process.env.ES_PASSWORD!,
});

// AWS SigV4 — Amazon OpenSearch Service / IAM-protected domains
elasticsearchPlugin({
  url: "opensearch://search-mydomain.us-east-1.es.amazonaws.com",
  awsRegion: "us-east-1",
  // Explicit keys are optional — omit them to use the ambient AWS environment
  // chain (AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY / AWS_SESSION_TOKEN).
  awsAccessKeyId: process.env.AWS_ACCESS_KEY_ID,
  awsSecretAccessKey: process.env.AWS_SECRET_ACCESS_KEY,
});

// Elastic Cloud ID + API key
elasticsearchPlugin({
  cloudId: process.env.ES_CLOUD_ID!,
  apiKey: process.env.ES_API_KEY!,
});
```

**AWS SigV4 credentials** resolve from the explicit `awsAccessKeyId` /
`awsSecretAccessKey` / `awsSessionToken` fields first, then fall back to the
ambient AWS environment chain (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`,
`AWS_SESSION_TOKEN`). Every request is signed fresh (Signature Version 4) with
the configured region and service.

### Profiling with `atlas init` / `atlas diff`

The CLI semantic-layer profiler supports the same auth modes and engines as the
plugin, configured through `ATLAS_ES_*` environment variables — credentials are
**never** carried in the URL
([#3309](https://github.com/AtlasDevHQ/atlas/issues/3309)):

| Variable | Maps to | Description |
| -------- | ------- | ----------- |
| `ATLAS_DATASOURCE_URL` | `url` | `elasticsearch://host[:port]` or `opensearch://host[:port]` |
| `ATLAS_ES_CLOUD_ID` | `cloudId` | Elastic Cloud ID — the endpoint when no URL is set |
| `ATLAS_ES_API_KEY` | `apiKey` | Base64 API key |
| `ATLAS_ES_USERNAME` / `ATLAS_ES_PASSWORD` | `username` / `password` | HTTP Basic (both required) |
| `ATLAS_ES_AWS_REGION` | `awsRegion` | Selects AWS SigV4; keys come from the ambient `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` (+ `AWS_SESSION_TOKEN`) chain |
| `ATLAS_ES_AWS_SERVICE` | `awsService` | SigV4 service code (default `es`) |
| `ATLAS_ES_ENGINE` | `engine` | Explicit engine override |

The auth-mode precedence (SigV4 → Basic → API key) is the plugin's — the CLI
builds the same config shape and hands it to the same resolver.

```bash
# Basic-secured self-hosted cluster
export ATLAS_DATASOURCE_URL="elasticsearch://logs.internal:9200?ssl=false"
export ATLAS_ES_USERNAME="atlas_reader"
export ATLAS_ES_PASSWORD="…"
bun run atlas -- init

# Amazon OpenSearch Service via SigV4
export ATLAS_DATASOURCE_URL="opensearch://search-mydomain.us-east-1.es.amazonaws.com"
export ATLAS_ES_AWS_REGION="us-east-1"   # keys from the AWS env chain
bun run atlas -- init

# Elastic Cloud — no URL needed, the Cloud ID names the endpoint
export ATLAS_ES_CLOUD_ID="my-deployment:…"
export ATLAS_ES_API_KEY="…"
bun run atlas -- init
```

## SQL query surface

Ask a tabular or aggregate question over a single index in chat and the agent
answers through the standard `executeSQL` tool — the same 4-layer validation
pipeline as any SQL datasource. The connection POSTs your statement to the
engine's SQL API, follows the response `cursor` across pages up to the row cap,
and normalizes the result to Atlas's `{ columns, rows }`.

| Engine | Endpoint | Response shape |
| ------ | -------- | -------------- |
| Elasticsearch | `POST /_sql?format=json` | `{ columns, rows, cursor }` |
| OpenSearch | `POST /_plugins/_sql?format=jdbc` | `{ schema, datarows, cursor }` |

ES/OpenSearch SQL **is** standard SQL, so the plugin declares
`parserDialect: "PostgresQL"` with no custom validator. PostgreSQL's
double-quoted identifier quoting matches index-name quoting
(`FROM "logs-2024.01.01"`). Each index is a table — there are **no JOINs across
indices**.

Beyond the base DML/DDL guard, the connection blocks the catalog/schema
disclosure verbs `SHOW …` and `DESCRIBE …` (they enumerate every index/field and
so bypass the index whitelist). These are anchored to the statement start, so a
field literally named `show` or `description` mid-query is fine, and
`ORDER BY … DESC` is unaffected.

## Logical sources: indices, patterns, aliases & data streams

`atlas init` profiles a cluster into entity YAMLs. Beyond one entity per concrete
index, it represents three kinds of **logical source** as a single queryable
entity, so a time-partitioned family is one table rather than one-per-day
([#3269](https://github.com/AtlasDevHQ/atlas/issues/3269)):

| Source | `table:` value | How it's discovered |
| ------ | -------------- | ------------------- |
| **Index pattern** | `logs-*` | Two or more indices sharing a base with a trailing date (`logs-2024.01.01`, `logs-2024-01-01`, `events-2024`) or rollover sequence (`metrics-000001`) are collapsed to `<base>-*`. |
| **Alias** | the alias name | `GET /_alias` — the alias's backing indices are unioned into one entity. |
| **Data stream** | the stream name | `GET /_data_stream` — the stream's hidden `.ds-…` backing indices are unioned. |

Each logical entity's dimensions are the **union** of its backing indices'
mappings (members of a family share a template). A concrete index that backs an
alias/data-stream, or that collapses into a pattern, is represented **only** by
the logical entity — never also as a standalone index. A lone dated index with no
sibling stays a concrete index (no surprise wildcard). The entity's `table:` is
the logical name, which is exactly what both query surfaces and the index
whitelist key on; the YAML **filename** is slugged (`logs-*` → `logs-star.yml`)
so the `*` never reaches the filesystem.

Querying is unchanged from a concrete index — name the logical source directly:

```sql
SELECT level, COUNT(*) FROM "logs-*" GROUP BY level
```

```text
queryElasticsearch(index: "logs-*", body: { ... })
```

<Callout type="warn">
A pattern entity authorizes the **literal pattern string**, not a fixed set of
indices. The cluster expands `logs-*` at query time, so the entity grants access
to whatever matches it *then* — **including indices created after `atlas init`
ran**. Declaring a `logs-*` entity is your explicit authorization of that family.
If a family must not expose future or by-pattern indices, declare
concrete-index entities instead of a pattern.

Pattern auto-detection is conservative to avoid collapsing unrelated families:
it only groups indices whose suffix is a real date (`(19|20)YY[.-MM[-DD]]`) or a
zero-padded rollover counter (`000001`). Id-suffixed indices like `app-12345`
stay separate.
</Callout>

## Query DSL surface

For questions SQL can't express — full-text relevance ranking, deeply-nested
aggregations, geo/span queries — a **static** datasource also registers a
dedicated `queryElasticsearch` agent tool that issues a **read-only** Query DSL
request (`POST /<index>/_search` | `_count`) and flattens the response to Atlas's
`{ columns, rows }`. The tool inputs are a single explicit `index` (a concrete
index, alias, data stream, or declared index-pattern entity), an optional
`endpoint` (`_search` default, or `_count`), and an optional Query DSL `body`.

The surface is gated by a **default-deny** read-only validator: only read
endpoints are allowed; mutating/administrative endpoints, smuggled bulk-write
actions, mutating (`ctx`-referencing) scripts, ad-hoc wildcard / `_all` /
system-index targets are rejected. On top of those always-on structural rails,
the tool enforces **per-index membership** against the semantic layer: the
queried index must be a profiled entity (the same index/table whitelist
`executeSQL` validates against), so the DSL surface honors the identical access
boundary as the SQL surface ([#3307](https://github.com/AtlasDevHQ/atlas/issues/3307)).
A wildcard target is allowed **only** when it is itself a declared index-pattern
entity (e.g. `logs-*`) — a deliberately-authorized fan-out, not an ad-hoc one
([#3269](https://github.com/AtlasDevHQ/atlas/issues/3269)).
When a connection has no semantic layer the membership check is skipped and only
the structural rails apply. Every `_search` is resource-bounded — `size` clamped
to `ATLAS_ROW_LIMIT`, a search `timeout`, and a non-aggregation `terminate_after`
(`ATLAS_ES_TERMINATE_AFTER`).

The `queryElasticsearch` tool is registered **only** for a static,
`atlas.config.ts`-defined datasource; in SaaS per-workspace mode the connection
is queried over the SQL surface (per-workspace DSL routing is a later slice).

## Security

- **Read-only.** Only `SELECT` reaches the cluster, through Atlas's standard
  4-layer validation (regex DML/DDL guard → AST single-`SELECT` parse → index
  whitelist → auto-`LIMIT` + statement timeout) plus the `SHOW`/`DESCRIBE` guard.
- **Secrets encrypted at rest.** `apiKey`, `password`, `awsSecretAccessKey`, and
  `awsSessionToken` are `secret: true` — encrypted via `encryptSecretFields` and
  masked in the admin UI.
- **Secret-scrubbed errors.** Connection, health, query, and mapping errors are
  scrubbed before they reach the agent, the user, or logs: any literal secret is
  redacted and messages tripping auth-context markers are collapsed to a generic
  message (the detail stays in server logs). Query errors still surface the
  actionable engine reason (e.g. `Unknown column [foo]`) so the agent can
  self-correct.

## Troubleshooting

### `401 Unauthorized` / `403 Forbidden`

Check the auth mode matches the cluster: API-key clusters need a Base64 `apiKey`;
secured self-hosted clusters use `username`/`password`; Amazon OpenSearch Service
domains use `awsRegion` + AWS credentials. Health and connection errors are
scrubbed, so inspect server logs for the full reason.

### AWS SigV4 — "no credentials"

Setting `awsRegion` selects SigV4. Provide `awsAccessKeyId` + `awsSecretAccessKey`
explicitly, or ensure the ambient `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY`
environment variables are set in the API process.

### TLS certificate errors

Managed clusters are HTTPS by default. For a plaintext local cluster append
`?ssl=false` to the URL.
