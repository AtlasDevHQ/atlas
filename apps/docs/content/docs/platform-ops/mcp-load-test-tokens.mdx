---
title: MCP Load-Test Tokens
description: Mint short-lived bearer JWTs for k6-driven MCP load testing in CI
---

`POST /api/v1/admin/load-test/mcp-token` mints a short-lived,
MCP-scoped JWT bearer that k6 (or any HTTP client) can carry against
`/mcp/{workspace_id}/sse` to exercise the hosted MCP surface under
load. The token is signed by the same regional Better Auth JWKS the
OAuth 2.1 path uses — the MCP edge verifier (`verifyAccessToken`)
resolves the matching public key through `/api/auth/jwks` and verifies
identically to a real OAuth-issued token. There is no "load-test only"
verifier path.

This endpoint exists because the OAuth 2.1 ceremony (DCR + auth-code +
PKCE + token exchange) is the right shape for an actual MCP client
(Claude Desktop, Cursor) but the wrong shape for CI: every re-run
would either pollute production with a permanent client/user or wedge
in the 5-step flow at 2 AM. A bespoke mint endpoint trades token
expiry for ceremony — every run gets a fresh, bounded-lifetime bearer.

## Auth + access

- Caller must hold the **`platform_admin`** role and have **MFA enrolled**
  (the `platformAdminAuth` + `mfaRequired` middleware chain).
- Per-endpoint **rate limit**: 10 mints/minute per admin, in-memory
  per region. Bursts above this return 429 with a `Retry-After`
  header. The shared admin RPM bucket would let a runaway loop mint
  thousands of tokens before tripping; the per-endpoint cap bounds the
  blast radius without forcing a DB-failure-sensitive code path.

## Request

```http
POST /api/v1/admin/load-test/mcp-token
Authorization: <platform-admin session>
Content-Type: application/json

{
  "workspaceId": "<org-id>",
  "ttlSeconds": 300,
  "region": "us"
}
```

| Field | Required | Default | Notes |
|---|---|---|---|
| `workspaceId` | yes | — | Atlas organization id the token will be scoped to. The MCP path must match (`/mcp/{workspaceId}/sse`); a path/claim mismatch returns 403 at the verifier. |
| `ttlSeconds` | no | 300 | Server-side ceiling is **3600s (1 hour)**. Requests above the cap return **400, not silently clamped** — caps cannot be widened by accident. Floor is 60s. |
| `region` | no | this region | Region key matching `residency.regions[*]` in `atlas.config.ts`. Defaults to the API instance's own region. The token's audience is bound to this region — see [Region constraint](#region-constraint) below. |

## Response

```json
{
  "bearer": "eyJhbGciOiJFZERTQSI...",
  "workspaceId": "ws-abc",
  "audience": "https://api.useatlas.dev/mcp",
  "issuer": "https://api.useatlas.dev/api/auth",
  "expiresAt": "2026-05-07T01:00:00.000Z",
  "sub": "loadtest:ws-abc:9f3c1a8b04e25d61",
  "scope": "mcp:read"
}
```

The `bearer` is what you carry in `Authorization: Bearer <token>`. The
`sub` is a synthetic identifier — every load-test bearer's subject
starts with `loadtest:`, which makes load traffic trivially filterable
in audit log queries (`actor_id LIKE 'loadtest:%'`). The `scope` is
always `mcp:read` — write capability is gated at the tool layer and
this endpoint never grants it.

## Recommended client usage

```bash
# In CI (GitHub Actions example):
TOKEN=$(curl -s -X POST \
  -H "Authorization: Bearer $ATLAS_PLATFORM_ADMIN_KEY" \
  -H "Content-Type: application/json" \
  -d '{"workspaceId":"'"$LOADTEST_WORKSPACE_ID"'","ttlSeconds":600}' \
  "$ATLAS_REGION_API/api/v1/admin/load-test/mcp-token" \
  | jq -r .bearer)

# Drive k6 with the bearer as an env var so it never lands in stdout:
MCP_BEARER="$TOKEN" k6 run loadtest/mcp-profile.js
```

Two anti-patterns to avoid:

1. **Do not embed bearers in logs, command echoes, or test fixtures.**
   The audit row carries `jti` only; the bearer must not appear
   anywhere your retention policy could leak. Treat it like any other
   short-lived credential — pass via env var or stdin, never
   command-line argument.
2. **Do not reuse a bearer across CI runs.** Mint per-run with an
   appropriate TTL (5–10 minutes is enough for most load profiles).
   Long-lived tokens defeat the bounded-blast-radius property the
   short TTL exists to provide.

## Region constraint

The token's `aud` claim is bound to the resolved region's `/mcp`
URL. The MCP edge verifies `aud` exactly — a token minted against
`api.useatlas.dev` and used against `api-eu.useatlas.dev` will fail
with **401 invalid_bearer** (audience mismatch). A second guard fires
even before the verifier: `checkResidency` on the MCP route returns
**421 misdirected_request** if the workspace's region doesn't match
the API instance, with a `correctApiUrl` hint pointing at the right
regional endpoint.

In practice: call `POST /api/v1/admin/load-test/mcp-token` against
the same regional API host you intend to load-test against. For a
multi-region load profile, mint one bearer per region.

## Errors

| Status | Code | Cause / Remediation |
|---|---|---|
| 400 | `invalid_body` | Missing `workspaceId`, `ttlSeconds` outside `[60, 3600]`, or malformed JSON. The error message is the underlying Zod issue — actionable as-is. |
| 400 | `unknown_region` | The requested `region` is not in `residency.regions`. Add it to `atlas.config.ts` or omit the parameter to use this API's region. |
| 400 | `region_missing_api_url` | The region exists but has no `apiUrl` — set `residency.regions["<region>"].apiUrl` in `atlas.config.ts`. |
| 401 | — | Caller is not authenticated. |
| 403 | — | Caller is not `platform_admin`, or has not enrolled MFA. |
| 404 | `not_configured` | The internal database (`DATABASE_URL`) is not set; there is no JWKS table to mint against. |
| 429 | `rate_limited` | Per-admin mint cap exceeded. Includes `Retry-After` header. |
| 500 | `mint_failed` | Unexpected mint failure (decryption error, malformed JWK row, jose import error). The underlying cause is in the API logs at `requestId`; an `audit_log` row with `status: failure` is also written. |
| 503 | `jwks_not_initialized` | The Better Auth `jwks` table is empty. Hit any `/api/auth/*` endpoint on this region once to seed the keypair, then retry. |

## Audit trail

Every successful mint writes an `admin_action_log` row with
`action_type: "load_test.mint_mcp_token"` via the **awaited**
`logAdminActionAwait` helper (not fire-and-forget). The audit row is
the security control on this credential issuance surface — a DB hiccup
mid-mint surfaces to the caller as a 500 rather than leaving a silent
token in the wild.

Metadata fields:

| Field | Notes |
|---|---|
| `workspaceId` | Org id the token was scoped to. |
| `region` | Resolved region (or null on self-hosted). |
| `ttlSeconds` | TTL granted. |
| `sub` | Synthetic actor id stamped on the token. Pivot column for forensic queries. |
| `jti` | JWT id. The MCP `mcp_session.start` audit row carries the same `jti`, so a reviewer can pivot from a suspicious session row back to the mint that authorized it. |
| `expiresAt` | ISO 8601 expiry. |

The `bearer` is **never** in metadata. If you find a bearer in any
audit row, treat that as a P1 incident — the contract is that
issuance and use are correlated only via the `jti`.

## Self-hosted

The endpoint works on self-hosted deployments too — operators load-test
their single instance the same way. The `platform_admin` role gate is
the access control; nothing here is SaaS-specific.

When the `region` parameter is omitted on a self-hosted instance with no
`residency.regions` configured, the audience and issuer fall back to
the request origin (matching the same shape `hosted.ts:resourceAudience`
reads at the verifier). Set `ATLAS_PUBLIC_API_URL` if your verifier
audience is not the request `Host`.
