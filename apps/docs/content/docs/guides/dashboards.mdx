Dashboards let you pin query results from chat and organize them into persistent, shareable layouts. Each card stores its SQL, chart configuration, and cached results — and can be refreshed on demand or on a schedule.

<Callout title="Prerequisites">
- An [internal database](/deployment/authentication#managed-auth) configured (`DATABASE_URL`)
- At least one query result in chat to add
</Callout>

---

## Creating a Dashboard

### From Chat

1. Run a query in chat
2. On the SQL result card, click the **Dashboard** button (grid icon)
3. Choose **New Dashboard** and enter a title, or select an existing dashboard
4. Pick a chart type (bar, line, pie, area, scatter, or table)
5. Click **Add to Dashboard**

The card is saved with its SQL, chart configuration, and current result data.

### From the Dashboard List

Navigate to `/dashboards` to see all dashboards. Click **New Dashboard** to create an empty one, then add cards from chat.

## Managing Cards

Each dashboard card shows:

- **Title** — click to edit inline
- **Chart visualization** — rendered from cached data (if chart type is not "table")
- **Data table** — the underlying rows and columns
- **Last refreshed** — timestamp of the most recent data

### Environment (1.4.4+)

If your workspace has more than one **connection group** (an "environment") configured under **Admin → Connections → Environments**, the **Add to Dashboard** dialog includes an **Environment** picker. Pick the group your card should run against — Atlas resolves it to the group's primary member at view time, falling back to the first member by creation order when no primary is pinned. Moving the primary to a different member (e.g. from `us-int` to `eu`) retargets every card scoped to that environment without editing the cards themselves.

Single-environment workspaces don't see the picker — cards run against the workspace default, same as pre-1.4.4. A card pointing at an environment with zero members will refuse to refresh until an admin adds one, surfacing as a 500 with the offending group name; the dialog warns up-front so you don't save a card that can't run.

### Card Actions

| Action | How |
|--------|-----|
| **Reorder** | Drag the grip handle (⠿) to rearrange cards |
| **Refresh** | Click the refresh icon on a single card |
| **Edit title** | Click the pencil icon |
| **Remove** | Click the trash icon (removes the card, not the underlying query) |

## Refreshing Data

Cards store cached results. To update them:

- **Manual refresh** — Click the refresh icon on a single card, or **Refresh All** in the header
- **Auto-refresh** — Set a schedule from the timer dropdown:
  - Every 15 minutes, hourly, every 6 hours, daily, or weekly (Mon 9am)

Auto-refresh uses the scheduler tick engine. The next refresh time is computed from the cron expression and displayed in the dashboard header.

<Callout type="info">
Auto-refresh requires `ATLAS_SCHEDULER_ENABLED=true`. See [Scheduled Tasks](/guides/scheduled-tasks) for setup.
</Callout>

## AI Suggestions

Click **Suggest Cards** to have the AI analyze your existing cards and propose 2–3 complementary metrics. Suggestions are grounded in your [semantic layer](/getting-started/semantic-layer) — the AI only references tables and columns it knows about.

Each suggestion shows:

- **Title** and **reason** — why this card would complement your dashboard
- **Chart type** and **SQL preview**
- **Add** — accepts the suggestion and creates a real card
- **Dismiss** — removes the suggestion

All suggested SQL is validated before being shown. Invalid queries are filtered out automatically.

## Sharing

Click the **Share** button to generate a public link. Options:

| Setting | Description |
|---------|-------------|
| **Public** | Anyone with the link can view the dashboard (no login required) |
| **Org-scoped** | Only authenticated members of your organization can view |
| **Expiry** | 1 hour, 24 hours, 7 days, 30 days, or never |

Shared dashboards display at `/shared/dashboard/[token]` with read-only access to cached data. Viewers cannot modify cards or trigger refreshes.

To revoke access, open the share dialog and click **Revoke**.

## Chat as editor (1.4.6+)

Dashboards have a built-in chat-bound editor. From a dashboard page, open the chat drawer to talk to Atlas about the cards you can see — the conversation is scoped to that dashboard and the agent gets a `boundDashboardId` context so its `executeSQL` / `createCard` / `updateCardSql` / `removeCard` tools target this dashboard automatically. Use it to ask "why is this card flat?", iterate on a SQL definition, or stage a new card without leaving the dashboard.

### Per-user drafts

Edits from the bound chat (and from direct admin actions when the flag is on) go into your **personal draft** of the dashboard — never directly to the published copy your teammates see. The draft is exact-three-way merged against the persisted baseline on publish, so a teammate publishing a non-overlapping change while you draft does **not** stomp your work; an overlapping change surfaces a "baseline changed" banner with a one-click rebase.

The draft badge in the dashboard header shows your pending change count. Click **Publish** to promote the draft via `POST /api/v1/dashboards/:id/draft/publish`; the diff modal previews every staged change before you confirm. Discard with **Discard draft** to revert to the published view.

### Stage tracker

Destructive operations (card removal, SQL rewrites) return a `stage_required` envelope rather than applying immediately. The UI overlays a ghost on affected cards until you confirm — `Keep` discards the staged change, `Apply` commits it to your draft. The stage tracker is idempotent: applying or discarding a pending change twice is a no-op.

### History tab

Each dashboard's chat drawer has a **History** tab listing prior chat sessions tied to this dashboard (workspace-wide). Open any session for a read-only transcript so you can see what your team asked Atlas before — useful when picking up a dashboard mid-investigation.

### Feature flag

Set `ATLAS_DASHBOARD_DRAFTS_ENABLED=false` to disable per-user drafts entirely — admin mutations go straight to the published copy as in 1.4.5 and earlier. The flag defaults to `true`. See the [environment-variables reference](/reference/environment-variables#settings).

## API Reference

All dashboard operations are available via the REST API:

| Endpoint | Description |
|----------|-------------|
| `GET /api/v1/dashboards` | List dashboards |
| `POST /api/v1/dashboards` | Create a dashboard |
| `GET /api/v1/dashboards/:id` | Get dashboard with cards |
| `PATCH /api/v1/dashboards/:id` | Update title, description, or schedule |
| `DELETE /api/v1/dashboards/:id` | Delete a dashboard |
| `POST /api/v1/dashboards/:id/cards` | Add a card |
| `PATCH /api/v1/dashboards/:id/cards/:cardId` | Update card title or position |
| `DELETE /api/v1/dashboards/:id/cards/:cardId` | Remove a card |
| `POST /api/v1/dashboards/:id/cards/:cardId/refresh` | Refresh a single card |
| `POST /api/v1/dashboards/:id/refresh` | Refresh all cards |
| `POST /api/v1/dashboards/:id/suggest` | Get AI card suggestions |
| `POST /api/v1/dashboards/:id/share` | Generate share link |
| `DELETE /api/v1/dashboards/:id/share` | Revoke share link |
