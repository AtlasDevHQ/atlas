Atlas Cloud's Annex I sub-processor list lives at
[useatlas.dev/dpa#annex-i-subprocessors](https://www.useatlas.dev/dpa#annex-i-subprocessors)
and is the source of truth for any compliance-team monitoring. Two
machine-readable channels mirror it for procurement automation:

- **RSS / Atom feed** at `/sub-processors/feed.xml` — pull-based, no
  registration required, rendered by every standard feed reader.
- **Webhook subscriptions** via `POST /api/v1/sub-processor-subscriptions`
  — push-based, HMAC-signed deliveries to a customer-supplied URL on
  every add, change, or removal.

Email notifications to account admins continue to be the default channel
and are not affected by either of these surfaces.

## RSS / Atom

Add this URL to your feed reader or status-monitoring tool:

```
https://www.useatlas.dev/sub-processors/feed.xml
```

The feed is also auto-discoverable from `/dpa` via a
`<link rel="alternate" type="application/atom+xml">` element, so most
browser-based RSS extensions detect it automatically when the page loads.

One `<entry>` is emitted per current sub-processor. The `<updated>`
element carries the entry's `changed_at` ISO date — use that as the
de-dupe key. Entries are stable: the same sub-processor produces the
same `<id>` (a `tag:useatlas.dev,YYYY-MM-DD:sub-processors/<name>` URI)
across publications.

## Webhook subscriptions

### Register

```bash
curl -X POST https://app.useatlas.dev/api/v1/sub-processor-subscriptions \
  -H "Authorization: Bearer ${ATLAS_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://hooks.example.com/sub-processors",
    "token": "rotate-me-some-32-char-random-string"
  }'
```

The `token` is the shared secret Atlas uses to HMAC-sign every delivery
to your URL. It is stored encrypted at rest and **cannot be retrieved
after registration** — keep your own copy.

Successful registration returns:

```json
{ "id": "subp_a1b2c3d4e5f6a7b8c9d0e1f2" }
```

<Callout type="info">
The endpoint is auth-gated. Use a managed-mode session cookie or a
simple-key bearer issued from your workspace settings. Anonymous
registration is rejected with `401`.
</Callout>

### Receive

When the sub-processor list changes, Atlas POSTs one event per change to
your URL. The body is JSON:

```json
{
  "event": "added",
  "entry": {
    "name": "ExampleVendor",
    "purpose": "Cloud infrastructure (compute, storage)",
    "region": "United States",
    "since": "2026-04",
    "changed_at": "2026-04-15"
  }
}
```

`event` is one of:

| Event | Meaning |
|-------|---------|
| `added` | A new sub-processor joined the list. |
| `changed` | An existing entry's purpose, region, or `changed_at` updated. The body also includes a `previous` field with the prior values. |
| `removed` | A sub-processor was removed. The `entry` field carries the last-known shape. |

Two headers ride alongside the body:

| Header | Value |
|--------|-------|
| `X-Webhook-Timestamp` | Unix seconds at signing time. |
| `X-Webhook-Signature` | `sha256=<hex>` over `${X-Webhook-Timestamp}:${body}`. |

### Verify the HMAC signature

Reject any request whose signature does not match. Reference Node.js
implementation:

```ts
import crypto from "node:crypto";

function verify(rawBody: string, headers: Headers, token: string): boolean {
  const ts = headers.get("x-webhook-timestamp");
  const sig = headers.get("x-webhook-signature");
  if (!ts || !sig) return false;

  // Reject requests outside a 5-minute window to block replays.
  const now = Math.floor(Date.now() / 1000);
  if (Math.abs(now - Number(ts)) > 300) return false;

  const expected =
    "sha256=" +
    crypto.createHmac("sha256", token).update(`${ts}:${rawBody}`).digest("hex");

  // timingSafeEqual is required — `===` leaks the answer via timing.
  const a = Buffer.from(sig);
  const b = Buffer.from(expected);
  return a.length === b.length && crypto.timingSafeEqual(a, b);
}
```

The wire format is intentionally symmetric with the inbound
[`@useatlas/webhook`](https://www.npmjs.com/package/@useatlas/webhook) plugin's
verification helper — if you already have receivers wired up for that
plugin, you can reuse the same primitive.

### Delivery semantics

- Atlas retries `5xx` responses and transport errors with capped
  exponential backoff (3 attempts, 1s/2s base). `4xx` responses are
  treated as permanent failures and not retried — fix your endpoint and
  contact `legal@useatlas.dev` to replay.
- Delivery is **at-least-once**. De-dupe in your handler using
  `(entry.name, event, entry.changed_at)` as the key.
- Each tick fans out the diff against the most recent published
  snapshot; if a tick crashes mid-fan-out, the next tick re-derives and
  re-delivers the same diff.

## Slack adapter recipe

Atlas does not store customer Slack webhook URLs (credential-management
is yours, not ours). Run a tiny adapter in front of your Slack incoming
webhook:

```ts
// adapter.ts — deploy as a Vercel/Cloudflare/Lambda function.
// Set env vars: ATLAS_WEBHOOK_TOKEN, SLACK_WEBHOOK_URL.

export async function POST(request: Request): Promise<Response> {
  const raw = await request.text();
  if (!verify(raw, request.headers, process.env.ATLAS_WEBHOOK_TOKEN!)) {
    return new Response("invalid signature", { status: 401 });
  }
  const payload = JSON.parse(raw) as {
    event: "added" | "changed" | "removed";
    entry: { name: string; purpose: string; region: string };
  };

  const verb = payload.event === "added"
    ? "added"
    : payload.event === "removed"
      ? "removed"
      : "updated";

  await fetch(process.env.SLACK_WEBHOOK_URL!, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      text: `Atlas ${verb} sub-processor *${payload.entry.name}* (${payload.entry.region}) — ${payload.entry.purpose}`,
    }),
  });

  return new Response(null, { status: 200 });
}
```

Reuse the `verify` helper from the previous section.

## Unsubscribing

Email `legal@useatlas.dev` from the address that registered the
subscription. The Atlas team revokes the row manually — there is no
self-serve `DELETE` endpoint today, since procurement workflows
typically need an audit trail on the off-boarding event.
