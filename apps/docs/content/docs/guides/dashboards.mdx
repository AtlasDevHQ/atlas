Dashboards let you pin query results from chat and organize them into persistent, shareable layouts. **Chart cards** store a SQL query, chart configuration, and cached results — refreshable on demand or on a schedule; **text cards** store markdown for [section headers](#section-headers-text-cards) and explainers (no SQL, no data).

<Callout title="Prerequisites">
- An internal database configured (`DATABASE_URL`)
- At least one query result in chat to add
</Callout>

---

## Creating a Dashboard

### From Chat

1. Run a query in chat
2. On the SQL result card, click the **Dashboard** button (grid icon)
3. Choose **New Dashboard** and enter a title, or select an existing dashboard
4. Pick a chart type (bar, line, pie, area, scatter, or table)
5. Click **Add to Dashboard**

The card is saved with its SQL, chart configuration, and current result data.

### From the Dashboards Surface

Navigate to `/dashboards` and create a board from any surface origin: **New dashboard** in the dashboard switcher, **New** in the **View all** modal, or **Create your first dashboard** on the empty state. Enter a title and the new board opens on its canvas with the [bound editor](#chat-as-editor) already open — describe what you want to see and the agent builds the charts right there. Prefer an empty board? Close the drawer: the dashboard stays a normal empty canvas you can fill by pinning results from chat.

## Managing Cards

A dashboard holds two kinds of card: **chart cards** (a SQL query + a visualization) and **text cards** (markdown [section headers](#section-headers-text-cards)). Each chart card shows:

- **Title** — click to edit inline
- **Chart visualization** — rendered from cached data (if chart type is not "table")
- **Data table** — the underlying rows and columns
- **Last refreshed** — timestamp of the most recent data

### Environment

If your workspace has more than one **connection group** (an "environment") configured under **Admin → Connections → Environments**, the **Add to Dashboard** dialog includes an **Environment** picker. Pick the group your card should run against — Atlas resolves it to the group's primary member at view time, falling back to the first member by creation order when no primary is pinned. Moving the primary to a different member (e.g. from `us-int` to `eu`) retargets every card scoped to that environment without editing the cards themselves.

Single-environment workspaces don't see the picker — cards run against the workspace default, the same as a single-connection setup. A card pointing at an environment with zero members will refuse to refresh until an admin adds one, surfacing as a 500 with the offending group name; the dialog warns up-front so you don't save a card that can't run.

### Card Actions

| Action | How |
|--------|-----|
| **Reorder** | Drag the grip handle (⠿) to rearrange cards |
| **Refresh** | Click the refresh icon on a single card |
| **Edit title** | Click the pencil icon |
| **Download CSV** | Open the card menu (**⋯**) → **Download CSV** — exports the card's current result at your parameter values (see [Exporting](#exporting)) |
| **Remove** | Click the trash icon (removes the card, not the underlying query) |

### Section headers (text cards)

Not every card is a chart. A **text card** is a markdown section block — a header or a one-line explainer — that you drop into the grid to organize a wall of charts: "Top of funnel", "Conversion", "Cohorts". It has **no SQL, no chart, and fetches no data**; it just renders sanitized markdown and lays out in the same 24-column grid as every other tile (full-width by default, so it reads as a banner above the charts grouped beneath it).

Ask the agent for them in chat:

> "Build a growth dashboard and group the cards under section headers."

The `createDashboard` tool emits a text card as `{ kind: "text", content: "## Top of funnel", layout }` — for any dashboard with 4+ cards it adds a full-width header above each cluster of related charts. Because a text card carries no query, it never touches the SQL validation/execution pipeline, and **Refresh** skips it.

<Callout type="info" title="Markdown is sanitized">
Text-card content is rendered with the same markdown renderer as chat — headings, lists, tables, bold/italic, links, and code. Raw HTML is **not** evaluated (no `rehype-raw`), so a text card can't inject scripts or markup into the page.
</Callout>

### KPI / scorecard cards

A **KPI card** is a chart card with `chartConfig.type: "kpi"` — a compact scorecard rather than a plot. It's the first thing a reader looks at, so lead a dashboard with two or three of them ("Revenue", "Active users", "Conversion") and put the trend charts below. Each KPI card renders:

- **A big formatted number** — the headline metric.
- **A delta chip** (▲/▼ + %) — the change versus a comparison value, when one is configured. Up is green, down is red.
- **A sparkline** — drawn automatically when the query returns a trend (two or more rows) instead of a single value.

The card's `sql` returns the headline metric — a single row for a plain scorecard, or a time-ordered multi-row trend (the last row is the headline, and the value column draws the sparkline). Either way `categoryColumn` names the label/time column and `valueColumns[0]` the metric column. A `chartConfig.kpi` block controls the rest:

| Field | Description |
|-------|-------------|
| `valueFormat` | How the big number renders: `currency` (`$1.2M`), `number` (`1.2K`), `percent` (a ready figure like `12.3` → `12.3%`, not `0.12`), or `duration` (seconds → `1h 2m`). Defaults to `number`. |
| `comparisonSql` | Optional second single-number query — usually the same metric over the **prior period**. The UI computes the percentage change versus the headline value to draw the delta chip. Omit it for a plain big number. |
| `comparisonLabel` | Caption under the chip, e.g. "vs. last month". |

Ask the agent for them in chat:

> "Lead the dashboard with KPI cards for revenue, new customers, and churn, each showing the change vs. last month."

A KPI card binds [parameters](#parameters) exactly like any other card, and the delta tracks whatever window the parameter bar selects. Every `:placeholder` — including those in `comparisonSql` — must be declared in the dashboard's `parameters`. The example below references the prior window via a `:prev_from` parameter (declared alongside `:date_from` / `:date_to`); declaring it is what lets the comparison query bind at render time.

```json
{
  "title": "Revenue",
  "sql": "SELECT 'Revenue' AS label, SUM(amount) AS total FROM orders WHERE created_at >= :date_from AND created_at < :date_to",
  "chartConfig": {
    "type": "kpi",
    "categoryColumn": "label",
    "valueColumns": ["total"],
    "kpi": {
      "valueFormat": "currency",
      "comparisonSql": "SELECT SUM(amount) AS total FROM orders WHERE created_at >= :prev_from AND created_at < :date_from",
      "comparisonLabel": "vs. prior period"
    }
  }
}
```

<Callout type="warn" title="The comparison query runs through the same guard">
`comparisonSql` is executed as a second query through the **identical** SQL pipeline as the primary — SELECT-only, AST-validated, table-whitelisted, auto-LIMITed, statement-timed-out — and it binds the dashboard's parameters the same way (server-side, never interpolated). A comparison query that fails validation simply drops the delta chip; it never breaks the headline number.
</Callout>

### Goal lines & thresholds

Metrics with a **target** — a revenue goal, an error-rate budget, an SLA — read better with the target drawn in. Add a `chartConfig.thresholds` array (up to five per card); each entry is `{ value, color?, label? }` where `value` is required and `color` (any CSS colour — hex, `rgb()`, or a named colour) and `label` are optional:

- On a **bar / line / area** card, each threshold draws a horizontal dashed reference line at `value`, with the optional `label` captioned at the line. A target above the current data still shows — the axis stretches to fit it.
- On a **KPI card**, the **first** threshold is the goal: it colours the headline number green/red for above/below target and adds a small "Target: … · above/below" callout. Pair it with `kpi.inverse: true` for a lower-is-better metric (error rate, latency, cost), so being **below** target reads as the good (green) outcome.

```json
{
  "title": "Revenue by month",
  "sql": "SELECT month, SUM(amount) AS revenue FROM orders GROUP BY month",
  "chartConfig": {
    "type": "bar",
    "categoryColumn": "month",
    "valueColumns": ["revenue"],
    "thresholds": [{ "value": 1000000, "label": "Target", "color": "#f59e0b" }]
  }
}
```

Just ask in chat — the agent sets a target line whenever you name a goal:

> "Add our $1M monthly revenue target as a goal line, and flag error rate red when it goes above 2%."

A card with no `thresholds` renders exactly as before, so adding one is always opt-in.

### Event annotations

Where goal lines are **horizontal** targets, event annotations are **vertical** markers — they call out *when* something happened on a time-series chart: a product launch, a campaign start, a pricing change. Add an `annotations` array on the card (up to 20 per card); each entry is `{ x, label, color? }` where `x` and `label` are required and `color` (any CSS colour — hex, `rgb()`, or a named colour) is optional:

- On a **line / area** card, each annotation draws a vertical dashed reference line at `x`, captioned with `label`. `x` must match a value on the card's time/category axis (e.g. the ISO date `"2026-01-15"` or a month label `"Jan"`) — a marker for a date outside the chart's window simply doesn't draw.
- **Bar, pie, scatter, KPI, and table** cards ignore annotations — a categorical or numeric axis has no time context to mark. The field is carried through unchanged, so switching a card's chart type never loses it.

Unlike `thresholds` (which live inside `chartConfig`), `annotations` is a **card-level** field — it survives a chart-type change and belongs to the card's time context rather than a specific visualization:

```json
{
  "title": "Weekly signups",
  "sql": "SELECT week, COUNT(*) AS signups FROM users GROUP BY week ORDER BY week",
  "chartConfig": {
    "type": "line",
    "categoryColumn": "week",
    "valueColumns": ["signups"]
  },
  "annotations": [
    { "x": "2026-01-15", "label": "Product launch", "color": "#10b981" },
    { "x": "2026-03-01", "label": "Spring campaign" }
  ]
}
```

Just ask in chat — the agent adds markers whenever you name a dated event:

> "Mark the product launch on Jan 15 and the spring campaign on Mar 1 on the signups chart."

A card with no `annotations` renders exactly as before, so adding one is always opt-in.

## Parameters

A static dashboard ages: "signups over the last 4 weeks" means "the 4 weeks as of when the SQL was written." **Parameters** fix this — they give a dashboard a top-level filter bar (a date range, a region, a segment) that every card binds to, so the same dashboard stays useful for months.

### How it works

A dashboard declares a list of parameters. Each card's SQL references them with `:<key>` placeholders:

```sql
SELECT date_trunc('week', created_at) AS week, COUNT(*) AS signups
FROM signups
WHERE created_at >= :date_from AND created_at < :date_to
GROUP BY week
ORDER BY week
```

A parameter definition is `{ key, type, default, label }`:

| Field | Description |
|-------|-------------|
| `key` | Placeholder name — referenced in SQL as `:<key>`. Lower-snake identifier (`date_from`, `region`). |
| `type` | `date`, `text`, or `number`. Drives the control in the bar and the value coercion. |
| `default` | Value used when the viewer hasn't picked one. For `date`, an ISO date (`2026-01-01`) or a relative expression (`now`, `now - 30 days`, `now - 1 month`). For `text`/`number`, a literal. `null` for none. |
| `label` | Shown above the control in the parameter bar. |

The parameter bar renders above the grid. Changing any control re-renders **every card** with the new values; the URL carries the selection (`?dparams=...`) so a shared or bookmarked link reproduces the exact view. Clearing the bar (**Reset**) returns to the cached snapshot, which is rendered with the parameters' defaults.

<Callout type="warn" title="Values are bound, never interpolated">
Parameter values are substituted **server-side at execution time through parameterized queries** — the `:<key>` placeholder is rewritten to the driver's positional bind (`$1` on PostgreSQL, `?` on MySQL) and the value travels through the driver's bind protocol, never concatenated into the SQL text. The query still runs through the full Atlas pipeline (SELECT-only, AST-validated, table-whitelisted), so the injection surface stays closed. Parameter binding is supported on PostgreSQL and MySQL datasources.
</Callout>

### From chat

Ask the agent to build a parameterized dashboard and it declares the parameters for you:

> "Build a signups dashboard with a date range filter, defaulting to the last 30 days."

The `createDashboard` tool accepts a `parameters` array and writes cards that reference `:date_from` / `:date_to` instead of hardcoding dates. Every `:placeholder` a card uses must be declared — the tool rejects a card that references an undeclared parameter.

### Rendering with parameters (API)

`POST /api/v1/dashboards/:id/cards/:cardId/render` executes a single card with supplied values and returns the rows **without** persisting them — it's the ephemeral, per-viewer render the bar uses:

```json
{ "parameters": { "date_from": "2026-01-01", "date_to": "2026-02-01" } }
```

Omitted parameters fall back to their defaults. Invalid values (a malformed date, a non-numeric number) are rejected with `400 invalid_parameters` before any SQL runs. The persisted card cache (shown on load and used by **Refresh**) is always rendered with the parameters' defaults.

## Drilldown & cross-filtering

Parameters don't have to be driven from the bar — you can drive them by **clicking the data**. Give a card a drilldown target and clicking a bar, slice, or table row sets that parameter to the clicked category, refiltering every other card that binds it. This turns a dashboard into an interactive cross-filter: click "EMEA" in a region chart and the revenue trend, the funnel, and the table all narrow to EMEA.

### Setting a drilldown target

A card opts in by naming a target parameter on its chart config:

```ts
chartConfig: {
  type: "bar",
  categoryColumn: "region",
  valueColumns: ["revenue"],
  drilldown: { targetParam: "region" }, // clicking a bar sets :region
}
```

The target must be a declared dashboard parameter, and the card's `categoryColumn` is the value that gets bound (a click on any other detected axis is ignored). A card with no `drilldown` is inert on click — existing view and interaction are unchanged. Cards built from chat inherit this automatically when the agent declares a matching parameter.

### Cross-filter chips

Active cross-filters show as a **chips bar** above the grid — one chip per filter (`Region: EMEA`), each with a remove (×), plus **Clear all**. Like the parameter bar, the selection lives in the URL (`?dparams=...`), so a shared or bookmarked link reproduces the filtered view and it survives a reload.

- **Re-click to deselect.** The clicked bar / slice / row renders **selected** (the rest dim); clicking the same element again clears that filter.
- **Compatible vs. incompatible cards.** A filter only changes cards whose SQL binds the relevant `:param`. Cards that don't are left untouched and visibly marked **"Not filtered"** (dimmed), so an unchanged tile reads as intentional rather than broken.
- **Filters compose.** Multiple active filters combine with **AND** — region *and* date range *and* segment — and a single click triggers one batched refetch of every card, never a request per card.

<Callout type="warn" title="Bound, never interpolated">
A drilldown sets a dashboard parameter and the value flows through the **exact same parameterized render path** as the bar (`$1`/`?` positional binds, full SELECT-only + AST + whitelist pipeline). Clicking a category never string-interpolates it into SQL — the injection surface stays closed.
</Callout>

## Refreshing Data

Cards store cached results. To update them:

- **Manual refresh** — Click the refresh icon on a single card, or **Refresh All** in the header
- **Auto-refresh** — Set a schedule from the timer dropdown:
  - Every 15 minutes, hourly, every 6 hours, daily, or weekly (Mon 9am)

Auto-refresh uses the scheduler tick engine. The next refresh time is computed from the cron expression and displayed in the dashboard header.

<Callout type="info">
Auto-refresh requires `ATLAS_SCHEDULER_ENABLED=true`. See [Scheduled Tasks](/guides/scheduled-tasks) for setup.
</Callout>

### Tile status

Each tile carries its **own** status, so you can trust a board tile by tile rather than reading a single page-level banner:

| Status | Meaning |
|--------|---------|
| **Loading** | The card is fetching. |
| **Fresh** | Cached data reflects the current definition and active parameters. |
| **Stale** | Cached data is older than the current definition/params (e.g. right after a publish, before the async refresh lands). Shown as a **color-shifting age caption** (muted → amber → red) plus a subtle body dim and a one-click **retry** — a board with one stale tile reads as one amber caption, never an alarming banner. |
| **Errored** | The query failed. Visually distinct from *empty*. |
| **Empty** | The query succeeded but returned zero rows — distinct from *errored*, so "no matching rows" never reads as "broken." |
| **Not filtered** | A cross-filter doesn't apply to this card (its SQL doesn't bind the parameter); the tile is marked rather than silently left unchanged. |

A tile that fails a parameter or cross-filter update stays labeled with its data's age (e.g. *"as of 2:04pm"*) and offers retry — it never silently substitutes the old unfiltered result for the failed new one.

## Exporting

### A single card to CSV

Every chart, table, and KPI card has a **Download CSV** action in its card menu (the **⋯** button in the tile header). It exports the card's **current result at your parameter values** — the same rows you see, with the dashboard's [parameters](#parameters) bound exactly as on screen.

- The export runs through the **same query pipeline** as the on-screen render — SQL validation, parameter **binding** (values are never spliced into the query text), and the row cap. It never opens a second query path.
- It respects the auto-LIMIT (`ATLAS_ROW_LIMIT`, default 1000). If the result was capped, Atlas warns you the export is **truncated** — narrow the query or tighten the parameters to export the full set.
- Cells are escaped per RFC 4180, and any cell beginning with `=`, `+`, `-`, or `@` is neutralized so a spreadsheet won't execute it as a formula ([CSV injection](https://owasp.org/www-community/attacks/CSV_Injection)). Genuine numbers are left intact.
- The filename derives from the card title plus a UTC timestamp, e.g. `monthly-revenue-20260605-123045.csv`.

Text cards carry no data, so they have **no** Download CSV action.

Under the hood this is `POST /api/v1/dashboards/:id/cards/:cardId/render?format=csv` with the same `{ "parameters": … }` body as the [render endpoint](#rendering-with-parameters-api). It streams `text/csv` as an attachment and sets `X-Atlas-Truncated: 1` (plus `X-Atlas-Row-Count`) when the row cap was hit.

### The whole dashboard to PDF or PNG

Use **Export** in the dashboard header to render the entire board — at your current parameter values — to a PDF or PNG.

## AI Suggestions

Click **Suggest Cards** to have the AI analyze your existing cards and propose 2–3 complementary metrics. Suggestions are grounded in your [semantic layer](/getting-started/semantic-layer) — the AI only references tables and columns it knows about.

Each suggestion shows:

- **Title** and **reason** — why this card would complement your dashboard
- **Chart type** and **SQL preview**
- **Add** — accepts the suggestion and creates a real card
- **Dismiss** — removes the suggestion

All suggested SQL is validated before being shown. Invalid queries are filtered out automatically.

## Sharing

Click the **Share** button to generate a public link. Options:

| Setting | Description |
|---------|-------------|
| **Public** | Anyone with the link can view the dashboard (no login required) |
| **Org-scoped** | Only authenticated members of your organization can view |
| **Expiry** | 1 hour, 24 hours, 7 days, 30 days, or never |

Shared dashboards display at `/shared/dashboard/[token]` with read-only access to cached data. Viewers cannot modify cards or trigger refreshes.

<Callout type="info" title="A shared link is a data-only snapshot">
The public/shared endpoint serves a **minimal, data-only snapshot** — per-card title, chart config, annotations, cached rows, and layout. It **never** carries the card `sql`, connection/owner/org identifiers, refresh schedule, or parameter definitions; those are stripped at the DTO type layer, not redacted after the fact, so a field can't leak by omission. Instead of interactive controls, the snapshot carries a **frozen, display-only parameter summary** (each parameter's label + resolved value, e.g. *"Date: Jun 1–Jul 1 · Region: All"*) so a viewer knows the frame the numbers were computed in.
</Callout>

**Org-scoped** shares open end to end for a logged-in teammate — the shared page forwards the viewer's own session, so "requires login" sharing works rather than 403-ing everyone. An invalid share request fails closed with `400` and never silently downgrades the share to public, and editing a share's expiry or visibility keeps the same link (token rotation is explicit and warns that prior links die). Share views are rate-limited on the real viewer identity, not the web server's IP.

To revoke access, open the share dialog and click **Revoke**.

### Embedding a shared dashboard

The share dialog's **Embed** tab provides an iframe snippet for dropping a shared dashboard into another page — docs, a wiki, an internal portal:

```html
<iframe
  src="https://your-atlas-app.example.com/shared/dashboard/{token}/embed"
  width="100%"
  height="600"
></iframe>
```

The `/embed` variant renders the **same** shared view (same token, same data-only snapshot, same expiry and revocation) inside a frame — it is not a second surface, so nothing leaks that the standard shared link wouldn't. Add `?theme=dark` for dark host pages. The embed route sets `noindex`, and the standalone `/shared/dashboard/[token]` page stays non-framable; only the `/embed` path is allowed as a frame ancestor. The dialog's copy is share-mode aware — an org-scoped share is described as members-only, not "anyone with the link."

One scope note for **org-scoped** shares: inside a *cross-site* iframe the browser withholds the viewer's session cookie (`SameSite`), so the embed shows a members-only sign-in notice instead of the dashboard. Embed org-scoped shares on pages served from the same site as Atlas, or use a public share for third-party embedding.

## Chat as editor

Dashboards have a built-in chat-bound editor. From a dashboard page, open the chat drawer to talk to Atlas about the cards you can see — the conversation is scoped to that dashboard and the agent gets a `boundDashboardId` context so its `executeSQL` / `createCard` / `updateCardSql` / `removeCard` tools target this dashboard automatically. Use it to ask "why is this card flat?", iterate on a SQL definition, or stage a new card without leaving the dashboard.

### Viewing vs. editing

A dashboard opens **read-only** — you can't nudge a live board by accident. Entering edit is an explicit act: click **Edit** (or open the bound chat editor, which counts as entering edit) to fork/load your draft, activate the drag/resize/rename handles, and reveal a persistent **"Editing your draft · Publish · Discard"** bar. While you're editing, the **canvas renders your draft** — every card shows the version you're working on (including in-progress SQL edits), never a stale published copy. On mobile, layout editing is desktop-only: the Edit affordance is hidden with a one-line explanation rather than shown-and-inert, so the phone is a clean viewing surface. A fullscreen tile is a real focus-trapped dialog, and drilldown is reachable by keyboard.

### Per-user drafts

**Every** edit — direct manipulation (drag, resize, rename, delete, schedule) and bound-chat/agent edits alike — lands in your **personal draft** of the dashboard, never directly on the published copy your teammates see. (Drafts are the model, not a toggle: the old `ATLAS_DASHBOARD_DRAFTS_ENABLED` flag was retired in favor of unconditional drafts.) The draft is exact-three-way merged against the persisted baseline on publish, so a teammate publishing a non-overlapping change while you draft does **not** stomp your work; an overlapping change surfaces a "baseline changed" banner with a one-click rebase. Draft cards are executable — parameter render, single-card refresh, and CSV export run the **draft's** SQL, so you can validate a change before publishing it.

Each draft card carries its own **draft cache** — private cached rows plus the instant they were captured. Forking a draft copies the published cards' cached data, so your draft opens showing the same numbers the board showed; from then on, refreshing a card while editing runs the draft's SQL and persists the result to *your* draft cache only (it survives a page reload, and never touches the published data your teammates see). A card that has never been published is fully operable inside the draft — refresh, parameter render, retry, and CSV export all work before its first publish — and its tile status and age caption reflect the draft cache's capture instant. Publishing or discarding the draft clears its cache.

<Callout type="info" title="Private until first publish">
A brand-new dashboard is **visible only to its creator until its first publish**. It joins your organization's dashboard list the moment you publish it the first time — so a half-built or abandoned never-published dashboard never clutters a teammate's list.
</Callout>

Publish returns as soon as the new definitions are promoted and then **asynchronously** refreshes every card whose SQL changed; those cards render **stale** (labeled with their data's age) until the refresh lands — you never see a new title over old numbers presented as current.

The draft badge in the dashboard header shows your pending change count. Click **Publish** to promote the draft via `POST /api/v1/dashboards/:id/draft/publish`; the diff modal previews every staged change before you confirm. Discard with **Discard draft** to revert to the published view.

### Destructive edits + inline undo

Destructive operations (card removal, SQL rewrites) apply to your draft immediately, just like every other edit — the canvas reflects the change at once (publish later promotes it to the live board). The chat surfaces a one-click **Undo** alongside the edit: undoing a removal restores the card verbatim (same data), and undoing a SQL rewrite puts the previous query back. There is no accept/discard step and no ghost overlay — the draft is the single edit mechanism, and undo is just the inverse draft edit.

### History tab

Each dashboard's chat drawer has a **History** tab listing prior chat sessions tied to this dashboard (workspace-wide). Open any session for a read-only transcript so you can see what your team asked Atlas before — useful when picking up a dashboard mid-investigation.

### Draft retention

Per-user drafts are always on — every chat edit lands on the caller's private draft, and **Publish** promotes it. A draft is cleared the moment you publish or discard it; a draft you fork and then walk away from is reaped by the scheduler after `ATLAS_DASHBOARD_DRAFT_RETENTION_DAYS` (default `30`) days without an edit, so the drafts table can't grow unbounded. Set it to `0` to disable cleanup. It's a hot-reloadable setting (Admin → Settings → Dashboards) — no redeploy to change. See the [environment-variables reference](/reference/environment-variables#settings).

## API Reference

All dashboard operations are available via the REST API:

| Endpoint | Description |
|----------|-------------|
| `GET /api/v1/dashboards` | List dashboards |
| `POST /api/v1/dashboards` | Create a dashboard |
| `GET /api/v1/dashboards/:id` | Get dashboard with cards |
| `PATCH /api/v1/dashboards/:id` | Update title, description, schedule, or parameters |
| `DELETE /api/v1/dashboards/:id` | Delete a dashboard |
| `POST /api/v1/dashboards/:id/cards` | Add a card |
| `PATCH /api/v1/dashboards/:id/cards/:cardId` | Update card title or position |
| `DELETE /api/v1/dashboards/:id/cards/:cardId` | Remove a card |
| `POST /api/v1/dashboards/:id/cards/:cardId/refresh` | Refresh a single card (persists; uses parameter defaults) |
| `POST /api/v1/dashboards/:id/cards/:cardId/render` | Render a card with supplied parameter values (ephemeral) |
| `POST /api/v1/dashboards/:id/refresh` | Refresh all cards |
| `POST /api/v1/dashboards/:id/suggest` | Get AI card suggestions |
| `POST /api/v1/dashboards/:id/render` | Render the whole board with supplied parameter values (ephemeral; body-less returns defaults) |
| `POST /api/v1/dashboards/:id/preview-card` | Preview a card's SQL before saving (org-verifies the connection) |
| `POST /api/v1/dashboards/:id/export` | Export the whole dashboard to PDF or PNG |
| `POST /api/v1/dashboards/:id/screenshot` | Render a headless screenshot of the board |
