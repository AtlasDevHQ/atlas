Atlas caches query results in-memory with tenant-isolated, claim-scoped cache keys. This page covers operator-level configuration: the settings registry, environment variables, and plugging in an external cache backend.

<Callout title="Prerequisites">
- Atlas server running (`bun run dev`)
- For external backends: a Redis instance or compatible key-value store
</Callout>

For cache monitoring, stats, and flush operations, see [Cache Monitoring](/guides/caching).

---

## Configuration

The three cache knobs live in the **settings registry**, so they are tuned at
runtime with no restart:

- `ATLAS_CACHE_ENABLED` and `ATLAS_CACHE_TTL` are **workspace-scoped**
  (`workspace override > platform override > env var > default`) — each
  workspace tunes them from **Admin → the Cache page**.
- `ATLAS_CACHE_MAX_SIZE` is **platform-scoped** (the LRU is one process-wide
  backend shared by every workspace in the region) — a platform admin sets it.

A change to the TTL or max size is reconciled onto the running backend
immediately; the hit/miss counters survive.

### Environment Variables

On self-hosted, the env vars set the boot default for each knob (Admin
overrides still take precedence at runtime):

| Variable | Default | Description |
|----------|---------|-------------|
| `ATLAS_CACHE_ENABLED` | `true` | Enable/disable query result caching |
| `ATLAS_CACHE_TTL` | `300000` | Cache TTL in milliseconds (default: 5 minutes) |
| `ATLAS_CACHE_MAX_SIZE` | `1000` | Maximum cached entries. LRU eviction when full |

<Callout type="warn" title="The `atlas.config.ts` `cache:` block was removed">
The `cache:` block is gone from the config schema — a config file that still
carries one **fails validation at boot**. Delete the block and use **Admin →
the Cache page** or the `ATLAS_CACHE_*` env vars above. (It was deprecated —
ignored with a boot warning — for one release before removal.)
</Callout>

See [Configuration](/reference/config#query-cache) for full details.

## External Cache Backend (Redis)

Plugins can replace the default in-memory LRU cache with an external backend (e.g., Redis) by implementing the `PluginCacheBackend` interface from `@useatlas/plugin-sdk`. Every method is **`Promise`-returning**, so an out-of-process backend is actually implementable:

```typescript
import type {
  PluginCacheBackend,
  PluginCacheEntry,
  PluginCacheScope,
} from "@useatlas/plugin-sdk";

const redisCacheBackend: PluginCacheBackend = {
  async get(key: string): Promise<PluginCacheEntry | null> { /* ... */ },
  // `scope` carries the owning org + connection so entries can be
  // invalidated per-workspace, not just fleet-wide.
  async set(key: string, entry: PluginCacheEntry, scope: PluginCacheScope): Promise<void> { /* ... */ },
  async delete(key: string): Promise<boolean> { /* ... */ },
  async flush(): Promise<void> { /* every workspace's entries */ },
  async flushByOrg(orgId: string): Promise<number> { /* one org's entries; returns count removed */ },
  async stats() { /* return { hits, misses, entryCount, maxSize, ttl } */ },
};
```

Return `cacheBackend` from your plugin object and Atlas will use it instead of the built-in LRU. Only one plugin cache backend is supported per server — the first plugin with a `cacheBackend` wins.

The backend is **shape-validated at registration** (required methods present, plus a bounded `stats()` probe that must resolve promptly with the documented numeric shape). A backend that fails validation is never swapped in: the offending plugin fails its init and goes red in plugin health, while the cache degrades to the built-in LRU so queries keep working.

See [Plugin Authoring Guide](/plugins/authoring-guide) for building custom plugins.

## Troubleshooting

### Cache disabled but queries still appear cached

**Cause:** A plugin is providing a cache backend that ignores the `enabled` flag.

**Fix:** Check your plugin configuration. Remove or disable the cache backend plugin.

### External backend connection failures

**Cause:** The Redis (or other external) cache backend is unreachable.

**Fix:** Atlas degrades to the built-in LRU cache when the plugin backend fails registration validation, and the plugin shows red in plugin health. Check your Redis connection string and network connectivity. Review server logs for cache backend validation/initialization errors.
