Atlas's [Twenty CRM](https://twenty.com) integration is an **Action
Target**: Atlas upserts a Person (and optionally creates a Note) in
Twenty for each lead event — demo signups, sales-form submissions, new
user accounts. It is not a chat platform.

Plugin actions and queries read credentials **only** from per-workspace
settings — `atlas.config.ts` for self-hosted, or the `twenty_integrations`
row populated by Admin → Integrations → Twenty. The `TWENTY_API_KEY`
environment variable is reserved for Atlas's own SaaS lead-capture
pipeline (`ee/src/saas-crm/`), is never read by a plugin install
(customer workspace, or Atlas's own team workspace on app.useatlas.dev),
and never participates as a fallback (#2850).

Self-hosted operators have two install paths:

- **Admin UI (recommended)** — submit a `baseUrl` + `apiKey` through
  **Admin → Integrations → Twenty**. The key is encrypted at rest in
  the dedicated `twenty_integrations` table.
- **`atlas.config.ts`** — pass `apiKey` + `baseUrl` to `twentyPlugin({...})`
  in your config. Useful for ops-as-code setups where the operator
  threads `process.env.TWENTY_API_KEY_FOR_WORKSPACE_X` (or any other
  name) into the plugin config explicitly.

<Callout type="warn" title="No Atlas-SaaS default for Base URL">
  The admin UI form requires you to type your own Twenty hostname. It
  does **not** default to `https://crm.useatlas.dev` — that URL is
  Atlas's own internal Twenty instance, and a silent default would
  route every self-hosted operator's leads into Atlas's CRM. Enter the
  hostname of your own Twenty install (e.g. `https://crm.example.com`).
</Callout>

## Required Twenty setup (one-time)

Two **custom fields** must exist on the `Person` object in your Twenty
instance before Atlas can dispatch:

| Field name              | Required | Type | Purpose                                                                                          |
| ----------------------- | -------- | ---- | ------------------------------------------------------------------------------------------------ |
| `atlasFirstSource`      | yes      | Text | Sticky first-touch attribution — set once on a Person, never overwritten (e.g. `DEMO`).          |
| `atlasLastSource`       | yes      | Text | Most recent touch — overwritten on every dispatch (e.g. `DEMO`, `SIGNUP`, `SALES_FORM`).         |
| `atlasStripeCustomerId` | yes      | Text | Stripe `customer.id` of a paying customer; stamped on the trial→paid conversion path.            |
| `atlasIp`               | optional | Text | Client IP captured on demo / sales-form submissions; useful for fraud triage. Skip-safe.         |

Create them under **Twenty Settings → Data Model → Person → + Add
Field**. The Atlas SaaS layer runs a startup verification against
Twenty's REST OpenAPI spec (`/rest/open-api/core`) — missing **required**
fields are logged as a hard error and Atlas disables dispatch until the
schema is fixed. Missing **optional** fields (currently only `atlasIp`)
are silently dropped from the dispatch payload, so you can leave them
out without breaking lead capture.

The probe uses the REST OpenAPI spec rather than the metadata GraphQL
surface because the GraphQL schema has drifted between Twenty releases
and the old `ObjectFilter` shape no longer matches current Twenty. The
REST OpenAPI document is the authoritative, workspace-scoped contract.

## Option 1: Admin UI configuration

1. Open **Admin → Integrations** in the Atlas web app.
2. Find the **Twenty CRM** card and click **Install**.
3. Fill in:
   - **Base URL**: the hostname of your Twenty instance (required, no
     default). Both `https://` and `http://` are accepted — use `http`
     only for private-network / dev deployments.
   - **API key**: a bearer token from Twenty → **Settings → API &
     Webhooks**. The key is masked on submit and stored encrypted via
     AES-256-GCM (F-41 selective-field encryption).
4. Save. The next plugin action dispatches using the admin-configured
   credentials.

To remove the admin override, click **Disconnect** on the Twenty card.
Atlas drops both the `workspace_plugins` catalog row and the
`twenty_integrations` credential row. Subsequent plugin actions for
this workspace fail with an actionable `TwentyCredentialError` ("no row
in twenty_integrations — configure under Admin → Integrations →
Twenty") until either the admin UI is re-installed or the workspace
config in `atlas.config.ts` supplies an `apiKey`. **There is no
environment-variable fallback** — `TWENTY_API_KEY` is platform-only.

## Option 2: `atlas.config.ts` configuration

```ts
import { defineConfig } from "@atlas/api/lib/config";
import { twentyPlugin } from "@useatlas/twenty";

export default defineConfig({
  plugins: [
    twentyPlugin({
      // Required — point at YOUR Twenty install. There is no default;
      // a default URL would silently route a self-hosted operator at
      // Atlas's own internal CRM (`crm.useatlas.dev`).
      baseUrl: "https://crm.example.com",
      // Operator-supplied: thread your own secret through. Naming the
      // env var after the workspace makes multi-workspace ops obvious.
      apiKey: process.env.MY_TWENTY_API_KEY!,
    }),
  ],
});
```

This path is useful for ops-as-code workflows where install state lives
in version control rather than the admin UI. Multiple workspaces in the
same self-hosted deployment configure independent plugin instances.

## What about `TWENTY_API_KEY`?

The `TWENTY_API_KEY` (and optional `TWENTY_BASE_URL`) environment
variables are reserved for Atlas's own SaaS lead-capture pipeline at
`ee/src/saas-crm/` — the code that POSTs demo / sales-form / signup
events into Atlas's CRM at `crm.useatlas.dev`. **No plugin install**
— customer workspace, or Atlas's own team workspace on app.useatlas.dev
— reads from this env var, including as a fallback.

This split (#2850) makes two leak scenarios structurally impossible:

- **Direction 1** — A customer install with a missing apiKey can never
  silently route writes through Atlas's operator key, because plugin
  actions don't read env.
- **Direction 2** — A future change in `ee/src/saas-crm/` can't
  accidentally read a customer workspace's `twenty_integrations` row,
  because the platform path is env-only and `scripts/check-twenty-resolver-imports.sh`
  blocks the cross-import.

## Credential resolution

Plugin actions resolve credentials per workspace via
`resolveWorkspaceCredentials(workspaceId, { deployMode, lookup })`:

1. **`twenty_integrations` DB row for this workspace** — decrypted via
   `db/secret-encryption.ts` (F-41 selective-field encryption). The
   table is listed in `INTEGRATION_TABLES`, so F-47 key rotation and
   F-42 residue audits cover it automatically.
2. **Actionable `TwentyCredentialError`** — "no row in
   twenty_integrations — configure under Admin → Integrations →
   Twenty" (with a self-hosted variant that also mentions
   `atlas.config.ts`).

Decrypt failures (key rotation / corrupt ciphertext) fail closed: the
caller (e.g. the outbox dispatcher) dead-letters the row with a clear
message rather than silently routing to the wrong Twenty.

The decrypted API key is **never** returned from any GET endpoint —
the admin UI only displays a "configured" indicator + the (plaintext)
base URL.

## SaaS-mode encryption gate

When `ATLAS_DEPLOY_MODE=saas` is set, the install handler refuses to
persist credentials unless `ATLAS_ENCRYPTION_KEYS` is also configured.
Without the keyset, `encryptSecret` would fall back to plaintext (a
dev-only convenience), and a misconfigured SaaS deploy must fail
closed at the credential boundary. The error message points the
operator at the missing env var.
