## Verdict

**Sound periphery, spine reconnected — the model is still split.** The parts of this surface that are usually flimsy in a young feature are genuinely good and should be preserved wholesale:

- **The evidence tools are elevation-grade.** `profileTable` / `checkDataDistribution` enforce the table whitelist + content-mode gate and ride the unified profiler capability across plugin dbTypes (`packages/api/src/lib/tools/profile-table.ts:83-102`, `check-distribution.ts:41-69`); **both** proposal tools now run their test queries through the full `runUserQueryPipeline` (`validate-proposal.ts:166-170`, `propose-amendment.ts:164-168` — #3338, #4485).
- **The apply discipline is real, and the fix wave deepened it.** One baseline resolver for diff-preview and write-back — `resolveAmendmentBaseline`, with the contract "the row's OWN group is authoritative for every write-back" (`expert/apply.ts:61-111`, `:96-100`) — and one shared `applyAmendment` (upsert-by-identity, throw-on-missing-target; `apply.ts:338-400`), so preview and apply can no longer use divergent semantics or scopes (#4488). Auto-approve now applies in the same flow with revert-to-pending on failure (`propose-amendment.ts:235-271`, `revertAmendmentToPending` at `internal.ts:1657-1669`; #4486). Every apply snapshots a version and a rollback endpoint exists (`apply.ts:161-170`); the whitelist blast radius is structurally contained — no amendment type can touch `table:` or add an entity (`whitelist.ts:143-153`).
- **The governance seams exist, are tested, and now fail closed on SaaS.** Billing gate before any LLM spend with #3437's admin-not-exempt rationale (`admin-semantic-improve.ts:401-422`) — now mirrored on the wizard enrich path with token metering (#4489, `wizard.ts:884-921, 1046-1064`); `requirePermission("admin:semantic")` (`:377`); intent-based audit vocabulary (`lib/audit/actions.ts:298-301`); org-scoped review claims (`internal.ts:1770-1783`); settings-registry knobs, not raw env (`lib/settings.ts:581-633`). NULL-org rows are invisible and unreviewable on SaaS via `isSaasModeForGuard` — errored config treated as SaaS, i.e. fail-closed (`internal.ts:1502-1516, 1682-1688, 1724-1731, 1768-1775`) — and the scheduler that produces them is boot-guarded off in saas mode (`scheduler.ts:36-48`; #4487).
- **The admin-shell wiring is unusually complete.** Sidebar entry with a live pending-count badge (`admin-sidebar.tsx:52-78`), hub-page entry + health widget (`admin/semantic/page.tsx:855, 896`), command-palette parity, and the DiffViewer/proposal-card presentation (`improve/page.tsx`) is dark-mode-correct and well-shaped. New since the fixes: `proposals.ts` is a pure, unit-tested extraction module with the invariant "no `dbId` ⇒ no card" (`improve/proposals.ts:59-103`, `:73-74`).

The first pass called the spine severed: chat-proposal review was structurally dead (C1). The fix wave reconnected it — every rendered proposal now carries its `learned_patterns` row id and all reviews go through `/amendments/{id}/review` (#4484) — and converged the mechanics: one resolver, one apply, one query pipeline, one billing gate. **What it deliberately did not touch is the model.** The surface still has two identities for "a proposal": the DB amendment row that now does all the work, and an in-memory, index-keyed session subsystem that survives as vestigial-but-shipped surface — a per-process Map, four dead routes published in OpenAPI and the docs, `sessionId` wire fields the client still round-trips, and a UI source-swap that hides the pending queue whenever a chat is live. Around it, the same seam questions stand: no staleness guarantee between the diff the admin approved and the document apply mutates; the expert persona still smuggled in as a "Warnings" bullet; the agent still learning nothing from decisions. The grill is still needed — and its agenda is sharper now.

## Ranked findings (surviving, re-verified post-fix)

### HIGH

#### H1 — Dual proposal identity: the DB row now does all the work, but the in-memory session subsystem ships on as vestigial surface `verified` (doc-only — the grill's centerpiece, now a delete-or-make-real decision)
The user-facing breakage is gone; the split model remains, downgraded from "severed spine" to "dead machinery still bolted to the airframe":
- **The Map and its empty sessions survive unchanged:** module-level `sessions = new Map<…>()` (`admin-semantic-improve.ts:47`); the chat route still creates every web session with `createSession([])` (`:474`) and nothing in the API ever populates `state.proposals` — the only populator in the repo remains CLI interactive mode (`packages/cli/lib/improve/interactive.ts:221`). The web page itself now documents the corpse: "the in-memory `/proposals/{index}` route is never populated in the web flow and always 404s" (`page.tsx:315-319`).
- **Four dead routes, still OpenAPI-published and documented:** `GET /sessions` (`:533-549`, full-Map iteration per request), `GET /sessions/{id}` (`:552-572`), `POST /proposals/{id}/approve` (`:575-638` — carrying a working YAML-apply branch at `:602-603` that can never be reached), `POST /proposals/{id}/reject` (`:641-678`). Zero product consumers (web/SDK/CLI) — only tests reference them. Shipped in `apps/docs/openapi.json:62822, 62927, 63131, 63293` and generated API-reference pages; the guide documents them as the review API (L1).
- **Latent hazards, inert but undeleted:** `findSessionForProposal`'s most-recent-session-for-org fallback (`:322-346`, fallback `:336-345`) and `advanceAndRecord`'s forced-skip loop (`:352-368`) are unreachable today only because every web session's proposals array is empty — repopulate it and the cross-admin cross-talk and silent-skip semantics return.
- **Durability/eviction unchanged:** per-process Map wiped on restart; unknown `sessionId` silently mints a fresh session (`:468-479`, returned via `x-session-id` `:513`; client re-syncs `page.tsx:242-245`); `sessions.set` (`:478`) is the only mutation — no TTL, no cap, one entry per chat session until restart. All three SaaS regions still pin `numReplicas: 1` (`deploy/api*/railway.json:35-36`) — a second coupling to that pin alongside the MCP session map (#2109, ADR-0020). Blast radius is smaller now (the session carries no proposal state in the web flow), but the machinery is pure liability.
- **The UI source-swap survives:** `const proposals = chatProposals.length > 0 ? chatProposals : pendingAmendments` (`page.tsx:286`) — an active chat hides pre-existing DB pending rows until reload. What's fixed: both sources share the DB identity, decided chat proposals flip their real row so `/pending-count` no longer inflates, and the client decision Map (`page.tsx:225-227, 330-332`) is now cosmetic display state over authoritative DB truth.
- **Failure scenario today:** no clicks break — but the surface ships a documented API that always 404s, a session abstraction that holds nothing, and a panel that hides the queue mid-session; every future change pays the two-model tax.

#### M1 — Decision-path atomicity: review approve is apply-then-claim; auto-approve is insert-then-apply; the invariant is caller discipline (doc-only)

## Grill agenda

The design questions the findings force — walk this list in the `/grill-with-docs` session. (Renumbered questions keep their first-pass numbers; the fix wave sharpened rather than answered them.)

1. **The session subsystem: delete or make real?** The fix chose the DB row as the review identity; nothing consumes sessions anymore. Do the Map, the four dead routes (`/sessions`, `/sessions/{id}`, `/proposals/{index}/approve|reject` — OpenAPI-published, docs-documented, always-404), the `sessionId`/`x-session-id` wire fields, and the `improveAccept` audit action get deleted — or does the session become durable (ADR-0020 precedent) and populated? Deleting is also an OpenAPI/docs contract change — how is it staged? (H1, M9, L1)
2. **What does the admin hold while reviewing — one list or two?** The source-swap survives (`page.tsx:286`): a live chat hides pre-existing pending rows. Should the panel unify live-session and pending-queue items (they are now the same DB rows) instead of swapping sources? What does "(N pending)" mean when both exist? (H1, M6)
3. **When an admin decides, what does the agent learn — and how?** Web decisions now durably feed the *scheduler's* suppression, but the chat agent still learns nothing: no synthetic message, no rejected-keys in the system prompt, no dedup at insert (`Date.now()`-uniquified keys), and the CLI's key format has drifted from context-loader's. What backs the route's "will not re-suggest" promise for the chat path, and do the three rejected-key implementations converge? (H7)
4. **What guarantees approve-what-you-saw, end to end?** Scope can no longer diverge (#4488) — but nothing excludes a concurrent write between propose and approve (`apply.ts:38-40` documents this deliberately). Base-content hash with 409-on-mismatch, re-diff at review time, or accept last-write? In the same seam: claim-then-apply ordering for the review route, apply-then-mark (or a reconciler) for auto-approve's insert-then-apply window, "approved ⇒ applied" by construction rather than caller discipline, which connection the test query targets (M12), and whether masked sample rows belong at rest in `learned_patterns` at all. (M1, M12, H4/H3 residue)
5. **Is the expert agent a *mode* of `runAgent` or a Warnings bullet?** What deterministic context gets front-loaded (health score, audit patterns, analyzer output, prior decisions) vs tool-discovered? Does the analyzer pipeline serve the chat path at all, and should `maxSteps` follow the workspace knob? (H6)
6. **What is the scheduler's affirmative SaaS story?** It is now boot-guarded OFF on SaaS, fail-closed (#4487) — so the question is no longer "how do we stop the leak" but "does SaaS ever get autonomous improvement": per-org ticks against DB entities, or self-hosted-only forever? Should "global scope" stay a NULL sentinel or become an explicit tier (the guard is hand-repeated in three queries — a shared `orgScopeClause` helper is the ratchet)?
