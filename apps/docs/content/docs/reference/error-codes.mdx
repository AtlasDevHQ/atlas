Atlas uses structured error codes across its API, SDK, and chat interface. Every error response includes a `code` field identifying the failure, a human-readable `message`, and a `retryable` flag indicating whether the client should retry.

<Callout>
Error codes are defined in [`@useatlas/types/errors`](https://github.com/AtlasDevHQ/atlas/blob/main/packages/types/src/errors.ts) and shared across all Atlas packages.
</Callout>

## Server Error Codes (ChatErrorCode)

These codes are returned by the Atlas API in JSON error responses. Each maps to an HTTP status code and a retryable classification.

### Retryable Errors

Transient failures where retrying the same request may succeed. Use exponential backoff for retries.

| Code | HTTP Status | Description | Common Cause | Fix |
|------|-------------|-------------|--------------|-----|
| `rate_limited` | 429 | Too many requests | Client exceeding the per-user rate limit | Wait for `retryAfterSeconds` then retry. Reduce request frequency |
| `provider_rate_limit` | 503 | AI provider rate limited | Upstream LLM provider (Anthropic, OpenAI, etc.) throttling requests | Wait and retry with backoff. Consider upgrading your provider plan |
| `provider_timeout` | 504 | AI provider timed out | LLM took too long to respond, or query exceeded `ATLAS_QUERY_TIMEOUT` | Retry with a simpler question. Increase `ATLAS_QUERY_TIMEOUT` if queries are complex |
| `provider_unreachable` | 503 | Cannot reach the AI provider | Network issue between Atlas and the LLM provider, or provider outage | Check provider status page. Retry after a short delay |
| `provider_error` | 502 | AI provider returned an error | Unexpected error from the LLM provider (500, malformed response, etc.) | Retry after a short delay. Check provider status if persistent |
| `internal_error` | 500 | Server error | Unhandled exception, database connection failure, or pool exhaustion | Retry after a short delay. Check server logs using the `requestId` |

### Permanent Errors

Retrying these will not help â€” the request or configuration must be changed.

| Code | HTTP Status | Description | Common Cause | Fix |
|------|-------------|-------------|--------------|-----|
| `auth_error` | 401 | Authentication failed | Invalid API key or revoked token | Check your API key or sign in again. See [Authentication](/deployment/authentication) |
| `session_expired` | 401 | Session expired | Session token has expired or been revoked | Sign in again to get a new session |
| `forbidden` | 403 | Access denied | User lacks the required role (e.g., admin endpoints require `admin` role) | Request the appropriate role from your Atlas administrator |
| `forbidden_role` | 403 | Admin role required | Non-admin user tried to access an admin endpoint | Sign in with an admin account or request the admin role |
| `org_not_found` | 400 | No active organization | Request requires an active organization but none is selected | Select an organization in the org switcher and try again |
| `configuration_error` | 400 | Atlas is not fully configured | Missing environment variables, invalid config file, or startup diagnostics failed | Run `atlas doctor` to identify the issue. See [Troubleshooting](/guides/troubleshooting) |
| `no_datasource` | 400 | No datasource configured | `ATLAS_DATASOURCE_URL` is not set | Set `ATLAS_DATASOURCE_URL` in your environment. See [Environment Variables](/reference/environment-variables) |
| `invalid_request` | 400 | Invalid request | Malformed JSON body or missing required fields | Check the request body format. See [API Reference](/reference/api) |
| `validation_error` | 422 | Request validation failed | Request body doesn't match the expected schema (wrong types, missing fields) | Check the `details` field for specific field errors |
| `not_found` | 404 | Resource not found | Conversation, entity, or other resource doesn't exist or isn't owned by the caller | Verify the resource ID. The resource may have been deleted |
| `conversation_budget_exceeded` | 429 | Conversation has reached its step ceiling | Conversation hit the per-conversation step cap (`ATLAS_CONVERSATION_STEP_CAP`, default 500) â€” F-77 atomic budget | Start a new conversation. The current thread cannot be continued |
| `plan_limit_exceeded` | 429 | Plan limit exceeded | Workspace has exceeded its plan's query or token limit (including the 10% grace buffer) | Upgrade your plan or wait until the next billing period |
| `provider_model_not_found` | 400 | AI model not found | The model specified in `ATLAS_MODEL` doesn't exist at the configured provider | Check `ATLAS_MODEL` and `ATLAS_PROVIDER` values. See [Environment Variables](/reference/environment-variables) |
| `provider_auth_error` | 503 | AI provider authentication failed | Invalid or expired LLM provider API key (e.g., `ANTHROPIC_API_KEY`) | Verify your provider API key. Regenerate it if expired |

### Billing & Workspace Errors

These codes are returned by billing enforcement and workspace status middleware. They block requests before the agent loop runs.

| Code | HTTP Status | Description | Common Cause | Fix |
|------|-------------|-------------|--------------|-----|
| `trial_expired` | 403 | Trial has expired | 14-day SaaS trial ended | Upgrade to a paid plan |
| `billing_check_failed` | 503 | Billing check failed | Failed to fetch plan data from internal DB | Retry â€” transient infrastructure issue |
| `workspace_check_failed` | 503 | Workspace check failed | Failed to verify workspace status | Retry â€” transient infrastructure issue |
| `workspace_throttled` | 429 | Workspace throttled | Workspace triggered abuse detection thresholds | Wait for the throttle delay to pass and retry. See [Abuse Prevention](/platform-ops/abuse-prevention) |
| `workspace_suspended` | 403 | Workspace suspended | Admin suspended the workspace | Contact your workspace administrator |
| `workspace_deleted` | 404 | Workspace deleted | Workspace has been permanently deleted | Create a new workspace |

## Route-Response Error Codes

The `ChatErrorCode` catalog above is scoped to the chat-stream endpoint â€” the agent loop enforces it exhaustively at compile time via `Record<ChatErrorCode, boolean>` in [`packages/types/src/errors.ts`](https://github.com/AtlasDevHQ/atlas/blob/main/packages/types/src/errors.ts). Admin, mode, favorites, and regional-routing endpoints return additional codes that follow the same `{ error, message, requestId }` envelope but aren't consumed by the agent's retry logic. Clients parsing responses from those endpoints should handle the codes below.

### Mode & workspace state

| Code | HTTP Status | Trigger | Fix | Retryable |
|------|-------------|---------|-----|-----------|
| `demo_readonly` | 403 | Write against the `__demo__` connection while in published mode | Switch to developer mode to manage demo content. See [Developer Mode](/guides/developer-mode) | No |
| `workspace_migrating` | 409 | Write attempted while the workspace's data is being moved to a new region | Transient â€” retry once the migration completes (typically minutes). See [Data Residency](/platform-ops/data-residency) | Yes |
| `misdirected_request` | 421 | Request reached a regional API that doesn't hold this workspace's data | Use the `correctApiUrl` field in the response body to retarget. The SDK can be initialized with the correct regional URL once and cached. See [Regional API](/platform-ops/regional-api) | No |

### Starter-prompt favorites

| Code | HTTP Status | Trigger | Fix | Retryable |
|------|-------------|---------|-----|-----------|
| `duplicate_favorite` | 409 | Pinning a prompt whose text already exists in the caller's favorites | Already pinned â€” no action needed | No |
| `favorite_cap_exceeded` | 400 | User has pinned the maximum number of favorites | Unpin a prompt first, or raise `ATLAS_STARTER_PROMPT_MAX_FAVORITES` (self-hosted) | No |
| `invalid_favorite_text` | 400 | Favorite text is empty, too long, or otherwise malformed | Fix the client-side validation; trim whitespace and enforce length limits before POSTing | No |

### Content-mode registry (#1515)

These codes surface from the `ContentModeRegistry` service that drives the draft/published mode system (connections, prompt collections, starter prompts, semantic entities). They typically indicate an infrastructure or configuration problem rather than a user error.

| Code | HTTP Status | Trigger | Fix | Retryable |
|------|-------------|---------|-----|-----------|
| `upstream_error` | 500 | Publish phase failed (`PublishPhaseError`), unknown content-mode table (`UnknownTableError`), or an exotic table missing its `readFilter` adapter (`ExoticReadFilterUnavailableError`) | Check server logs with the `requestId`. For publish failures, the whole transaction rolled back â€” no partial state. For unknown-table errors, the caller passed a table name not registered in `CONTENT_MODE_TABLES` (server-side bug). | Yes (transient) / No (config bug) |

`upstream_error` is also used for generic DB execution failures (`QueryExecutionError`) and chat/delivery failures from the scheduler â€” it covers any 500-class issue that originated at an upstream layer (DB, LLM, adapter) rather than in Atlas's own request handling.

<Callout>
These codes aren't part of the exhaustively-typed `ChatErrorCode` catalog, so the SDK's `isChatErrorCode()` / `isRetryableError()` guards return `false` / `undefined` for them. Parse the raw `error` field yourself when calling endpoints that may return them: `POST /api/v1/starter-prompts/favorites`, `/api/v1/admin/connections/*`, `/api/v1/admin/semantic/*`, and any write request that hits the regional-routing or migration middleware (including the chat stream itself for `workspace_migrating` / `misdirected_request`).
</Callout>

### Install & OAuth (#2742, #2659)

The workspace installer (`POST /api/v1/admin/installs`, `DELETE /api/v1/admin/installs/:catalogSlug`, `PATCH .../config`) and the lazy-OAuth refresh flow that backs Salesforce, Jira, and Linear surface the codes below. Tags are defined in [`packages/api/src/lib/effect/errors.ts`](https://github.com/AtlasDevHQ/atlas/blob/main/packages/api/src/lib/effect/errors.ts); per-status mapping lives in [`mapTaggedError`](https://github.com/AtlasDevHQ/atlas/blob/main/packages/api/src/lib/effect/hono.ts).

| Code | `_tag` | HTTP Status | Trigger | Fix | Retryable |
|------|--------|-------------|---------|-----|-----------|
| `conflict` | `AlreadyInstalledError` | 409 | Pillar-singleton violation â€” a `chat` or `action` install already exists for `(workspaceId, catalogSlug)`, or a `datasource` install with the same `installId` already exists. Body carries `catalogSlug` and `pillar`. Backed by the `workspace_plugins_singleton` partial unique index as the defensive backstop | Uninstall the existing row before reinstalling, or pick a different `installId` for datasource installs. The admin UI surfaces "already installed; disconnect first to reinstall" | No |
| `bad_request` | `ConfigSchemaError` | 400 | Submitted `config` failed validation against `plugin_catalog.config_schema`. Body carries `fieldErrors` (per-field message arrays) and `formErrors` (top-level issues) so the admin form can highlight wrong inputs without a follow-up roundtrip | Fix the highlighted fields in the install form. Missing required fields and wrong-typed values are catalog-declared and surface here before the per-handler Zod layer runs | No |
| `not_found` | `CatalogNotFoundError` | 404 | The requested catalog row is missing or kill-switched. Body carries `catalogSlug` | Verify the slug. Kill-switched catalogs require an operator to re-enable the row in `plugin_catalog` | No |
| `not_found` | `InstallNotFoundError` | 404 | `uninstall` or `updateConfig` targeted an install row that doesn't exist for `(workspaceId, catalogSlug)`. Body carries `catalogSlug`. Distinct from `CatalogNotFoundError` (the catalog itself is missing) so the admin UI can render the right toast | Refresh the installs list â€” the row may have been removed by another admin | No |
| `bad_request` | `InvalidInstallIdError` | 400 | Datasource install's user-facing `installId` slug failed validation. Body carries `installId` and `reason` (`"pattern"` for invalid characters, `"reserved"` for sentinel collisions like `default`) | Use `^[a-z][a-z0-9_-]*$`. The `__demo__` sentinel is the only pre-cutover exception preserved by migration 0094 | No |
| `upstream_error` | `PlatformOAuthExchangeError` | 502 | OAuth code-for-token exchange failed at the upstream Platform (Slack's `oauth.v2.access`, Jira's `oauth/token`, etc.) â€” non-OK response, network failure, or malformed payload. Raw upstream error code stays in logs only; the response message is translated | Check the Platform's status page. Restart the install flow once it recovers. If the failure is persistent and the Platform is healthy, verify the Connected App / OAuth App credentials in operator config | Yes |
| `conflict` | `IntegrationReconnectRequiredError` | 409 | A lazy-OAuth integration's refresh-token rotation failed permanently. The `platform` field names which one: **Salesforce** (`invalid_grant`, `inactive_user`, `org_locked`, `inactive_org`), **Jira** (Atlassian `invalid_grant`, stored refresh token rejected, or `offline_access` scope dropped), or **Linear** (`invalid_grant`, refresh token rejected, or `actor=app` scope revoked). The install's `workspace_plugins.config.status` has already been flipped to `"reconnect_needed"` | Re-run the named integration's OAuth flow from the admin Installs page. The stored refresh token is no longer accepted upstream and no amount of retrying will recover it â€” verify the App still grants the required scopes (`offline_access` for Jira, `actor=app` for Linear) before reinstalling | No |

<Callout>
The `conflict` wire code is reused for `IntegrationReconnectRequiredError` rather than introducing a `reconnect_required` code â€” the `_tag` plus the error's `platform` field in logs, and the admin UI's install-status surface (`workspace_plugins.config.status === "reconnect_needed"`), disambiguate. One tag covers Salesforce, Jira, and Linear (collapsed from three per-platform tags at the rule of three â€” see #2708).
</Callout>

### Platform reachability â€” Telegram, Discord, Teams (#2748, #2749, #2752)

Static-bot install handlers verify the supplied routing identifier (Telegram `chat_id`, Discord `guild_id`, Teams `tenant_id`) against the upstream Platform before persisting the install. Input-shape failures and reachability failures map to 400 because both are admin-correctable; only network-layer outages get the 502 retryable surface.

| Code | `_tag` | HTTP Status | Trigger | Fix | Retryable |
|------|--------|-------------|---------|-----|-----------|
| `bad_request` | `TelegramChatIdInvalidError` | 400 | `chat_id` is not a numeric integer, empty, or pasted as `@username` | Re-paste the chat id as a signed integer (negative for groups, positive for users). The Telegram bot dialog shows the id when you message `/start` to `@RawDataBot` | No |
| `bad_request` | `TelegramReachabilityError` | 400 | Telegram Bot API returned a non-OK envelope to `getChat` â€” chat not found, bot not a member, or wrong chat type. Message includes Telegram's verbatim `description` | Re-paste the id, add the bot to the chat, or fix the bot token. The verbatim Telegram message is in `message` | No |
| `upstream_error` | `TelegramApiUnavailableError` | 502 | Telegram Bot API was unreachable at the network layer â€” DNS, TLS, timeout, or malformed response | Retry after a short delay. If persistent, check Telegram's status page | Yes |
| `bad_request` | `DiscordGuildIdInvalidError` | 400 | `guild_id` is not a 17â€“20 digit snowflake â€” pasted as an invite code or server name | Enable Developer Mode in Discord, right-click the server, and copy the numeric server ID | No |
| `bad_request` | `DiscordReachabilityError` | 400 | Discord API returned a non-OK envelope to `GET /api/v10/guilds/{guild_id}` â€” Unknown Guild, Missing Access, bot not in guild. Message includes Discord's verbatim text | Re-paste the id, re-run the bot install link to grant the bot access, or fix the bot token | No |
| `upstream_error` | `DiscordApiUnavailableError` | 502 | Discord API was unreachable at the network layer | Retry after a short delay. If persistent, check Discord's status page | Yes |
| `bad_request` | `TeamsTenantIdInvalidError` | 400 | `tenant_id` is not a Microsoft Entra ID tenant GUID (8-4-4-4-12 hex digits), empty, or pasted as a domain like `contoso.onmicrosoft.com` | Paste the tenant GUID from the Entra ID admin portal, not the verified domain | No |
| `bad_request` | `TeamsReachabilityError` | 400 | Microsoft tenant discovery returned a non-OK response â€” the tenant doesn't exist, has been deleted, or is unresolvable. The upstream `status` is logged with `requestId` for forensics but is not part of the response body | Verify the tenant GUID in the Entra ID admin portal. Wrong GUIDs are the most common cause | No |
| `upstream_error` | `TeamsApiUnavailableError` | 502 | Microsoft tenant discovery endpoint unreachable at the network layer | Retry after a short delay. If persistent, check Azure status | Yes |

### Scheduler & Delivery

The scheduler runs proactive prompts and report deliveries on workspace cron schedules. Per-task errors surface to the admin scheduler page; delivery errors surface to the recipient channel's history view.

| Code | `_tag` | HTTP Status | Trigger | Fix | Retryable |
|------|--------|-------------|---------|-----|-----------|
| `timeout` | `SchedulerTaskTimeoutError` | 504 | Scheduled task exceeded its wall-clock budget. `taskId` and `timeoutMs` are on the error instance and surface in the scheduler run history + structured logs, not in the HTTP response body | Reduce the work the task performs (narrower window, fewer rows) or raise the per-task timeout in scheduler config. Long-running deliveries should be split into batches | Yes |
| `upstream_error` | `SchedulerExecutionError` | 500 | Scheduled task threw during execution â€” agent loop failed, query failed, or template rendering crashed. `taskId` and `runId` are on the error instance; surface in scheduler run history + logs, not the response body | Check the task's run history for the failed `runId` and the structured log with that `requestId`. Re-run manually from the scheduler page once the underlying issue (datasource down, prompt change, etc.) is resolved | Yes |
| `upstream_error` | `DeliveryError` | 502 | Delivery to a recipient channel failed â€” Slack/Teams/Discord/email/webhook adapter returned an error. The error instance carries `channel`, `recipient`, and `permanent` (true = no auto-retry, e.g. bot kicked from channel; false = transient, e.g. 503 from upstream); these fields are persisted to the delivery-history table and structured logs but are **not** in the HTTP response body | Operators read the delivery-history table or the scheduler admin page to see `permanent`. When `permanent: true`, fix the channel state (re-add the bot, fix the webhook URL) and re-run. When `permanent: false`, the scheduler auto-retries on the next tick | Server-side scheduler decides â€” clients receive only `{ status, code, message }` |

### Region migration

| Code | `_tag` | HTTP Status | Trigger | Fix | Retryable |
|------|--------|-------------|---------|-----|-----------|
| `conflict` | `UnsafeRegionMigrationResetError` | 409 | Operator attempted to reset a region migration that had already passed Phase 3 (cutover). Re-running Phase 1 on a workspace that already moved would re-export stale data and corrupt the destination. `migrationId`, `workspaceId`, `targetRegion`, and `sourceRegion` are on the error instance and structured logs (correlate via `requestId`), not in the response body | Follow the data-residency manual-intervention runbook in [Data Residency](/platform-ops/data-residency). The code path is closed by design; this is not retryable | No |

| `entity_ambiguous` | `AmbiguousEntityError` | 409 | `getEntity(orgId, type, name)` matched more than one row because the same `(org, entity_type, name)` triple exists in multiple `connection_group_id` scopes. ### Governance & Enterprise

These cover the always-403 governance gates and the 503 fallback raised when an enterprise feature is enabled but its EE Layer failed to bind. See [Enterprise & SaaS Gating](/platform-ops/enterprise) for the broader feature-flag model.

| Code | `_tag` | HTTP Status | Trigger | Fix | Retryable |
|------|--------|-------------|---------|-----|-----------|
| `forbidden` | `EnterpriseGateError` | 403 | Caller tried to use an enterprise feature on a self-hosted install. The `feature` field (e.g. `"sso"`, `"residency"`) is on the error instance and structured logs but is not in the HTTP response body â€” the surfaced `message` typically names the feature | Enable enterprise (`ATLAS_ENTERPRISE_ENABLED=true` with the `/ee` module present) or remove the call site. Self-hosted users get the full core product without enterprise governance | No |
| `forbidden` | `ApprovalRequiredError` | 403 | Query matched one or more approval rules and must be reviewed before execution. The matched `rules` array is on the error instance and structured logs but is not in the HTTP response body | Submit the query for approval via the admin Approvals page, or rewrite to avoid the matched rules (e.g. narrower row count, different tables) | No |
| `enterprise_load_failed` | `EnterpriseUnavailableError` | 503 | `ATLAS_ENTERPRISE_ENABLED=true` but the consumer Tag still reports `available: false` â€” the `@atlas/ee/layers` dynamic import in `ConditionalEELayer` failed at boot. The `tag` field is on the error instance and the `enterprise.load_failed` structured log (correlate via `requestId`), not in the response body. Distinct wire code from `service_unavailable` so SaaS monitoring can split this signal from generic infra outages | Operator-side: check boot logs for `event: "enterprise.load_failed"` and the underlying import error (typically a missing dep or a build that didn't include `/ee`). Self-hosted installs never raise this â€” they leave `ATLAS_ENTERPRISE_ENABLED` unset and the no-op default short-circuits cleanly | Yes â€” after operator intervention; client retry without operator action is futile |

<Callout>
`EnterpriseError` (the `requireEnterprise()` / `requireEnterpriseEffect()` guard) is **not** in `ATLAS_ERROR_TAG_LIST` â€” `lib/effect/hono.ts:isEnterpriseError` duck-types on `err.name === "EnterpriseError"` and maps the response to 403 with `code: "enterprise_required"` before `mapTaggedError` runs. Treat it as equivalent to `EnterpriseGateError` for client-side handling.
</Callout>

## Client Error Codes (ClientErrorCode)

These codes are detected **client-side** by the SDK and chat UI before parsing a server response. They represent network-level failures or HTTP status patterns.

| Code | Description | Common Cause | Fix | Retryable |
|------|-------------|--------------|-----|-----------|
| `api_unreachable` | Cannot connect to the Atlas API | Network failure, DNS resolution error, server not running, or CORS issue | Check the API URL configuration and ensure the server is running | Yes |
| `auth_failure` | HTTP 401 detected before JSON parsing | API key not sent, expired session token, or wrong auth header format | Check your API key or sign in again | No |
| `rate_limited_http` | HTTP 429 detected before JSON parsing | Rate limit hit â€” the response wasn't valid JSON (e.g., from a reverse proxy) | Wait 30 seconds and retry | Yes |
| `server_error` | HTTP 5xx detected before JSON parsing | Server crashed or upstream proxy error (502, 503) | Retry after a short delay | Yes |
| `offline` | Browser reports no network connection | Device is offline (detected via `navigator.onLine === false`) | Reconnect to the network â€” the client may auto-retry when connectivity is restored | Yes |

<Callout>
Client error codes appear in the `clientCode` field of `ChatErrorInfo`, while server error codes appear in the `code` field. When the server returns valid JSON with a known error code, the server code takes precedence.
</Callout>

## SDK Error Codes

The `@useatlas/sdk` includes additional codes in the `AtlasErrorCode` type beyond `ChatErrorCode`. Three are client-side codes detected by the SDK itself (never returned by the server). One (`not_available`) is returned by server admin and conversation endpoints.

| Code | Description | Origin | Retryable |
|------|-------------|--------|-----------|
| `network_error` | `fetch()` threw an error (connection refused, DNS failure, stream interrupted) | SDK client-side | Yes |
| `invalid_response` | Server returned a 2xx status but the body wasn't valid JSON | SDK client-side | No |
| `unknown_error` | Server returned an error response with an unrecognized error code | SDK client-side | No |
| `not_available` | Feature not available (e.g., conversation history without `DATABASE_URL`) | Server (admin/CRUD routes) | No |

## Error Response Format

All Atlas API error responses follow a consistent JSON structure:

```json
{
  "error": "rate_limited",
  "message": "Too many requests. Please wait before trying again.",
  "retryAfterSeconds": 12,
  "retryable": true,
  "requestId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `error` | `string` | The error code (one of the codes listed above) |
| `message` | `string` | Human-readable description of the error |
| `retryable` | `boolean` | Whether retrying the same request may succeed |
| `retryAfterSeconds` | `number?` | Seconds to wait before retrying (only for `rate_limited`; client-side parsing clamps to 0â€“300) |
| `requestId` | `string?` | Server-assigned UUID for log correlation. Quote this when reporting issues |
| `details` | `object?` | Additional context (e.g., Zod validation issues for `validation_error`) |
| `diagnostics` | `object[]?` | Startup diagnostic results (only for `configuration_error`) |

## Error Handling with Retry Logic

Use the `retryable` flag on `AtlasError` to build generic retry logic without hard-coding error codes:

```typescript
import { AtlasError, createAtlasClient, type QueryResponse } from "@useatlas/sdk";

const atlas = createAtlasClient({
  baseUrl: "https://api.example.com",
  apiKey: "your-api-key",
});

function sleep(ms: number): Promise<void> {
  return new Promise((resolve) => setTimeout(resolve, ms));
}

async function queryWithRetry(
  question: string,
  maxRetries = 3,
): Promise<QueryResponse> {
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      return await atlas.query(question);
    } catch (error) {
      if (!(error instanceof AtlasError)) throw error;

      // Permanent error â€” retrying won't help
      if (!error.retryable) throw error;

      // Last attempt â€” no more retries
      if (attempt === maxRetries) throw error;

      // Rate limited â€” use the server-provided delay
      if (error.code === "rate_limited" && error.retryAfterSeconds) {
        console.log(`Rate limited â€” waiting ${error.retryAfterSeconds}s`);
        await sleep(error.retryAfterSeconds * 1000);
        continue;
      }

      // Other transient errors â€” exponential backoff
      const delay = Math.min(1000 * 2 ** attempt, 30_000);
      console.log(`${error.code} â€” retrying in ${delay}ms (attempt ${attempt + 1}/${maxRetries})`);
      await sleep(delay);
    }
  }

  // Unreachable, but satisfies TypeScript
  throw new Error("Retry loop exited unexpectedly");
}
```

### Handling Specific Codes

Refine behavior for specific error codes after checking `retryable`:

```typescript
try {
  await atlas.query("Revenue by region");
} catch (error) {
  if (!(error instanceof AtlasError)) throw error;

  switch (error.code) {
    case "auth_error":
    case "session_expired":
      // Redirect to login or prompt for a new API key
      redirectToLogin();
      break;
    case "forbidden_role":
      // User doesn't have admin access
      showPermissionDenied(error.message);
      break;
    case "configuration_error":
    case "no_datasource":
      // Show setup instructions â€” the server isn't ready
      showSetupGuide(error.message);
      break;
    case "provider_model_not_found":
    case "provider_auth_error":
      // Admin needs to fix server config
      showAdminAlert(error.message);
      break;
    default:
      if (error.retryable) {
        // Generic transient error â€” show retry UI
        showRetryButton(error.message);
      } else {
        // Permanent error â€” show the message to the user
        showError(error.message);
      }
  }
}
```
