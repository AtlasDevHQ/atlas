Self-hosted Atlas exposes MCP over stdio (see the [MCP guide](/guides/mcp)). Atlas SaaS adds a hosted MCP endpoint over Streamable HTTP that speaks **OAuth 2.1 + PKCE + Dynamic Client Registration (RFC 7591)** ŌĆö the authorization model the [MCP spec](https://modelcontextprotocol.io/specification/2025-03-26/basic/authorization) requires.

There are three paths to connect ŌĆö pick whichever fits your context. **All three end up at the same place**: a workspace-scoped JWT in your agent's config and a row in your account at **Settings ŌåÆ AI Agents**.

## Option 1 ŌĆö In the workspace UI (recommended for most users)

Sign into Atlas, open **Settings ŌåÆ AI Agents**, and click **Connect new agent**. The 3-step wizard:

1. Pick your agent ŌĆö Claude Desktop, Cursor, Continue, ChatGPT, or any generic MCP client
2. Review what the agent will request (scopes are workspace-read by default)
3. Copy the JSON config block, paste it into your agent's MCP settings, restart

That's it. When the agent first connects it bootstraps itself via DCR + PKCE, opens your browser to approve, and stores the resulting JWT. The new agent shows up in the same Settings page with a Revoke button.

This is the right path if you're a workspace user who isn't comfortable in a terminal ŌĆö or if you just want the click path. No admin involvement required.

## Option 2 ŌĆö The CLI installer (one command, power-user path)

```bash
bunx @useatlas/mcp init --hosted --write
```

That's it. The installer:

1. Discovers Atlas's authorization server at `/.well-known/oauth-authorization-server/api/auth`
2. Registers itself as a public OAuth client via DCR (no admin pre-issuance needed)
3. Opens your browser to `/oauth2/authorize` with PKCE (same shape as `gh auth login` / `gcloud auth login`)
4. Captures the callback on a one-shot `127.0.0.1` listener
5. Exchanges the auth code for a JWT access token
6. Merges a workspace-scoped server block into your MCP client config (Claude Desktop, Cursor, or Continue ŌĆö auto-detected) at mode `0600`

A `.bak` of any prior config is written first. Restart your MCP client to pick up the new server.

```bash
# Print to stdout instead of writing
bunx @useatlas/mcp init --hosted

# Force a specific client + override the API region
bunx @useatlas/mcp init --hosted --client cursor --api-url https://api-eu.useatlas.dev --write
```

The resulting block looks like:

```json
{
  "mcpServers": {
    "atlas": {
      "url": "https://api.useatlas.dev/mcp/<your_workspace_id>/sse",
      "headers": { "Authorization": "Bearer eyJhbGciOiJSUzI1NiIsŌĆ”" }
    }
  }
}
```

The `code_verifier` and the registered `client_id` are kept in memory only ŌĆö never written to disk. The JWT is the single artifact persisted, and only inside your client's config file.

This is the right path if you want the install scripted, you're on a self-hosted deployment, or you're connecting a client the wizard doesn't list.

## Option 3 ŌĆö Direct from the agent (zero install)

Standards-compliant MCP clients (Claude Desktop, ChatGPT, Cursor, Gemini, Anthropic agents) can connect directly without the wizard or the CLI ŌĆö they bootstrap via DCR + PKCE on their own. Just point the agent at the workspace MCP URL:

```
https://<region>.api.useatlas.dev/mcp/{workspace_id}/sse
```

The agent does discovery, registration, and the OAuth dance unattended. The protocol details below describe what those clients see end-to-end. This is the right path if your agent vendor publishes "paste an MCP URL" UX.

<Callout type="info">
Whichever path you pick, your connected agents land in the same **Settings ŌåÆ AI Agents** view in Atlas ŌĆö list, last-use timestamp, outstanding-token count, per-agent revoke. Workspace admins additionally see *every* agent registered in the workspace at **Admin ŌåÆ OAuth Clients**.
</Callout>

## URL pattern

```
https://<region>.api.useatlas.dev/mcp/{workspace_id}/sse
```

| Region | Hostname |
|--------|----------|
| US | `api.useatlas.dev` |
| EU | `api-eu.useatlas.dev` |
| APAC | `api-apac.useatlas.dev` |

Hit any region with a workspace assigned to a different region and you'll get a `421 Misdirected Request` (see [Cross-region routing](#cross-region-routing) below). Use the workspace's home region.

The single endpoint accepts:

- `POST` ŌĆö JSON-RPC frames (`initialize`, `tools/list`, `tools/call`, etc.)
- `GET` ŌĆö SSE notifications stream
- `DELETE` ŌĆö explicit session termination

The MCP SDK handles dispatch automatically ŌĆö clients pointed at the URL above just work.

## Authentication

Atlas's authorization server lives at `/api/auth`. Three discovery documents drive the OAuth flow:

| Document | URL |
|----------|-----|
| Authorization-server metadata (RFC 8414) | `/.well-known/oauth-authorization-server/api/auth` |
| OIDC discovery | `/.well-known/openid-configuration/api/auth` |
| Protected-resource metadata (RFC 9728) | `/.well-known/oauth-protected-resource/mcp/{workspace_id}` |

A standards-compliant MCP client:

1. Hits the resource URL `/mcp/{workspace_id}/sse` without a bearer
2. Receives a `401` with `WWW-Authenticate: Bearer realm="Atlas MCP", resource_metadata="ŌĆ”"` pointing at the protected-resource document
3. Reads the document, finds Atlas's `/api/auth` issuer
4. Registers as a public OAuth client via `POST /api/auth/oauth2/register` (DCR ŌĆö no admin pre-issuance required)
5. Sends the user through `/api/auth/oauth2/authorize` with PKCE + `scope=openid mcp:read offline_access`
6. The user signs in (if needed), reviews and approves on `/oauth2/consent`
7. Client exchanges the code at `/api/auth/oauth2/token` and gets a JWT access token + refresh token
8. Reconnects to `/mcp/{workspace_id}/sse` with `Authorization: Bearer <jwt>`

```http
Authorization: Bearer eyJhbGciOiJSUzI1NiIsŌĆ”
```

### Scopes

| Scope | Grants |
|-------|--------|
| `openid` | The user's Atlas identity (`sub` claim) |
| `profile` | Name and avatar |
| `email` | Email address |
| `offline_access` | A refresh token so the client doesn't have to send the user back through the browser |
| `mcp:read` | Query workspace data through the MCP endpoint ŌĆö required for any agent that wants to use Atlas as a data source |
| `mcp:write` | Reserved for future write paths (run mutations, edit semantic layer). Currently the hosted MCP surface is read-only, so a token without `mcp:write` works for every shipping MCP tool |

### Audience

Tokens are bound to a per-region resource audience (`https://<region>.api.useatlas.dev/mcp`). A token issued at `api.useatlas.dev` won't verify at `api-eu.useatlas.dev` ŌĆö audience mismatch is the first line of residency defense. The protected-resource metadata advertises the correct audience for each workspace.

### Workspace claim

Every issued JWT carries a custom claim `https://atlas.useatlas.dev/workspace_id` set to the workspace the user authenticated against. The hosted MCP route verifies this claim matches the `{workspace_id}` path segment ŌĆö mismatch is `403 forbidden`. The server never reveals whether a given workspace exists.

## Workspace scoping

```http
GET /mcp/org_b/sse                 ŌåÉ path workspace
Authorization: Bearer <jwt>        ŌåÉ workspace_id claim = org_a
ŌåÆ 403 forbidden
```

This guards against probing: a user authenticated against workspace A cannot pivot to workspace B by URL.

## Cross-region routing

Atlas guarantees data residency from 1.0.0 onward. The MCP endpoint enforces this strictly: a request for an EU workspace that lands on the US instance returns `421 Misdirected Request`, even when the API is configured for graceful (non-strict) routing on the data path.

```json
{
  "error": "misdirected_request",
  "message": "MCP requests for this workspace must be directed to the eu-west region.",
  "correctApiUrl": "https://api-eu.useatlas.dev",
  "expectedRegion": "eu-west",
  "actualRegion": "us-east",
  "requestId": "ŌĆ”"
}
```

MCP clients do not follow HTTP redirects by default, so the response body carries the correct URL. Clients should read `correctApiUrl` and reconnect against that hostname.

## Sessions and identity

A POST without the `mcp-session-id` header is treated as a session-init request ŌĆö the SDK creates a fresh server-side MCP session bound to the bearer's verified identity and returns the new session id in the response header. Subsequent frames send the header back and route to the same session. POSTs with an unknown session id return `404 unknown_session`; the client recovers by re-initializing.

The session lives in memory on the API instance that handled init. The hosted deployment runs each regional API service (`api`, `api-eu`, `api-apac`) pinned to a single replica because Railway's load balancer does not support sticky sessions; horizontal scale is achieved by adding regions, not by adding replicas within a region.

`DELETE` on the URL (with the `mcp-session-id` header) tears the session down. Sessions are also garbage-collected when the underlying connection drops.

## Audit

A `mcp_session.start` row is written to the workspace's admin action log when a session is established ŌĆö once per session, not per JSON-RPC frame. The row metadata carries `sessionId`, `orgId`, `clientId` (the registered OAuth client ŌĆö e.g. `claude-desktop` or a DCR-allocated UUID), and `region`.

OAuth client lifecycle (register, rotate, delete) and token lifecycle (issue, revoke) are recorded by Better Auth's `oauthProvider` plugin in its own tables (`oauthClient`, `oauthAccessToken`, `oauthRefreshToken`, `oauthConsent`).

## Inspecting and revoking clients

Two surfaces, same data. Pick the one that fits the situation.

### Settings ŌåÆ AI Agents (per-user)

Any workspace user can see **their own** connected agents at **Settings ŌåÆ AI Agents** (`/settings/ai-agents`). Each row shows the registered name, redirect URIs, registration date, last-use timestamp, and outstanding token count. The per-row **Revoke** invalidates every access token, refresh token, and consent record for that (client, user) pair ŌĆö the agent will re-register cleanly via DCR on its next authorization attempt.

Use this when you want to manage your own agents without bothering an admin: a laptop you don't use anymore, an agent you're done experimenting with, or a quick "rotate everything" before lending the device.

### Admin ŌåÆ OAuth Clients (workspace-wide)

Workspace admins additionally see **every** OAuth client registered against the workspace at **Admin ŌåÆ Users & Access ŌåÆ OAuth Clients** (`/admin/oauth-clients`). Same per-row Revoke; same atomic revocation semantics. An `oauth_client.revoke` row lands in the admin action log carrying the per-table revocation counts (and identifying whether the actor was a user revoking their own client or an admin revoking someone else's).

Use this when:

- A laptop running an authorized MCP client is lost and the user can't sign in to revoke from their own settings
- A teammate offboards and every agent they registered should be invalidated
- A registered client name no longer matches the agent it was issued to (clean it up ŌĆö DCR will re-register fresh)

### Per-client setup notes

Documentation anchors used by the in-product wizard "How to paste this in `{agent}`" links:

#### Claude Desktop

Open `~/Library/Application Support/Claude/claude_desktop_config.json` (macOS) or `%APPDATA%\Claude\claude_desktop_config.json` (Windows). Merge the wizard's block into the top-level `mcpServers` object. Restart Claude Desktop.

#### Cursor

Open Cursor ŌåÆ **Settings ŌåÆ Cursor Settings ŌåÆ MCP**, click **Add new MCP server**, and paste the wizard's block. Cursor reloads its MCP servers without a restart.

#### Continue

Open `~/.continue/config.json`. Add the wizard's `mcpServers.atlas` block under the top-level `mcpServers` key. Continue picks up the change on its next reload.

#### ChatGPT

In ChatGPT's **Connectors / MCP servers** UI, click **Add server** and paste the URL from the wizard's config block. ChatGPT performs DCR + PKCE on its own when you first invoke a tool.

## Operational notes

- **Max sessions per instance** is bounded by `ATLAS_MCP_MAX_SESSIONS` (default 100). Hitting the cap returns `503 too_many_sessions`. The cap counts both registered sessions and in-flight initializations.
- **Single replica per regional service.** Sessions are in-process ŌĆö a frame routed to a different replica than the one that ran `initialize` returns `404 unknown_session`. Railway's load balancer does not support sticky sessions ([Railway docs](https://docs.railway.com/reference/scaling)), so each regional API service (`api`, `api-eu`, `api-apac`) is pinned to `numReplicas: 1` in `deploy/<service>/railway.json`. Atlas's hosted deployment scales horizontally by adding regions, not replicas-within-region. The path to lifting the cap (move sessions out of memory into the existing internal Postgres) is tracked in [#2069](https://github.com/AtlasDevHQ/atlas/issues/2069); raising `numReplicas` before that ships will silently break every active MCP session on every reroute.
- **Region failure mode**: residency is fail-closed. If the region of the workspace cannot be looked up (internal DB unreachable), this surface returns `503 region_unavailable` rather than serving traffic that might cross residency.
- **Unauthenticated DCR** is enabled by default to support standard MCP clients. Operators with stricter threat models can opt out via `ATLAS_OAUTH_ALLOW_UNAUTH_DCR=false` ŌĆö but doing so prevents Claude Desktop / ChatGPT / Cursor from onboarding without manual client provisioning.
- A dedicated `mcp.useatlas.dev` hostname (regional CNAME-set) is on the ops backlog. Use the regional API hostnames in the meantime.

## Migrating from the v1.4.0-dev `mcp_tokens` bearer

PR A of #2024 introduced a custom `mcp_tokens` table for admin-minted bearers. PR C replaced it with OAuth 2.1 access tokens before v1.4.0 shipped ŌĆö there are no production tokens to migrate. Self-hosted dev DBs that ran migration `0046_mcp_tokens.sql` get the table dropped by `0047_drop_mcp_tokens.sql` automatically.

If you'd built a script against the legacy `/api/v1/admin/mcp-tokens` endpoints, replace it with:

- `POST /api/auth/oauth2/register` for client registration (RFC 7591 / DCR)
- `POST /api/auth/oauth2/token` with the `client_credentials` grant for machine-to-machine
- `POST /api/auth/oauth2/revoke` for revocation (RFC 7009)
