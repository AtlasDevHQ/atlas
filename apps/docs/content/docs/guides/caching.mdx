Atlas caches query results in-memory so that identical queries return instantly within the TTL. Cache keys are scoped by SQL, connection, organization, and user claims — so RLS-filtered queries and multi-tenant deployments never leak cached data across boundaries.

Only the agent's `executeSQL` path reads and writes this cache — dashboard tiles, metrics, and REST/OpenAPI datasource calls never cache — so the hit-rate stats below reflect agent query traffic only.

<Callout title="Prerequisites">
- Atlas server running (`bun run dev`)
- Caching is **enabled by default** — no configuration needed to start
- Admin role for cache management endpoints
</Callout>

## How It Works

1. When `executeSQL` runs a query, Atlas builds a cache key from five components:
   - **SQL text** (trimmed)
   - **Connection ID** (which datasource)
   - **Org ID** (multi-tenant isolation)
   - **User claims** (RLS context — e.g., `role: "admin"` vs `role: "member"`), deep-canonicalized so nested claim values count
   - **Resolved RLS configuration** — enabling or tightening RLS changes the key material, so every pre-change entry is orphaned instantly
2. The key is a SHA-256 hash of these components, so different orgs or RLS contexts never share cached results
3. On a **cache hit**, the stored result is returned instantly. The tool response is marked `cached: true` and carries the hit's **age** (`cacheAgeMs`), so the agent can caveat time-sensitive answers, and the audit log records the query's *original* execution duration (not 0ms), so slow-query analytics stay honest across hits
4. On a **cache miss**, the query runs against the database, and the result is stored in the cache for the configured TTL
5. When a user asks for fresh data ("refresh that number"), the agent sends `bypassCache` — the query re-runs against the database and the entry is re-frozen with the fresh result

## Cache Key Scoping

Cache keys include **all five components**, ensuring strict isolation:

| Component | Purpose |
|-----------|---------|
| SQL text | Same query = same key |
| Connection ID | Different datasources never share cache |
| Org ID | Multi-tenant isolation (even for identical SQL) |
| User claims | RLS-aware — different claim sets produce different keys |
| Resolved RLS config | An RLS policy change orphans all pre-change entries by construction |

Claim **keys** are sorted alphabetically before hashing, so key insertion order doesn't affect the cache key.

<Callout type="info">
A query like `SELECT * FROM orders` with `role: "sales"` produces a different cache key than the same query with `role: "admin"`. This ensures RLS-filtered results are never served to users with different access levels.
</Callout>

## Admin Cache Management

The [admin console](/guides/admin-console) provides a cache management UI at `/admin/cache` with per-workspace stats, an entry inspection table with per-entry delete, runtime enable/TTL controls, and a flush button.

### Stats Endpoint

```
GET /api/v1/admin/cache/stats
```

The payload is **scoped to the caller**: a workspace admin sees their own org's bucket; a platform admin sees fleet totals plus the resolved capacity.

| Field | Description |
|-------|-------------|
| `scope` | `workspace` (your org's bucket) or `platform` (fleet totals) |
| `enabled` | Whether caching is active |
| `hits` / `misses` | Cache hits and misses in your scope |
| `hitRate` | Lifetime hit rate (0–1); `null` before any access |
| `since` | When counting started (epoch ms); `null` while the cache is still warming |
| `windowHitRate` | Hit rate over the last hour; `null` with no recent activity |
| `windowTotal` | Accesses in the last-hour window |
| `entryCount` | Live cached entries; `null` when the cache backend can't report a count |
| `maxSize` | Configured capacity; `null` in the workspace view (capacity is platform framing) |
| `ttl` | TTL in milliseconds |

### Entry Inspection

```
GET /api/v1/admin/cache/entries
DELETE /api/v1/admin/cache/entries/{key}
```

Lists your org's live cached entries as **metadata rows** (capped SQL preview, connection, row count, age, TTL) — the cached rows themselves never leave the cache. Deleting one entry forces a single query fresh without flushing everything; the delete is org-authorized, so a workspace admin can only ever remove their own org's entries. `entries: null` means listing is unavailable on the current cache backend (plugin backends without a trustworthy org index).

### Flush Endpoint

```
POST /api/v1/admin/cache/flush
```

**Org-scoped by default**: removes only the caller's org's entries and returns the count. A platform admin can pass `{ "scope": "fleet" }` to clear every workspace's entries in the region — the admin UI discloses that blast radius before confirming. Flushing while caching is disabled returns `409`. The cache is also automatically flushed on config reload to prevent stale results from outdated settings.

## Troubleshooting

### Cache not reducing query latency

**Cause:** Cache keys include user claims. If every request has unique claims (e.g., a timestamp), every query is a cache miss.

**Fix:** Ensure RLS claims contain only stable values like `org_id` or `role`, not request-specific data.

### Cache appears stale after schema changes

**Cause:** The cache is not automatically invalidated when the database schema changes — only when the Atlas config file changes.

**Fix:** Flush the cache via `POST /api/v1/admin/cache/flush` or the admin console after schema changes. If only one result is stale, delete just that entry from the admin console's entry table (or `DELETE /api/v1/admin/cache/entries/{key}`) instead of flushing everything — or ask the agent to refresh, which bypasses and re-freshens the entry.

### Hit rate is very low

**Cause:** A structurally low hit rate is expected, not a fault — cache keys are exact hashes of the (trim-normalized) SQL, and the agent regenerates SQL nondeterministically, so repeat hits are largely incidental rather than guaranteed. Beyond that, the `maxSize` may be too small for your query diversity, or the TTL too short.

**Fix:** Tune the cache from **Admin → the Cache page** — the enable toggle and TTL apply to your workspace at runtime with no redeploy; max entries is a platform-wide setting. Self-hosted deployments can also set the `ATLAS_CACHE_*` env vars (see [Cache Configuration](/self-hosted/deployment/cache-configuration)). Monitor stats via `GET /api/v1/admin/cache/stats`.
