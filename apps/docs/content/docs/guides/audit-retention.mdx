Atlas includes configurable audit log retention with automatic purging and compliance-ready export formats. Workspace admins can set retention periods, configure soft-delete with delayed hard-delete, and export audit data in CSV or JSON for SOC2 compliance.

<Callout type="info" title="SaaS Feature">
  Audit log retention policies are available on [app.useatlas.dev](https://app.useatlas.dev) Business plans. Self-hosted deployments manage retention directly via their own database.
</Callout>

<Callout title="Prerequisites">
- Internal database configured (`DATABASE_URL`)
- Active Business plan on [app.useatlas.dev](https://app.useatlas.dev)
- Admin role required for all retention management endpoints
</Callout>

---

## How It Works

1. An admin sets a retention policy (30 days, 90 days, 1 year, custom, or unlimited)
2. A daily purge scheduler soft-deletes audit entries older than the retention window
3. Soft-deleted entries are permanently removed after a configurable hard-delete delay (default 30 days)
4. Soft-deleted entries are hidden from normal audit views but recoverable during the delay period
5. Compliance exports exclude soft-deleted entries and support date range filtering

## Retention Policy

### Setting a Policy

```bash
# Set 90-day retention with 30-day hard-delete delay
curl -X PUT /api/v1/admin/audit/retention \
  -H "Content-Type: application/json" \
  -d '{"retentionDays": 90, "hardDeleteDelayDays": 30}'
```

### Policy Options

| Setting | Values | Default |
|---------|--------|---------|
| `retentionDays` | 7+ or `null` (unlimited) | `null` (unlimited) |
| `hardDeleteDelayDays` | 0+ | 30 |

- Minimum retention period is **7 days** (enforced by validation)
- Setting `retentionDays` to `null` disables automatic purging (unlimited retention)
- `hardDeleteDelayDays` controls how long soft-deleted entries remain recoverable

### Viewing Current Policy

```bash
curl /api/v1/admin/audit/retention
```

Returns the current policy including last purge timestamp and count.

## Purge Behavior

### Automatic Purge

When enterprise features are enabled and an internal database is configured, Atlas automatically runs a purge cycle every 24 hours:

1. **Soft-delete**: Entries older than the retention window get a `deleted_at` timestamp
2. **Hard-delete**: Entries where `deleted_at` is older than the hard-delete delay are permanently removed

### Manual Purge

Admins can trigger purge cycles manually:

```bash
# Soft-delete expired entries
curl -X POST /api/v1/admin/audit/retention/purge

# Permanently delete soft-deleted entries past the delay
curl -X POST /api/v1/admin/audit/retention/hard-delete
```

### Observability

Every purge cycle logs entry counts:

```
Audit log entries soft-deleted { orgId: "org-1", softDeletedCount: 142, retentionDays: 90 }
Audit log entries permanently deleted { orgId: "org-1", hardDeletedCount: 85, delayDays: 30 }
```

### Scheduler self-audit

The purge scheduler also writes a row to `admin_action_log` for every tick
so compliance reviewers can distinguish a scheduled purge from a manual
one — and detect a scheduler that stopped running.

| Action type | Status | When | Metadata |
|---|---|---|---|
| `audit_log.purge_cycle` | `success` | Every 24 h tick, including zero-row cycles | `{ softDeleted, hardDeleted, orgs }` |
| `audit_log.purge_cycle` | `failure` | A cycle that errored (e.g., DB outage during soft-delete) | `{ error, softDeleted: 0, hardDeleted: 0, orgs: 0 }` |
| `audit_retention.hard_delete` | `success` | Only when a cycle actually removes rows (`count > 0`) | `{ deletedCount, orgCount, affectedOrgs: [{ orgId, deletedCount }] }` |

Both action types use the reserved actor `system:audit-purge-scheduler`
and `scope = "platform"`. Writing a `status: "failure"` cycle row is
what makes the absence of successful cycle rows over the retention
window attributable — a compliance reviewer can distinguish a silent
drop-off from an errored run. Manual purges from the admin surface emit
`audit_retention.manual_purge` / `manual_hard_delete` instead — the
distinct action types let forensic queries tell a routine retention
purge from an admin-triggered one.

## Compliance Export

Export audit logs in SOC2-ready format with optional date range filtering:

```bash
# Export as CSV
curl -X POST /api/v1/admin/audit/retention/export \
  -H "Content-Type: application/json" \
  -d '{"format": "csv", "startDate": "2026-01-01", "endDate": "2026-03-31"}'

# Export as JSON
curl -X POST /api/v1/admin/audit/retention/export \
  -H "Content-Type: application/json" \
  -d '{"format": "json"}'
```

### Export Fields

Both formats include: `id`, `timestamp`, `userId`, `userEmail`, `userLabel`, `authMode`, `sql`, `durationMs`, `rowCount`, `success`, `error`, `sourceId`, `sourceType`, `targetHost`, `tablesAccessed`, `columnsAccessed`, `orgId`.

### Limits

- Maximum 50,000 rows per export
- Response includes `X-Export-Truncated` and `X-Export-Total` headers when truncated
- Use date range filters to export larger datasets in chunks

## Admin UI

The retention settings are available in the admin console under **Audit Log > Retention** tab:

- **Retention period dropdown**: 30 days, 90 days, 1 year, custom, unlimited
- **Custom days input**: Shown when "custom" is selected (minimum 7)
- **Hard delete delay**: Days before permanent removal (default 30)
- **Last purge info**: Timestamp and count from most recent purge
- **Run Purge Now**: Manual trigger for immediate purge
- **Compliance Export**: Format selection (CSV/JSON) with date range picker

## Admin Action Log Retention

Two audit tables live side-by-side in the internal database:

- **`audit_log`** — query-level audit (every SQL run through the agent). Covered above.
- **`admin_action_log`** — privileged-action audit (every admin mutation, plugin install, role change, retention-policy update, etc.).

The admin-action log has its own retention policy, its own manual purge, and its own erasure surface for GDPR / CCPA right-to-erasure requests. Configured in **Audit Log &gt; Retention &gt; Admin action log** and surfaced at `/api/v1/admin/audit/admin-action-retention`.

### Recommended default — 7 years

Admin-action retention defaults to **2555 days (7 years)** — the common SOC 2 / HIPAA / ISO 27001 window for privileged-action logs. `null` (unlimited) is available for operators who prefer an explicit "keep forever." Minimum retention is 7 days (same guardrail as the query-audit trail) — shorter windows risk wiping evidence of an in-progress investigation.

Admin-action purge is a **direct hard-delete** — there is no soft-delete recovery stage. The 7-year window is long enough that a second window adds little safety relative to the admin-action volume.

### Setting a Policy

```bash
# Set 7-year retention (recommended)
curl -X PUT /api/v1/admin/audit/admin-action-retention \
  -H "Content-Type: application/json" \
  -d '{"retentionDays": 2555, "hardDeleteDelayDays": 30}'

# View current policy + last-purge metadata
curl /api/v1/admin/audit/admin-action-retention
```

### Manual Purge

```bash
curl -X POST /api/v1/admin/audit/admin-action-retention/purge
```

Returns per-org `{ orgId, deletedCount }` results. Normally runs automatically on the same 24-hour scheduler tick that purges the query-audit log.

### User Erasure (GDPR / CCPA right-to-erasure)

The admin-action log is the forensic record of what a user did while logged in as an admin. When a user invokes their right-to-erasure, you scrub their identifier — but preserve the row so the sequence of admin actions stays reconstructable.

```bash
# Preview — returns { anonymizableRowCount } without modifying state
curl "/api/v1/admin/audit/erase-user/preview?userId=usr_abc123"

# Erase — scrubs actor_id + actor_email → NULL, stamps anonymized_at = now()
curl -X POST /api/v1/admin/audit/erase-user \
  -H "Content-Type: application/json" \
  -d '{"userId": "usr_abc123", "initiatedBy": "dsr_request"}'
```

| Field | Meaning |
|---|---|
| `initiatedBy: "self_request"` | User hit a self-serve erasure button |
| `initiatedBy: "dsr_request"` | You processed a formal DSR letter on the user's behalf |

The erasure emits a `user.erase` audit row with `{ targetUserId, anonymizedRowCount, initiatedBy }` — **even when the row count is zero**. The regulator-facing contract is "we processed the request," and a zero-row erasure is still forensic evidence the request was handled.

### Pino / operational log boundary

<Callout type="warn" title="Pino logs are out of scope">
  Identifiers are removed from the admin-action audit log only. Pino / operational logs (stdout, Grafana Loki, or whatever sink you've wired up) are controlled by your log-aggregator retention policy. A compliance-minded operator processing an erasure request should also verify their log aggregator's retention window against their regulatory response deadline.
</Callout>

This is deliberate — the pino line "admin X changed connection Y" is operational telemetry an oncall engineer needs during triage. Redacting identifiers pre-write would force every triage flow to join on the DB row to identify the actor, defeating the purpose of the log line. Aggregators' own retention windows (days for SaaS Loki, operator-configured self-hosted) naturally age out identifiers within most regulators' response windows.

### Admin UI

Under **Audit Log &gt; Retention**, the page is split into two sibling sections:

- **Query audit log** — retention policy, compliance export, manual purge for `audit_log` (covered above).
- **Admin action log** — retention policy, manual purge, and **Erase user** button for `admin_action_log`.

The "Erase user" button loads a preview count before the confirm dialog so you can see the blast radius of a typo'd userId before committing. The confirm dialog shows the exact row count that will be scrubbed and the chosen initiator (DSR vs self-service).

### Scheduler self-audit

Like the query-audit trail, admin-action retention is audited by the same 24-hour scheduler:

| Action type | Status | When | Metadata |
|---|---|---|---|
| `admin_action_log.purge_cycle` | `success` | Every 24 h tick, including zero-row cycles | `{ deletedCount, orgs }` |
| `admin_action_log.purge_cycle` | `failure` | A cycle that errored | `{ error, deletedCount: 0 }` |
| `admin_action_retention.hard_delete` | `success` | Only when a cycle actually removes rows (`count > 0`) | `{ deletedCount, orgCount, affectedOrgs }` |
| `admin_action_retention.hard_delete` | `failure` | Any per-org DELETE errored (emitted even at zero total deletes) | `{ deletedCount, orgCount, affectedOrgs, failedOrgs }` |
| `admin_action_retention.manual_purge` | `success` / `failure` | Admin-triggered purge via API / UI | `{ deletedCount, retentionDays }` |
| `admin_action_retention.policy_update` | `success` / `failure` | Admin updated retention settings | `{ retentionDays, hardDeleteDelayDays, previousRetentionDays, previousHardDeleteDelayDays }` |
| `user.erase` | `success` (library) / `failure` (route) | Every erasure request, including zero-row | `{ targetUserId, anonymizedRowCount, initiatedBy }` |

The `user.erase` row is emitted regardless of the scrubbed row count — a zero-row erasure still proves the DSR request was processed.
